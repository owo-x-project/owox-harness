# 正本フォーマット

## 状態

採用済み。技術スタック (`docs/decisions/20260611-技術スタック.md`) の正本形式 (TOML) を見直し、差し替える。

## 背景

Phase 2 で人間が正本を編集する体験を検証した結果、次の指摘が出た。

- TOML の `[]`・`""`・`[[ ]]` の構文が直感的でなく、どこに何をどう書くか分かりにくい
- エンジニアでギリギリ、非エンジニア (vibe coder) には厳しい
- ルールが増えるとルート指示ファイルが肥大する (別決定: 最小ルート+段階的開示)

技術スタックの見直し条件「brand / rules の長文を TOML で人間が編集するのが苦痛になった時」に該当する。

技術スタックは「中核を型構造体に置き、TOML はその出入り口にする」としており、フォーマットは入出力の薄い層。型 (旗「型付き正本」の核) は Rust 側に残るため、出入り口の形式を差し替えても旗は壊れない。

## 判断基準

- `[]`・`""` 等の構文が不要で直感的に書ける
- owox-harness 自体は i18n しない。英語固定で自然に使える (フィールド名・生成物ラベル・雛形は英語。利用者が書く中身の言語は自由)
- 型付き正本の旗を保つ (読込後に Rust の型で検証できる)
- 依存を増やさない
- AI が生成・編集しやすい (この製品の本命は AI が会話で正本を書く流れ)
- 第1段階の正本はほぼ自由文 (brand 全項目・rules の方針) で、機械が厳密に要るのは少数

## 採用案

prose の正本は Markdown、machine 設定は TOML とする。

- prose (brand / rules / context / glossary): Markdown
  - 見出し (`## Section`) が項目に対応する。英語固定
  - 箇条書き (`- item`) が内容。単一段落の項目は本文テキスト
  - 構造化エントリ (不可逆操作・人間ゲート・用語集等) は箇条書きで `名前: 説明` で表す (区切りは最初のコロン)
  - パーサは行ベースで自作する (見出しと箇条書きのみの限定文法)
- machine 設定 (config.toml): TOML
  - 生成対象 CLI・出力先・ティア → モデル等の入れ子設定を素直に書ける。Markdown の箇条書きでは入れ子表現が弱い
  - 設定的なものは config.toml に集約する。今は `[targets.<cli>]`、今後の設定もここへ足す
- 型はどちらも読込後に Rust で検証する (必須欠落・未知見出し/キー・形式違反を弾く)

語彙の変更:

- `north_star` → `Vision` (一読で永続目的と分かる。当初 Goal にしたがセッション目標との誤読を避け Vision へ。`docs/decisions/20260613-Vision語彙とセッション目標.md`)

## 理由

- prose は Markdown が `[]`・`""` 不要で最も読み書きしやすく、AI も生成しやすい
- 英語固定は i18n を持たない方針と一貫し、語彙が分裂しない
- 型は読込後検証に回せるため、旗「型付き正本」を保ったまま入口だけ易しくできる
- targets は CLI・出力先・モデルの入れ子マップで、Markdown の箇条書きでは表現が弱い。TOML が素直
- 形式は「canon (中身) = Markdown / config (設定) = TOML」と性質で割れ、覚えやすい

## 捨てた案

- 全て Markdown に統一 (targets も Markdown)
- TOML 維持 + コメント雛形
- Markdown + frontmatter (構造化は frontmatter)

## 捨てた理由

- 全て Markdown は targets の入れ子マップが不自然になり、実物で分かりにくかった
- TOML 維持は構文障壁が残り、指摘の根本を解かない
- frontmatter は本文が易しくなる一方 frontmatter 側に構文が残り、混在になる

## 見直し条件

- 構造化エントリが増え、Markdown の箇条書き表現では曖昧になった時
- 自作パーサが肥大・複雑化した時 (その時はパーサ専用クレート採用を検討)
- prose 側にも入れ子設定が増え、Markdown で表しにくくなった時
- canon と config の線引きが曖昧になり、どちらの形式か迷うファイルが出た時
- 型検証の表現力が読込後検証で不足した時
