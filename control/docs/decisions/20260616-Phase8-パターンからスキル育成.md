# Phase8 パターンからスキル育成の導線

## 状態

採用 (実装済・検証済 `docs/validation/20260617-Phase8-対話検証と粗の是正.md`)。Phase8 (`docs/requirements/20260616-Phase8-経験を厚くする.md`) の設計。
スキルライフサイクル (`docs/decisions/20260613-Phase5-スキルと入口.md`) と腐敗検知の助言型 (`decay.rs`) を踏襲する。

## 背景

よく使う手順をスキルへ育てる導線が、AI の気づき任せで弱い。
育成の終端 (作成 → テスト合格 → 登録 → 昇格) は既にあるので、欠けている始端 (頻出手順の検知と提案) を足す。

設計判断はマネージャ問いで人間が確定済み:

- 使用履歴は MCP tool 呼び出し (入口コマンド列) と PreToolUse の Bash / 編集の両方で記録する。コマンドもシェル操作も拾い、よく使うパターンを完全に捉える
- 助言のみ。雛形自動生成はしない。人間が判断を握る理念・テストゲートの意味を守る

## 採用設計

### 使用履歴 (usage trail)

`.owox/usage.log` に追記専用。1行 = `<YYYYMMDD> <name>`。

- name は tool 名 / コマンド名のみ。引数は残さない (秘密漏れ・肥大の回避)
- 床コンテキストへ注入しない。検知時のみ読む
- 記録点は owox が既に通る2点:
  - MCP serve: 各 tool 呼び出しで1行追記 (入口コマンド = tool の列を捉える)
  - PreToolUse hook: Bash / apply_patch / Edit / Write の tool 名を1行追記 (シェル操作の列を捉える)
- 上限・巻き取り: 肥大を避け直近のみ保つ。方式は実装で決める (件数上限で先頭を捨てる等)

### 検知

core に routine.rs を新設 (decay と別関数・既存署名を変えない)。

- run_routine_suggestions: usage.log を読み、隣接する name の列 (長さ 2..=4) の出現回数を数える
- 閾値超え かつ 未スキル化 の列を提案として返す
- 閾値は quality.toml に `[routine]` 節を新設 (min_occurrences 既定 5・max_len 既定 4)。DecayConfig と同流儀で省略時既定
- 未スキル化の判定は最小: 同名・同手順のスキルが skills に無いか。精緻化は実装で

### 報告 (助言のみ)

- next の新節「育てられる手順」と verify.run の data に提案を載せる。腐敗検知と同じ advisory (commit を止めない)
- 次の行動を案内: SKILL.md + script を書く → skill.register (テストゲート) → skill.promote (人間昇格)
- 雛形自動生成 tool は作らない。skills ディレクトリは AI の作業域なので AI がファイルで書く

### 育成は既存ライフサイクルへ流す

提案 = 始端。技の質は既存のテスト合格ゲート + 人間昇格が担保する (`docs/decisions/20260613-Phase5-スキルと入口.md`)。
本フェーズは「気づき」を機械化するだけで、固定化の判断は人間が握る。

## 変更境界

触る:

- crates/core/src/routine.rs (新規・run_routine_suggestions と usage 読取)
- crates/core/src/quality.rs (`[routine]` 設定・RoutineConfig・RoutineRaw)
- crates/mcp/src/serve.rs (各 tool で usage 追記・next / verify.run へ提案・get_info)
- crates/mcp/src/hook.rs (PreToolUse で usage 追記)
- docs

触らない:

- スキルライフサイクル本体 (register / promote / テストゲート)
- run_decay / run_verify の署名
- targets/codex.rs (PreToolUse は既に登録済み。新 hook イベント不要)

守る振る舞い:

- 助言は止めない (advisory)。commit / stop ゲートは変えない
- usage は名前のみ (秘密を残さない)

## 危険

- 全 tool 呼び出し・全 PreToolUse で追記する I/O 負荷 → 1行 append で軽量。失敗しても作業を妨げない (best-effort)
- usage.log の肥大 → 上限巻き取り
- 提案ノイズ → 閾値 + 未スキル化 filter。実機で閾値調整
- 秘密混入 → 名前のみ記録で構造的に回避
- serve の全 tool に追記を差す重複 → 封筒返却の共通経路 (envelope_result) か dispatch 近くへ1箇所で差す。散在させない

## 検証方針

- 単体: usage 追記、N-gram 出現カウントと閾値、未スキル化 filter、上限巻き取り
- 結合: serve の tool 呼び出しで usage が増える、next / verify.run に提案が出る、PreToolUse で Bash 操作が記録される
- 手動 (実機 Codex): 同じ手順を繰り返すと提案が表面化し、そこから skill.register へ進める

## 捨てた案

- 雛形 (SKILL.md) の自動生成 → AI が手順を勝手に固定し、テストゲートの意味が薄れる (人間判断で却下)
- 使用履歴に引数を残す → 秘密漏れ・肥大
- 新 hook イベント (PostToolUse) を足す → PreToolUse で tool 名は取れる。イベントを増やさない
- 検知を MCP tool のみで行う → Bash の繰り返しを捉えられない (人間判断で両方採用)

## 見直し条件

- 提案の閾値・列長が実利用で合わない時
- 未スキル化 filter が弱く既存スキルを再提案する時
- usage 追記の負荷・肥大が問題になった時
