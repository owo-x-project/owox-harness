# Phase9 シェル書き込みの層ゲート被覆

`20260620-オーケストレーション具体化.md` の要実機確認「subagent 内 PreToolUse 発火」を対話 Codex で潰した際に見つけた層ゲートの粗の是正。実機検証は `docs/validation/20260620-Phase9-オーケストレーション実機.md`。

## 背景 (粗)

実機 Codex 0.141 の subagent (multi_agent の worker/explorer) は apply_patch/Edit/Write を持たず、読み書きを全て `exec_command` (PTY シェル) で行う。Codex はそのシェル呼出を tool_name=Bash として PreToolUse hook へ渡すので owox の機械強制は subagent 内でも発火する (例: `git tag -d` は不可逆検出で deny)。

ところが層ゲート (`crates/core/src/hook.rs` `layer_pre_action_gate`) の Bash 経路は `rm_targets` (削除) しか集めず、`>` `>>` `tee` `sed -i` 等のシェル書き込みを編集として検出していなかった。編集の層照合は apply_patch/Edit/Write のツール入力だけが対象だったため、subagent が契約面へシェルで書き込むと guarded ゲートを丸ごと回避する。親が同じ契約面を apply_patch で編集すれば deny されるのに subagent のシェル書き込みは素通り、という非対称が生じていた。

影響範囲は「書き込み (新規・追記・上書き) による契約面編集」。不可逆 (コマンド正規表現) と削除 (rm) の保護は subagent でも効いていた。subagent は編集を必ずシェルで行うため、subagent の契約面書き込みは常に素通りしていた。

## 採用した案

層ゲートと不可逆判定の Bash/シェル経路に書き込み先抽出 `write_targets` を足し、シェル書き込みを apply_patch 編集と同じく層照合へ載せる (`crates/core/src/hook.rs`)。

- 拾う経路: リダイレクト (`>` `>>` の連結 `>>file`/分離 `>> file`・fd 付き `2>`・両出力 `&>`)・`tee`・`sed -i`・`cp`/`mv` の宛先・`dd of=`。fd 複製 (`2>&1`) と読み込み (`<`) は除く
- 照合は既存と同じく `is_contract_surface` のみ。契約面への書き込みは deny、guarded 非契約面・free 層への書き込みは素通り (apply_patch の rule4 と完全に対称・操作前は止めず助言は別経路)
- 過剰検出は無害なので取りこぼしより過剰側へ倒す: 完全な shell 解析はせず、script 等を宛先候補に拾っても層 glob に当たらなければ deny にならない (例 `sed -i 's/a/b/' f` は `s/a/b/` も拾うが f だけが層に当たる)

## 採用理由

- guarded 境界の機械強制はオーケストレーション機能の前提 (subagent が guarded 境界を機械的に守る) であり、subagent の主編集経路で抜けると安全弁が無効化される
- apply_patch 経路と対称化することで「どのツール経路で書いても同じゲート」という不変条件が回復し、CLI/エージェント種に依存しない (将来性・清潔さ)
- 過剰側へ倒す近似は、契約面という狭いパス集合にしか deny が出ないため誤 deny の実害が小さく、取りこぼし (穴) の害の方が大きい

## 捨てた案

- シェル解析を完全にやる: shell 文法 (クォート・変数展開・サブシェル) の完全解析は非現実的で保守負債。狭い deny 対象に対し過剰検出近似で十分
- 不可逆判定 (`pre_tool_use_decision`) 側に書き込み検出を足す: 書き込みは不可逆でない (上書きは戻せる) ので層ゲートの責務。不可逆は削除・force push・tag 削除のまま
- guarded 非契約面のシェル書き込みも止める: apply_patch の rule4 (その他 guarded 編集は素通り) と非対称になり設計を崩す。対称性を優先

## 見直し条件

- subagent が `write_targets` の対象外コマンド (例 python の file write・mv による guarded ファイル退避=実質削除・here-doc の特殊形) で契約面へ書く事例が実機で出たら対象集合を広げる
- 過剰検出による誤 deny が実運用で頻発したら宛先抽出を絞る
- Codex が将来 worker に apply_patch 相当の構造化編集ツールを与えたら (multi_agent_v2 等)、シェル経路依存を見直す

## 確認

cargo 342 passed・clippy 0。実機 hook バイナリで subagent の実コマンド (`sh -c 'printf ... >> <契約面>'`) が deny へ変わり、guarded 非契約面・free 層書き込みは素通り維持を確認。対話 Codex TUI での subagent 経由 end-to-end 再確認は次セッション。
