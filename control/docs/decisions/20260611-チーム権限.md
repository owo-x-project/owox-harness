# チーム権限

## 状態

採用済み。

## 背景

運用の「個人からチームへ広げる」を支える。
owox は git-native (.owox/ がリポジトリ内) なので、認証やアクセス制御は再発明しない。

- `docs/decisions/20260611-制御方針.md`
- `docs/decisions/20260611-MCP設計.md`

## 大前提

identity / 認証 / アクセス制御は既存に乗る (git / PR レビュー / CODEOWNERS / ブランチ保護)。
owox が auth エンジンを作るのは対象外 (汎用道具化)。

## owox の役目

- ポリシー定義: 何を・誰が 承認/変更できるかを permissions.toml に持つ
- 承認の記録: ランタイム人間ゲートの承認を、誰が承認したか identity つきで来歴に残す
- solo から team へ広げる

## 権限モデル

- 最小の役割ベース: maintainer / contributor
- 必要なら拡張する

## 正本変更の承認

- brand / rules / decisions など正本はセンシティブ
- .owox/ の重要パスを CODEOWNERS にマップ生成し、git PR レビューで強制する
- owox は承認フローを抱えない (git に委譲)

## ランタイム人間ゲートの承認

- 役割で「誰がどのゲートを承認できるか」をポリシー化する (例: 不可逆=maintainer のみ)
- 承認は identity つきで来歴に残す
- identity は git の author / 設定から取得する
- この identity は監査記録であって認証ではない。詐称耐性は git ホスティング・PR レビュー・ブランチ保護に委ねる

## solo と team

- 既定は solo (自己承認)
- team モードで役割ゲートを有効にする

## 正本

- permissions.toml: 役割、誰が何を承認/変更できるか
- 生成: CODEOWNERS (.owox/ 重要パス → maintainer)

## 採用理由

- git-native なので、正本変更の承認は PR レビューにそのまま乗る
- auth を再発明せず、既存のチーム運用を壊さない
- 役割を最小にすると、多くのチームに合い設定も軽い
- 承認 identity を来歴に残すと、誰が判断したか追跡できる
- solo 既定で個人利用が軽いまま、team へ広げられる

## 捨てた案

- owox が権限・認証・identity をフル実装
- 権限を完全に git 任せ
- 細かい RBAC
- 役割なし (全員同一)
- 正本変更を owox 内承認フローで扱う

## 捨てた理由

- フル実装は認証再発明で重く、対象外の汎用道具化に近づく
- 完全に git 任せだと、ランタイム人間ゲートの承認者が追跡できない
- 細かい RBAC は初期から設定が肥大化する
- 役割なしはチームの承認境界を作れない
- owox 内承認フローは git と二重になる

## 見直し条件

- 役割が2つで足りなくなった時
- CODEOWNERS で守りきれない正本が出た時
- ランタイム承認の identity 取得が CLI 差で揺れた時
- team モードの設定が複雑化した時
