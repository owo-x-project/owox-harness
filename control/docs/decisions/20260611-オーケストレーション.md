# オーケストレーション

## 状態

採用済み。2026-06-20 に第1対象 Codex の subagent 実態調査で担当・エージェントタイプ・呼び出しルール・モデル割り当ての4観点を具体化した (`docs/decisions/20260620-オーケストレーション具体化.md`)。

## 背景

設計原則のマルチエージェントを具体化する。
owox は実行者ではない。オーケストレーションのパターンを正本で定義し、各 CLI ネイティブ subagent へ生成する。CLI が実行し、owox は制御方針で監督する。
PEV (計画-実装-検証) / GSD (並列の役割別エージェント) / BMAD (段階ゲート) を owox 流に採る。

- `docs/decisions/20260611-設計原則.md`
- `docs/decisions/20260611-制御方針.md`
- `docs/decisions/20260611-品質保証.md`

## owox の役目

- 役割・フェーズ・ゲート・handoff を正本で定義する
- 各 CLI ネイティブ subagent へ生成する
- 実行は CLI、監督は owox (ゲート・人間判断点)

## オーケストレーションパターン

- 骨格 = PEV: 調査 → 計画 → 実装 → 検証。レビューは merge 前
- 各役割 = 限定文脈 (最小コンテキスト) + 許可操作 (自律境界) + 出力成果物 (.owox/ へ) + 次フェーズゲート
- handoff = .owox/ の耐久成果物経由 (計画→来歴/tasks、実装→diff/work、検証→verification、レビュー→findings)。会話でない
- 並列 = 調査・レビューは fan-out (多視点)、実装はタスク単位で逐次
- スケール適応 = 小タスクは軽く (調査・重レビュー省略)、大はフル。state と連動

## エージェント定義に責務を持たせる (2026-06-13 追補)

各エージェントの「何をする存在か」(責務・振る舞いの定義) は agents.toml のエージェント定義に持たせる。
brand (プロジェクトの identity = Vision/価値/方針) には入れない。理由: 責務はエージェントの属性で、
プロジェクト identity とは別概念。brand に混ぜると両者が混ざる。

- 主エージェントも 1 つの定義として持つ (サブエージェントだけでなく)。主エージェントの責務が
  ルート指示ファイル (AGENTS.md 等) へ出る。サブエージェントの責務は各 CLI ネイティブ subagent 定義へ出る
- 各定義 = 責務 (何をするか) + 許可操作 (権限) + ティア + 文脈
- 現状 (Phase 4 時点) は未実装。AGENTS.md は汎用オリエンテーション (正本の在り処・読む先・用語定義の届き方) のみ。
  主エージェントの責務記述はこのエージェント定義を作る時に足す

## サブエージェント制御 (役割 → 権限)

- 役割 → 許可操作を agents.toml に型定義し生成する
- 例:
  - 調査: 読取のみ
  - 計画: 来歴・tasks 記録
  - 実装: コード編集可、commit 不可
  - レビュー: 読取のみ
  - 検証: verify 実行
- 不可逆操作は全役割で人間ゲート
- 各役割は文脈地図から必要分だけ受け取る

## 可逆性観での権限具体化 (2026-06-20 追補)

`docs/decisions/20260620-自律度根本方針と自動承認パス再設計.md` の根本方針 (核は人間が握り可逆な実作業は AI に任せる・3安全弁) を subagent 権限へ落とす。

- subagent は可逆な実作業に限定し核 (guarded 資産 = 契約面・rules・brand・glossary) を触らせない
- 役割 → 許可操作に加え許可層を agents.toml に型定義する。実装役割は supervised/free 層のコード編集のみ・commit 不可・guarded 層編集と不可逆操作は権限外
- 理由: subagent は人間と直接対話しない (handoff は `.owox/` 成果物経由) ため、二段ゲート (`layer_pre_action_gate`) で止まっても人間承認を単独で完結できない。核は主エージェント / 人間が握る
- subagent が guarded 層に到達したら層ゲートが止め、subagent は「核変更が必要」を成果物に残し主エージェントへ差し戻す
- auto 承認は点2 の機構 (層 × 不可逆の積・profile 性質で既定) を subagent も共有する。subagent 固有の追加機構は要らない
- 外部副作用 (commit/push/deploy 等の不可逆) は全役割で独立最優先ガード (根本方針の穴A) が止める

## モデル振り分け

- ティアベース宣言: 役割 → ティア (fast / balanced / strong / reasoning) を agents.toml に持つ
- 例: 調査・要約=fast / 実装=balanced / 計画・レビュー・検証=strong / 難所・敵対検証=reasoning
- ティア → 具体モデルのマップは config.toml の [targets.<cli>] に CLI・環境ごとに持つ (デフォルト+上書き)
- モデル乗り換えと CLI 差を吸収する

## 正本

- agents.toml: 役割・権限・ティア・オーケストレーションパターン (CLI 非依存)
- config.toml: machine 設定の集約。[targets.<cli>] にティア → 具体モデルのマップ (CLI・環境ごと)

## 採用理由

- パターンを正本定義して生成すると、CLI 非依存と品質安定を両立する
- handoff を耐久成果物にすると、会話揮発に頼らず文脈も最小になる
- 役割 → 権限の型定義で、レビューが書き換える等の事故を防ぐ
- ティアベースは、モデル変化と CLI 差に強い (乗り換え耐性)
- スケール適応で、小タスクの過剰オーケストレーションを避ける

## 捨てた案

- owox ランタイムが subagent を直接駆動
- subagent の使い方を CLI 任せ
- 役割一律の権限
- 具体モデルを正本へ直指定
- handoff を会話・メモリ経由

## 捨てた理由

- ランタイム駆動は CLI の API に依存し、CLI 非依存と矛盾する
- CLI 任せはパターンがばらつき品質が安定しない
- 一律権限は役割の事故を招く
- 具体モデル直指定はモデル変更・CLI 差で腐る
- 会話・メモリ handoff は揮発し、正本にならない

## 見直し条件

- フェーズ・役割が増えすぎて重くなった時
- ティアの粒度が用途に合わなくなった時
- ネイティブ subagent を持たない CLI を本格対応する時
- スケール適応の境界が曖昧になった時
- handoff 成果物が肥大化した時
