# オーケストレーション具体化

採用判断。2026-06-20。
`docs/decisions/20260611-オーケストレーション.md` (抽象設計) を、第1対象 Codex CLI の subagent 実態調査に基づき具体化する。
権限の可逆性観具体化は同正本の2026-06-20 追補に既出 (`docs/decisions/20260620-自律度根本方針と自動承認パス再設計.md` 連携)。本書は担当・エージェントタイプ・呼び出しルール・モデル割り当ての4観点を詰める。

## 背景: Codex subagent 実態 (2026-06-20 調査)

owox は subagent を実行せず各 CLI ネイティブ subagent 定義へ生成する側なので、生成先の機構を確定した。出典は OpenAI 公式 (developers.openai.com/codex/subagents・/hooks) ほか。control repo の実機メモ (memory) でも一部裏取り済。

- 定義は1エージェント1ファイル TOML (`.codex/agents/*.toml`・name/description/developer_instructions 必須・model/model_reasoning_effort/sandbox_mode/mcp_servers 可)。owox 正本名の agents.toml とは別物
- 自動委任しない。親が spawn_agent 等の tool call で明示起動する
- ファイル名指定 spawn 不可。spawn は agent_type + model/prompt 上書きのみ受ける (カスタム agent_type spawn の可否は要実機確認)
- per-subagent モデル指定可 (model + model_reasoning_effort=low/medium/high・省略時親継承)
- subagent 内のツール呼出も PreToolUse を発火する (turn スコープ・要実機確認)。承認モードは子へ伝播
- 各 subagent は独立コンテキスト窓を持ち親へ凝縮 summary のみ返す。トークンは単一より増える
- ツール単位の allow/deny リストは無い。制御は sandbox_mode (read-only/workspace-write) と mcp_servers の2軸のみ。hook は deny のみ
- 並列は multi_agent=true・max_threads 既定6・max_depth 既定1 (孫 spawn 不可)

## 採用した案

### 観点1 担当 (オーケストレータ)

- オーケストレータは主エージェント (親) で1階層のみ (Codex max_depth=1)。owox 自身は非実行 (生成・監督)
- Codex は自動委任しないので、owox は親の責務記述 (ルート指示ファイル) へオーケストレーションパターン (PEV 骨格・役割の存在・スケール適応基準) を能動提示する
- パターンの適用 (いつ何を spawn するか) は親が状況判断する。根本方針「最初を厚く握りあとは任せる」と一致

### 観点2 エージェントタイプ (役割 × 変種の2次元)

- 役割 = 基本5 (調査・計画・実装・レビュー・検証)。「何をする存在か」を agents.toml に型定義
- 変種 = 全役割共通の分岐軸。役割を増やさず専門性を出す
  - レビューの変種 = 観点 (lens)。`docs/decisions/20260614-Phase6-レビュー枠組み.md` の review.toml (correctness/design/security/plan-alignment/requirement/pruning) を再利用。同 decision は「ルータ (subagent 並列・ティア振り分け) が同契約を消費」と既に設計済
  - 多視点レビュー = レビュー役割 × 複数観点の fan-out / 敵対検証 = レビュー役割の敵対 (refute) 変種 / 庭師 = pruning 変種 (剪定スキル実行)
  - 将来の専門化は新役割でなく変種追加で対応 (役割数は5固定・設計原則の見直し条件131 を尊重)
- Codex 写像 = カスタム TOML 主体。役割を `.codex/agents/<役割>.toml` へ生成し agent_type で呼ぶ。変種は spawn 時の prompt 上書き (+model 上書き) で注入し、役割 × 変種ごとの TOML 爆発を避ける。owox 正本 agents.toml (CLI 非依存) → codex.rs が変換する既存生成パターン (canon→AGENTS.md) と同型
  - 最優先の実機確認: カスタム agent_type で spawn できるか。効かなければ prompt 上書き (developer_instructions 注入) で責務を与える経路へフォールバック

### 観点2 実機追補 (2026-06-20)

Codex 0.141 stable で custom agent_type spawn を実機確認した結果、不可と判明した。設計の安全弁どおりフォールバックを採用する。

- 採用: 組込み agent_type (worker/explorer 等) + spawn 時 prompt / developer_instructions 注入で役割の責務を渡す経路。床の `## Orchestration` 節が能動提示し、親が適用する
- `.codex/agents/*.toml` 生成は Codex 0.141 stable では消費されないため撤去。`crates/core/src/targets/codex.rs` の generate は hooks.json と config.toml の2ファイルのみを返す形へ戻した
- agents.toml 正本 (役割×変種・ティア・sandbox・許可層) とティア写像は CLI 非依存の真実源として保持する。custom agent_type 経路 (multi_agent_v2 成熟 or 他 CLI 対応) が来た時点で `.codex/agents/` 生成の右辺を復活させる

参照: docs/validation/20260620-Phase9-オーケストレーション実機.md

### 観点3 呼び出しルール

- スケール適応 (いつ使うか) = owox が task サイズと state (初期/安定/保守) から「小タスクは省略・大はフル」の基準を能動提示し、判断は親。既存 task/state 機構と連動 (新機構なし)
- 並列度 = 読取系 (調査・レビュー変種含む) は fan-out 並列・書込系 (実装) は逐次。書込並列のファイル競合を避ける (Codex subagent の worktree 分離可否が不明なため)。max_threads は Codex 設定に従う
- handoff = 既存 `.owox/` 層へマッピングし新層を作らない。一時成果 (findings・調査メモ) は work (ブランチ作業記憶)、昇格成果は task/来歴/knowledge/verification へ明示昇格。Codex summary は完了通知で、耐久な真実源は `.owox/` (会話に頼らない)

### 観点4 モデル割り当て

- ティアは4段 (fast/balanced/strong/reasoning) を維持。CLI 非依存・乗り換え耐性
- Codex の model_reasoning_effort は写像の右辺で吸収。config.toml `[targets.codex]` でティア → (model, model_reasoning_effort) を持つ (例 fast→mini+low / balanced→中+medium / strong→flagship+medium / reasoning→flagship+high)
- 役割 → ティアの既定は正本の中庸 (調査=fast・実装=balanced・計画/レビュー/検証=strong・親=reasoning)。変種で上書き (敵対検証=reasoning)。config.toml でプロジェクトごとに上書き可

## 採用理由

- 担当を親1階層に: Codex の max_depth=1 と自動委任なしに素直。能動提示で「使われない」を防ぎ、適用は親に委ねて根本方針と一致
- 役割 × 変種の2次元: 役割を5に固定し複雑化を防ぎつつ、既存 review 枠組み (観点=離散単位・ルータが消費) と直結。専門化が変種追加で型に乗る
- カスタム TOML 主体: Codex ネイティブ機構をそのまま使い清潔。実機リスクは prompt 上書きフォールバックで吸収
- 呼び出しルールを既存機構へ相乗り: task/state/work/来歴/knowledge/verification を再利用し新層・新機構を足さない。書込逐次で競合を回避
- ティア4段維持: 具体モデル直指定の陳腐化を避け、Codex の effort 軸は写像で吸収して他 CLI へも展開できる

## 捨てた案

- subagent を多階層 (孫 spawn) → Codex max_depth=1 で不可・複雑
- 専門役割を独立役割に増やす → 役割爆発・コンテキスト増 (変種で代替)
- 役割 × 観点をフラットな役割リストに展開 → 役割爆発
- Codex 写像を prompt 上書き主体に → ネイティブ機構を活かせない (人間判断でカスタム TOML 主体を採用・フォールバックには残す)
- 実装も並列 → worktree 分離可否が不明・ファイル競合 (将来 worktree 確認後に再検討)
- レビュー専用 findings 層を新設 → 新層・単一真実源精神に反する
- Codex summary 主体の handoff → 揮発・耐久性なし・正本の「会話でない」に反する
- ティアを (model 軸 × effort 軸) へ再設計 → CLI 非依存の抽象が崩れ effort 無い CLI で扱いにくい
- モデルティアの profile 性質連動 → 複雑 (将来 architecture 軸連動として拡張余地)
- スケール適応を owox が機械強制 → 硬直・親判断を奪う・根本方針と不整合

## 要実機確認 (target 検証で最優先)

- カスタム agent_type で生成 TOML を親が spawn できるか → **不可。フォールバック採用 (2026-06-20 実機確認済・上記追補参照)**
- subagent 内の apply_patch / MCP tool で PreToolUse が発火するか (owox 機械強制の subagent 成立)
- trusted / apps の host 要因が subagent の承認・ツール到達へどう出るか
- worktree 分離の可否 (実装並列の将来判断用)

## 見直し条件

- カスタム agent_type spawn が実機で効かない → 写像を prompt 上書き主体へ切替 (2026-06-20 実機確認・フォールバック採用済)
- multi_agent_v2 が stable 化 or custom agent_type 対応 CLI を本格対応 → `.codex/agents/` 生成を復活し右辺を custom TOML 主体へ戻す
- subagent 内で PreToolUse が発火しない → 機械強制の代替 (sandbox_mode/mcp_servers の構成) を再設計
- 役割の変種が増えすぎて複雑化した → 変種の整理・観点トリガの見直し
- 並列 fan-out のトークン増が問題になった → 並列度・スケール適応基準を絞る
- Codex の subagent スキーマが変わった → 生成 (codex.rs) を追従
