# Phase7 スライス2: コード/repo 腐敗検知の設計

## 状態

採用・実装済み・機械確証済み。Phase7 スライス2 (`docs/requirements/20260614-Phase7-腐らず成長する.md`)。
庭師の検出側 (`docs/decisions/20260613-庭師-剪定.md`)。owox はファイルシステムレベルに絞り構文解析は委譲する。
cargo 174 passed・MCP stdio/hook で target サンドボックス確証 (`docs/validation/20260614-Phase7-コードrepo腐敗検知.md`)。実機 Codex 対話検証は全スライス後。

## 背景

スライス1 でタスク・来歴の腐敗検知を decay 機構として実装した (`docs/decisions/20260614-Phase7-腐敗検知の中核.md`)。スライス2 はこれをコード/repo へ広げる。owox は言語非依存で構文解析を持たない (quality と同じ制約)。

人間判断 (plan の論点) を確認し確定した:

- owox 直接検出 = 内容一致の重複ファイル (言語非依存・誤検出ほぼ無)
- 死コード・未使用は `[[decay.checks]]` でプロジェクトの検査コマンドへ委譲し、owox が走らせ decay として報告する advisory な枠
- 報告は既存 decay チャネル。next はタスク・来歴 decay のみ (高速)・verify.run が repo decay も含む全体像・commit はコード decay を止めない

## 採用設計

### 重複ファイル検出は decay.rs の core 関数

decay.rs に `run_code_decay(work_dir, files, config) -> Vec<DecayFinding>` を足す。decay.rs が DecayFinding の生成元を一手に持つ (タスク・来歴は run_decay、コード/repo はここ)。ファイル列挙は mcp が `files.rs` で渡し、内容読取は core が行う (quality と同じ。core は git/時計は持たないがファイル読取は決定論で持てる)。

- 追跡ファイルを内容でグループ化し、完全一致が 2 つ以上あれば各ファイルを `duplicate-file` として報告 (detail に一致相手のパス)
- 内容一致はハッシュでなく内容そのものでグループ化し、衝突誤検出を作らない
- 空・極小ファイルの雑音を避けるため `min_duplicate_bytes` 未満は対象外 (既定 64)
- read_to_string できないファイル (バイナリ) は飛ばす

### 委譲検出 [decay.checks] は run_check を再利用

`run_code_decay` は重複検出の後に decay.checks を走らせる。

- quality.toml に `[[decay.checks]]` (name + command) を新設し DecayConfig へ持つ
- 規約: 検査コマンドが decay を見つけたら非ゼロ終了する。owox は `crate::verify::run_checks` を再利用し、通過 (exit 0) = decay 無し、非通過 = decay 有りとして `decay-check` finding を作る (detail に出力要約)
- 完了を判定する `[[verify.checks]]` (pass/fail ゲート) と別物。死コードは「完了失敗」でなく助言なので、ゲートを汚さず decay として扱う

### DecayConfig の拡張

quality.rs の DecayConfig へ足す (quality.toml から読む設定なので budgets/boundaries・既存 decay 閾値と同居)。

- `min_duplicate_bytes: usize` (既定 64)
- `checks: Vec<VerifyCheck>` (`model::VerifyCheck` を再利用。型の重複を作らない。TOML は raw 経由で解釈し model の VerifyCheckRaw と同じ流儀)

### 報告経路 (slice1 のチャネルに合流)

- next: タスク・来歴 decay のみ (slice1 のまま)。repo 走査と外部コマンドを伴うコード decay は載せない (next は素早い航法に保つ)
- verify.run: mcp が run_decay (canon) と run_code_decay (repo) の結果を結合し、既存の `decay_findings` 引数へ渡す。run_verify の署名は変えない (slice1 で data.decay は汎用化済み)。repo decay は is_structural()=false なので「N structural one(s) block commit」の数は崩れない
- commit ゲート: 変更なし。コード decay は advisory で止めない。commit を止めるのは構造的タスク腐敗 (done未検証・ゾンビ) のみ。コード decay を commit に載せると毎 commit で nag するため verify.run に集約する

### 責務分担

- core: run_code_decay (重複検出の fs 読取 + decay.checks の実行)。DecayFinding を返す
- mcp: files 列挙と work_dir 供給、verify.run で run_decay と run_code_decay を結合

## 変更境界

触るもの:

- crates/core/src/quality.rs: DecayConfig へ min_duplicate_bytes・checks、DecayRaw の `[decay]` 拡張 (decay.checks の raw 解釈)
- crates/core/src/decay.rs: run_code_decay (重複検出 + decay.checks)・補助関数・テスト
- crates/core/src/lib.rs: run_code_decay 公開
- crates/mcp/src/serve.rs: verify.run が run_code_decay を呼び decay を結合
- tests/fixtures・target サンドボックス・docs

触らないもの:

- run_decay (タスク・来歴 decay) と slice1 の検出規則
- run_verify の署名・data.decay の形・完了3区別
- commit ゲート (commit_gate)・next の描画 (タスク・来歴 decay のみ)
- quality の budgets/boundaries・verify.checks の pass/fail ゲート

守る振る舞い:

- core は git/時計を持たず決定論 (ファイル列挙は mcp が与える)
- コード decay は advisory・commit を止めない
- `[decay]`/`[decay.checks]` 無しでも従来どおり動く (重複検出は既定で走る・checks は空)

## 危険

- 大きな repo で全追跡ファイルを読むコスト。verify.run のみで走らせ next には載せない。quality も同じ列挙を使うため二重列挙は避けられる
- 重複検出が正当に同一な定型ファイル (空 __init__・LICENSE 複製) を拾う。min_duplicate_bytes で極小を外し、advisory に留め block しない
- decay.checks の規約 (非ゼロ終了 = decay) をプロジェクトが誤ると、decay 無しでも報告が出る。検査コマンドはプロジェクト記述で、verify.checks と同じ自己責任
- decay.checks の外部コマンド実行コスト。verify.run と限定し next・commit では走らせない

## 検証方針

- 単体: 重複ファイルの検出/非検出、min_duplicate_bytes 未満は無視、バイナリ読取失敗は飛ばす、decay.checks の通過=無報告・非通過=報告、`[decay.checks]` 無しは空
- 結合: verify.run の data.decay にタスク・来歴 decay と重複ファイル・decay-check が混ざって出る、next には repo decay が出ない、commit ゲートがコード decay で止まらない
- 手動 (後の一括対話検証へ): 同一内容ファイルを 2 つ置き verify.run が duplicate-file を出すか、decay.checks に非ゼロ終了コマンドを置き decay-check が出るか、next が高速のままか
- 未確認として残す: 未参照ファイル検出 (言語非依存で誤検出多・将来 opt-in)・重複の除外 glob (将来)・decay.checks の per-check severity (将来)

## 捨てた案

- owox-native の未参照ファイル検出 → 言語非依存では entry point・設定・asset を誤検出。委譲 (decay.checks) かプロジェクト判断に委ねる
- 死コードを verify.checks (pass/fail) で表す → 助言を完了失敗扱いにしゲートを汚す。advisory な decay.checks を分ける
- 重複検出を next にも載せる → 全ファイル読取で next が重くなる。verify.run に限定
- コード decay を commit で block → 毎 commit で nag・狼少年化。advisory に留める
- 重複検出をハッシュキーで持つ → 衝突誤検出の余地。内容そのものでグループ化する
- decay.checks 用に新しい検査型を作る → model::VerifyCheck と重複。再利用する

## 見直し条件

- コード decay の advisory で剪定が進まず block が要ると分かった時
- 未参照ファイル検出を入れる判断がついた時 (opt-in 規則の設計)
- 重複検出のコストが大きく対象限定・除外 glob が要る時
- decay.checks の非ゼロ終了規約が運用に合わない時
