# tool 整理仕上げと仕様管理取り込み

採用判断。2026-06-28。
解釈B (tool 文脈削減 + スキル誘導) の骨格実装済みを受け、残ギャップと仕様管理調査 (spec-kit / OpenSpec) の
取り込みを確定する。

## 背景

実装済みの骨格:

- 床 Entry map → SessionStart での意図ルーティング
- 入口 skill 13個 (`crates/core/src/commands.rs`: kickoff / req / design / next / status / decide / verify / review / task / issues / skill / memo / handoff)
- `context scope="diff"` と `scope="codebase"` (`crates/mcp/src/serve.rs` 987-989行目)
- MCP コンテキスト削減 (`docs/decisions/20260620-MCPコンテキスト削減.md`)
- routine → script-skill 検出 (`crates/core/src/routine.rs`: RoutineKind::ScriptSkill / Confidence / is_script_skill_candidate)

残ギャップ: tool のカテゴリ露出方法・mode 統合範囲・要件書式の機械検証・変更差分の追跡・検証陳腐化の腐敗検知。

## 採用: A — tool カテゴリを露出しない・コード化しない

カテゴリは設計上の整理概念に留め、MCP description 接頭辞・コード列挙は追加しない。

根拠:

- 入口 skill が作業→tool 実行順を脚本化済みで、AI は迷わない
- MCP フラット一覧へカテゴリ語を足すとトークン増になり MCP 削減 (20260620) と逆行
- 52 tool をコードでカテゴリ列挙すると追従で腐るリストを生む
- 既存の command_routing_findings (`crates/mcp/src/serve.rs`) と entry_map_tools (`hook.rs`) を真実源にした庭師の配線検査で入口導線は既に守られており十分

## 採用: B — mode 統合を恒久棚上げしない形で見送り

統合判断基準 (返り値の形が同じ + 危険度が同じ + AI が迷うだけ + 説明が短縮される) を全候補へ厳密適用。
今回の見送り理由:

- knowledge.{list,get,lookup} の読み3種のみが全基準を満たす唯一の安全候補
- 削減効果は tool 2個分と小さく発見性に触れる
- profile / branch は読み書き混在で返り値が各々違う → 統合しない
- gate.auto_* は自動承認=安全境界であり統合禁止 (解釈A で既に却下。`docs/decisions/20260620-MCPコンテキスト削減.md`)
- MCP 削減で起動時負荷は十分下がっており据え置きが清潔

恒久棚上げ防止のため発火条件を明記: tools/list の起動時負荷が実測で再問題化し、かつ統合判断基準を全て満たす
群 (まず knowledge 読み系) に限り着手する。

## 採用: C — 要件書式の軽量機械検証を追加

owox の受け入れ基準は既に given/when/then 構造を持つ (`crates/core/src/requirement.rs` の AcceptanceCriterion)。
追加するのは以下2点のみ:

1. given / when / then の空フィールドを弾く (検証不能な基準を正本に入れない)
2. 受け入れ基準が0件の要件を verify.run で警告する (create 時は許容し incremental 運用を壊さない)

SHALL/MUST 規範語の強制は採らない。理由: SHALL/MUST は OpenSpec が構造枠を持たない代償の文章規約。owox は
given/when/then 構造で既にテスト可能性を担保しており、規範語強制は多言語・自然文に合わず利益が薄い。owox の
強み (構造 + 検証 link の機械強制) を活かす。

## 採用: D — 要件変更の差分追跡を requirement 内の構造化変更履歴で持つ

現状は title / statement 変更時に rationale を decision へ記録しリンクするのみで「何が変わったか」が
機械可読に残らない。責務を3分する:

- 正本 = 現在状態 (requirement 本体)
- 変更の理由 (なぜ) = decision (既存据え置き)
- 何が変わったか = requirement 内の構造化変更履歴 (日付・decision リンク)

記録する変更履歴は owox が実際に持つ操作が生む変更に限る:

- 内容・ライフサイクル変更: title 変更・statement 変更・status 変更 (update_requirement 経由)
- 作成後の criterion 追加 (add_criterion 経由)

OpenSpec の criterion 単位 MODIFIED / REMOVED / RENAMED は保留する。理由: owox には criterion の編集・
削除操作が無く (`crates/core/src/requirement.rs` は add_criterion と link_verification のみ・criterion 編集/
削除 tool が存在しない)、これらに対応する種別を実装しても発火する操作が無くデッドコードになる
(`docs/decisions/20260611-設計原則.md`「使わない抽象は作らない」)。将来 owox が criterion 編集・削除操作を
持った時に種別を拡張する。

OpenSpec の「適用順 追加 → 変更 → 削除 → 改名の固定順」は、変更・削除・改名 (MODIFIED/REMOVED/RENAMED) が
入る将来拡張時の方針として位置づける。現状の履歴は title/statement/status 変更 + criterion 追加のみで、
全順序を持つわけではない。

二重構造は維持する: 正本=現在状態 / なぜ=decision / 何が変わったか=requirement 内の構造化履歴。
新サブシステムは作らず requirement ドメインの責務を正しく広げる。

## 採用: E — 検証陳腐化の腐敗検知を採用方針とし、実装は次スライスへ

一度 `met` になった要件が後のコード変更で古くなる問題を検知する方針は採用する。
ただし実装は次スライスへ持ち越す。

前提未決: owox の検証 link はチェック名/コマンドでありファイルパスを持たない
(`crates/core/src/requirement.rs` の verify フィールド)。
link → ファイル解決の方法を詰めてから着手する。詰めずに実装するとノイズ警告を生む。

## 捨てた案

- カテゴリの description 接頭辞付与 → トークン増・発見性に貢献せず
- カテゴリをコードで列挙 → 追従で腐るリストを生む
- C / D / E を「仕様進化サブシステム」に束ねる → 3つの関心が機械を共有せず強制抽象=設計原則違反
- mode 物理統合 → 危険度の違う tool の統合は 20260620-MCPコンテキスト削減.md / 解釈A で既に却下
- SHALL/MUST 規範語強制 → 上記 C 理由参照
- 差分を decision.rationale テキスト埋め → 機械可読性を自分で潰す (上記 D 理由参照)
- criterion delta taxonomy (MODIFIED / REMOVED / RENAMED) の全実装 → 発火する操作が owox に無く
  デッドコードになるため保留。操作が増えるまで種別を足さない
- E を今スライスで実装 → link→ファイル解決が未決でノイズ警告を生む

## 見直し条件

- tools/list 起動時負荷が実測で再問題化 → mode 統合 (まず knowledge 読み系) を再検討
- requirement.rs の verify link → ファイル解決の方法が定まる → E の実装着手
- owox が criterion の編集・削除操作を獲得 → 変更履歴の種別を MODIFIED / REMOVED / RENAMED へ拡張
- C の基準0警告が厳しすぎて incremental 運用を妨げる → warning 止まりを維持しているか確認
