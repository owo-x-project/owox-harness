## 決めたこと

### MCP サーバ実装手段: 公式 rust SDK rmcp

`owox serve` は rmcp 1.7.0 (features: server / transport-io) で実装する。
transport は stdio。serve だけ tokio runtime を起こす (generate / hook は同期のまま)。

採用理由:

- MCP プロトコルは規模が大きく (initialize / capabilities 交渉 / resources / tools / notifications)、自作は Phase 1 型のスキーマ取り違えを招く。SDK が契約を担保する
- 公式 SDK なので Codex 側の挙動変化・プロトコル更新に追従しやすい
- ServerHandler トレイトに list_resources / read_resource を実装するだけで resource を出せる。実装が薄い

捨てた案: JSON-RPC over stdio を自作 (serde_json は既にある)。

捨てた理由:

- markdown パーサは自作した前例があるが、MCP は表面積が桁違い。自作は当面の節約より将来の保守・追従コストが勝る
- 縦貫スライスを速く正しく立てる方を優先

依存の代償 (技術スタックの「依存少なく単一バイナリ」と緊張):

- tokio (runtime) ・tower ・schemars 等が入る。ただし serve のみ runtime を使い、配布は従来どおり単一バイナリ
- 緊張は許容範囲。見直し条件: バイナリ肥大・ビルド時間が問題化した時

### tokio runtime: current-thread + time のみ

- stdin/stdout の非同期入出力は blocking pool 経由 (io-std)。IO driver (enable_io / net 機能) は不要
- rmcp 内部通信がタイマーを使うため time だけ有効化する
- 最小機能 (rt / macros / io-std / time) に絞る

## 縦貫スライスの実装

- `crates/mcp/src/serve.rs`: OwoxServer (ServerHandler 実装)。get_info で resources 能力 + instructions 告知、list_resources で owox://brand、read_resource で型付き Canon を描画して返す
- main.rs に `serve` サブコマンド追加
- 失敗は構造化して返す: 未知 URI → resource_not_found、正本欠落・壊れ → internal_error
- serve は起動時の作業ディレクトリから `.owox` を上方探索する (パス引数があればそれを優先)。生成 config にパスを焼かないための前提

### MCP サーバ登録の生成: config.toml へ MergeToml

`owox generate codex` が `.codex/config.toml` に `[mcp_servers.owox]` を出す。

- config.toml は Codex の共有設定 (人間が model 等を書く場所)。owox が丸ごと生成すると他設定を壊す
- そこで GeneratedFile に書込みモード (`Write::Overwrite` / `Write::MergeToml`) を追加。config.toml は MergeToml で「owox のブロックだけ」差し込む
- マージは既存 `toml` 依存で実装 (read → 深いテーブルマージ → write)。人間の他設定・他 mcp_servers は残す。コメント・並び順は失う (データは保つ)。出力は決定論的で冪等 (再生成で同じ結果)
- 生成ブロックは `command="owox" args=["serve"]` のみ。パスを焼かないので移植可能 (serve が `.owox` を上方探索)

捨てた案: config.toml を他生成物同様に丸ごと上書き。捨てた理由: 人間の共有設定を壊す。
捨てた案: toml_edit でコメント保持。捨てた理由: 新規依存。.codex は生成領域でコメント保持の必要が薄く、データ保持で足りる。

## AGENTS.md オリエンテーションと「向き付け > 制限」(2026-06-13)

実機で「AGENTS.md が正本の写しだけで harness 由来の振る舞い案内が無く、用語定義を探しに行く」
と判明。対応と方針:

- 生成 AGENTS.md 先頭に最小のオリエンテーション節を追加 (`render_orientation`)。正本の写しでなく
  振る舞いの案内: 正本は .owox/ (編集しない)、読む先は owox://context、用語定義は自動で届くので
  探さない・無ければ .owox/glossary.md。これで探索の無駄が消え、push 漏れの退避先も示せる
  - 旧判断「編集禁止の注記は置かない (setup/README へ)」を反転。理由: これは一度きりの導入情報でなく
    毎セッション要る振る舞い案内。数行で価値が高く、最小コンテキストを壊さない
- Bash リダイレクト書き込み (printf/cat > file) の push 取りこぼしは許容。Bash 全走査はノイズ源
  (commit メッセージ・echo に誤発火、context rot)。push は best-effort、用語名の気づき + 退避先で優雅に劣化
- `.owox` を読めなくして MCP 経由を強制する案は却下。事実: Codex のサンドボックスは書込み制御で
  読取は隠せない (read-only でも inspect 可、.git/.codex は書込み禁止であって読取禁止でない)。
  hook での読取 deny はいたちごっこ。加えて旗「検証可能な正本」に反し、マルチ CLI 移植性も壊す

原則: **harness の仕事は向き付け (orient) と先回り支援であって制限 (restrict) ではない**。
オリエンテーションが本丸で、Bash の隙を無害にし、.owox ブロックを不要にする。

## tool 層・記録層の実装 (2026-06-13)

詳細は `docs/decisions/20260613-Phase4-tool記録層.md`。実装・実機確証済み:

- 共通返り値 (封筒: status / reason / next_actions / decision_ids / data / gate) を core の型 (Serialize) で実装。mcp が JSON テキストで返す
- tool: decision.record / gate.list / gate.approve / verify.run。rmcp の `#[tool_router]` + `#[tool_handler]` で配線。get_info は自前 (resources + tools 告知)
- 記録層 `.owox/decisions/`: 1 判断 1 ファイル Markdown。ID=日付+slug。日付は mcp が UTC で求め core へ渡す
- 人間ゲート: gate=status=open の来歴。verify.run が完了3区別を返し要件完了を needs_human にする流れを実機 stdio で確認
- 計算 resource owox://next (status=open の来歴)
- hook と MCP の重複は core に集約 (verify は core::verify、hook も同じ core を呼ぶ)
