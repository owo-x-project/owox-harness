# 設計判断の種別タグ

## 状態

採用・実装済み。`docs/decisions/20260620-要件分類とPRFAQ正本.md`(技術・設計は要件でなく来歴へ)を
精緻化。design を専用層に割らず、来歴の種別タグで first-class にする。

## 背景

owox-harness の target harness では design 判断を来歴(decisions)に置く方針 (20260620)。だが来歴が
方針・方向・設計を区別なく抱え、「この要件の設計判断だけ見る」発見性が無い。専用 design 層案も出たが、
来歴の機構(ID・状態・根拠・代替案・supersede・リンク・ゲート・腐敗検知)を丸ごと再実装する重複を生み、
「新サブシステムは作らない」(`docs/decisions/20260611-方向付け.md`)に反する。構造制約そのもの
(依存方向・層境界)は quality.toml の適応度関数が既に機械強制しており、design は quality.toml と
来歴の2層で実は被覆済み。足りないのは発見性とトレースだけ。

## 採用: 来歴に種別タグ

`Decision` に `kind: Option<DecisionKind>`(design / direction / policy)を足す。層は分けず来歴機構を
共用する。要件で層を割らず `RequirementKind` タグにした前例 (20260620) に倣う。

- design: 構造・責務境界・依存方向・拡張性の選択
- direction: 製品の方針・スコープ・優先の選択
- policy: 手順・規約・運用ルールの選択
- 無タグ (None) も許す(既存来歴・自動生成来歴はそのまま)

役割分担を明確化:

- 構造制約そのもの(依存方向・層境界)→ quality.toml の適応度関数(機械強制)
- 設計判断(なぜこの構造か・捨てた案・見直し条件)→ 来歴 kind=design

実装: `DecisionKind` enum、`## Kind` 節の render/parse、`record_decision_full` に kind 引数
(proposed_change / authorizes と同じ別引数方式で RecordInput は不変)、`record_decision_with_kind` と
decision.record tool の `kind` 引数、get_decision の json と next の Open decisions に種別タグ表示。
design 入口 skill は kind=design で記録するよう本文を更新。

## 捨てた案

- 専用 design 層 (.owox/design/) → 来歴機構の重複・quality.toml と重複・新サブシステム方針違反
- design を要件 kind に足す → 20260620 で却下済 (機能/非機能と直交せず来歴と重なる)
- 無タグのまま放置 → 設計判断が他判断に埋もれ発見性・トレースが欠ける(本件の動機)
- RecordInput に kind を持たせる → proposed_change / authorizes は別引数で渡す既存パターンと不一致・
  内部呼出 16 箇所を触る churn。別引数方式に揃える

## 見直し条件

- direction / policy が使われず design だけで足りると分かった時(タグを design 1値へ絞る)
- 設計判断が来歴の中で多すぎ、独立層・独立ビューが要ると分かった時
- quality.toml と来歴 kind=design の境界 (機械強制 vs 判断) が曖昧になった時
