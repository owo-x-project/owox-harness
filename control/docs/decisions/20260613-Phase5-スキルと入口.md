# Phase5 スキルと入口

## 状態

採用。`docs/requirements/20260613-Phase5-スキルと入口.md` の未決を設計で確定したもの。
是正あり: 実機対話検証で見つけた粗 (入口の resource 依存・openai.yaml スキーマ・応答言語) は `docs/decisions/20260613-Phase5-実機検証の是正.md` で転換。入口は resource でなく tool を呼ぶ、openai.yaml は policy/interface を出す。

## 背景

Phase5 の3つ (setup / スキル / 入口) の構造を固める。
前提は一次情報 (OpenAI 公式) で確認した Codex の現状 (要件文書の背景参照)。
既存構造 (canon → Target → write_all、記録層の 1 ファイル 1 件、verify の run_checks) を土台にする。

## 確認した一次情報 (技術選定の根拠)

- Codex の custom prompts は廃止予定・ホーム限定 (`~/.codex/prompts/`)。repo 共有不可。公式が skills を推奨
- Codex native Skills: repo 内 `.agents/skills/` を作業dir→repoルートへ遡って探索。SKILL.md の必須 frontmatter は name と description のみ。起動は暗黙 (description 一致で自動) + 明示 ($名前・/skills)。テスト・検証機構は無い
- 自動起動の抑止は skill 内 `agents/openai.yaml` の allow_implicit_invocation (既定 true) で行う
- SKILL.md は横断標準 (将来のマルチ CLI 生成で同じ本文を流用できる)

## 採用設計

### スキル正本のレイアウト

`.owox/skills/<skill>/` を正本 (git 管理)。

- SKILL.md: 横断標準のまま。frontmatter は name / description、本文は手順。これがそのまま生成物の本体になる
- tests/: 実行可能なテスト。owox が各テストを実行し全て成功 (終了コード 0) で合格
- scripts/: 任意。skill が使う補助スクリプト。生成物へ同梱する
- memory.md: 任意。経験メモリ (自由文)。改善時に追記する。生成物へは出さない (正本のみ)
- owox.toml: owox 固有の設定。SKILL.md を標準のまま保つため、owox 固有項はここへ分離する
  - implicit: 自動起動を許すか (技系の既定 true、入口系は false)
  - human_gate: 正本昇格・不可逆操作を含むため人間確認が要るか (既定 false)

owox 固有項を SKILL.md frontmatter に混ぜず owox.toml へ出すのは、SKILL.md の横断標準性 (移植可能) とドメイン分離を守るため。

### テスト合格ゲート (登録の機械検証)

- ゲートは tests/ の全テストが終了コード 0。実行は verify の run_check を再利用する (検査再実行と同じ原理。重複ロジックを core に集約)
- 合格した skill だけ生成対象になる。未合格は `.agents/skills/` へ出さない (= Codex から見えない = 未登録)
- 実行はセッション中に AI が tool で起こせる (下記 skill.register)。作成 → テスト → 合格で登録 → 失敗なら改善、の輪が回る
- テスト必須の境界は implicit で分ける: implicit=true (技系・自動起動) は tests/ 必須 (書き忘れを弾く)。implicit=false (入口系・明示のみ) は tests/ 免除 (tests/ 無し=合格扱い)。薄い入口に無意味なテストを強制しない

### 生成 (正本 → Codex)

- core に skill モジュールを置く。責務: 正本読込 (load_skills。frontmatter 検証・owox.toml 読込、テストは走らせない)、テスト実行 (run_skill_tests、run_check 再利用)、登録集合の算出 (合格分の filter)
- SKILL.md 本文の描画は core で行う (横断標準なので CLI 非依存。将来のマルチ CLI が再利用できる)
- 出力先と CLI 固有ファイルは Target が持つ。Codex は出力先 `.agents/skills/`、implicit=false の時だけ `agents/openai.yaml` に allow_implicit_invocation=false を出す
- Target を拡張する (skill 出力の口を足す)。既存の generate(canon) は canon ファイル (AGENTS.md / hooks / config) 専用のまま。skill は別経路 (テスト結果で filter 後に渡す) なので分ける

### 入口 (slash 相当) = 薄い skill

- commands.toml を正本に置く (名前・別名・呼ぶ MCP tool / resource・説明)
- 各コマンドを `.agents/skills/<command>/` の薄い skill として生成する。本文は対応する MCP tool / resource を呼ぶトリガ文
- 入口系は明示起動のみ (implicit=false → openai.yaml に反映)。誤って自動起動させない
- 入口系 skill はテスト不要 (owox 同梱の固定トリガ。tests/ 無しは合格扱い)
- namespace 既定 + 別名。プロジェクトが commands.toml で追加できる
- 第1段階: /kickoff /next /status /decide /verify /review /task /handoff

### ライフサイクルの状態と昇格 (人間ゲート)

skill は 2 軸を持つ。テスト軸 (failing / passing) と昇格軸 (draft / registered / promoted)。

- draft: 作成済み。テスト未合格 → `.agents/skills/` へ出さない (Codex から見えない)
- registered: テスト合格 → 生成され $名前 の明示起動で使える。ただし AI が育てた実験段階
- promoted: 人間が承認した昇格状態 (= 正本への昇格。`docs/decisions/20260611-設計原則.md` の人間ゲート)

owox.toml の implicit は「自動起動してよいという意図」。実際の自動起動 (openai.yaml allow_implicit_invocation=true) は implicit=true かつ promoted の両方で初めて解禁する。人間が信頼するまで勝手に発火させない。これで判断 (技系 implicit 既定 true) を昇格ゲートで安全に縛る。

- skill.promote: 昇格させる人間ゲートの口。gate.approve と同じく人間承認後に使う。promoted へ遷移し、来歴へ記録し、openai.yaml を再生成する
- 入口系 (owox 標準・implicit=false) は昇格不要 (自動起動しないため)

### スキルの改善 (経験メモリ更新)

- skill.remember: `.owox/skills/<名前>/memory.md` へ追記する。task.note と同型 (一時でなく skill の経験)。`.owox` 直読み禁止と整合し tool 経由でのみ触る
- memory.md は生成物へ出さない (正本のみ)。鮮度・腐敗検知は Phase7
- テスト失敗・実行誤りの学びを memory.md へ残し、再生成 (改善) を導く

### カタログ = tool (resource にしない)

- skill.list: 正本 `.owox/skills/` を走査し各 skill の 2 軸 (テスト: passing/failing、昇格: draft/registered/promoted) を返す。Codex は登録済みしか見えず、状態は「ファイルを読んでも得られない」ため tool で出す
- skill.register: 指定 skill のテストを実行し、合格なら `.agents/skills/` へ生成 (登録)、失敗なら失敗を封筒で返す (改善の輪)。ゲートをセッション中に起こす口
- skill.promote / skill.remember は上記ライフサイクル・改善節のとおり
- owox://skills resource は作らない (登録済みは Codex が native に列挙する。状態は skill.list が持つ)

### glossary 動的化 (tool 化 + session 注入)

用語集を静的 (AGENTS.md 列挙 + 不足時 `.owox` 直読み) から動的 (tool で CRUD・読取) へ移す。

- glossary.lookup: 用語の意味を引く。直読み禁止の読み経路
- glossary.add: 用語を追加する。語彙の追加は低リスクのため AI 可。来歴へ記録する
- glossary.update / glossary.remove: canonical な意味の変更・喪失は人間判断。open gate を立て gate.approve を要する (ブランドは人間が握る)
- 出現時の定義 push (UserPromptSubmit / PreToolUse) は維持する (能動的な最小注入)
- AGENTS.md の用語名列挙は廃止する。代わりに SessionStart hook が live な用語名を注入する (CRUD で変わっても古びない)

### canon 直読み禁止

Phase4 の「prose 正本は直読み可」(`docs/decisions/20260611-MCP設計.md`、serve.rs の get_info) を見直す。指針 canon の読み込みは tool / resource / session 注入へ一本化する。

- 対象: 指針 canon (brand / rules / context / glossary / decisions / tasks / state) の読み込み
- 読み経路は網羅する: brand・rules は AGENTS.md (常時)、context は owox://context、glossary は session 注入 (用語名) + 出現時 push (定義) + glossary.lookup、未決 decision・ready task は owox://next
- 例外: `.owox/skills/` は AI の著作・読み返す作業域。skill 著作はファイルで行うのが自然なため禁止対象から外す
- 強制の現実: Codex で読みを hard に止めるのは部分的 (PreToolUse は Bash/apply_patch/Edit/Write を拾うため `cat .owox/...` は deny できるが、内蔵閲覧が hook を通らない経路は止めきれない)。よって禁止は指示 (AGENTS.md・get_info から直読み案内を消す) + tool 提供 + Bash 読み deny が主で完全保証ではない
- 既知の穴: 過去の adopted decision 本文・task 詳細を読む tool が無い。M1 では使用頻度低 (owox://next が未決を出す) のため穴として記録し、必要になれば read tool / resource を足す

### setup (先行スライス)

- `owox setup [dir]`: 正本読込 → 生成 (canon ファイル + 登録済み skill + 入口 skill) → 検査 → 報告
- 検査: owox が PATH 上にあるか、生成 config が妥当か、hooks が認識される形か
- バイナリの install は範囲外 (配布は Phase9。GitHub Releases + setup.sh + checksum)。setup は owox 配置済みを前提に設定生成 + 検査に絞る
- 既存 generate は純粋な生成サブコマンドのまま残す。setup は generate + 検査 + 案内のまとめ

## 不採用

- 廃止予定の custom prompts (`~/.codex/prompts/`) 生成 — repo 共有不可・ホーム限定・将来破綻
- owox 固有項を SKILL.md frontmatter へ混在 — 横断標準性とドメイン分離を損なう
- カタログを resource に — 登録済みは Codex native、未登録状態は tool が適切
- skill を Canon へ載せ generate(canon) で出す — テスト実行 (副作用) を純粋な読込・生成に混ぜることになる
- テスト合格で即 implicit 有効 — 人間承認前に AI 育成 skill が自動発火する。昇格ゲートで縛る
- glossary を AGENTS.md に静的列挙 — 動的 CRUD で古びる。session 注入で live に
- 全 canon 読みを完全禁止 (skill 著作も含む) — skill 著作はファイルが自然。指針 canon の読みに絞る

## 段階分け (着手順 setup → スキル → 入口 → glossary 動的化)

1. setup: `owox setup` (既存生成 + 検査 + 案内)。別 target repo へ反復導入でき以降の実機検証が楽になる
2. スキル: skill 正本型 + owox.toml + テストゲート + 生成 + skill.list / skill.register / skill.promote / skill.remember
3. 入口: commands.toml + 入口 skill 生成 (implicit=false)
4. glossary 動的化 + canon 直読み禁止: glossary.lookup / add / update / remove、SessionStart の用語名注入、AGENTS.md と get_info の直読み案内除去、Bash 読み deny

各段階の生成物は setup が束ねる (後続スライスが setup へ繋がる)。

## 変更境界

触る:
- crates/core: 新規 skill モジュール・commands モジュール、target.rs / targets/codex.rs (skill 出力の口・AGENTS.md の用語名列挙除去)、hook.rs (SessionStart の用語名注入)、glossary CRUD、lib.rs (re-export)
- crates/mcp: 新規 setup サブコマンド・main.rs、serve.rs (skill.list / skill.register / skill.promote / skill.remember / glossary.lookup / glossary.add / glossary.update / glossary.remove、get_info の直読み案内除去)、hook.rs (Bash の `.owox` 読み deny。`.owox/skills/` は除外)
- 正本側: `.owox/skills/`・commands.toml の例 (target 検証用)

触らない:
- 既存の canon 型・load_canon・記録層・envelope の振る舞い (glossary 書込は追加、読込は不変)
- 既存 generate サブコマンドの出力構造の互換 (用語名列挙除去を除く)

守る振る舞い:
- 既存生成物が壊れない (config.toml の MergeToml 維持)
- verify / task.close の検査再実行ゲートと同原理でテストゲートを実装 (古い結果で通さない)

## 危険

- Codex の skills は実験的機能 (フラグ・config で有効化が要る場合がある)。実機で有効化手順を確認し setup の検査・案内へ反映する
- skill のテスト実行は任意コマンド。可逆性・サンドボックスの扱いを既存の不可逆 deny と整合させる (テスト名目の破壊的操作を防ぐ)
- `.agents/skills/` を owox が生成管理する一方、人間・他ツールが同 dir へ置く skill と衝突しうる。owox 管理分の境界 (owox 生成印) を design 実装で明示する
- canon 直読み禁止は Codex で完全強制できない (上記)。glossary の読み経路 (lookup・push・session 注入) が漏れると AI が定義を得られず詰まる。読み経路の網羅を検証で確かめる
- glossary.update / remove の人間ゲートが形骸化すると AI がブランドを書き換える。open gate を立てる実装で縛る

## 実装で確定した運用前提

- skill の tests/ は実行可能 (+x) が前提。テストは絶対パスを sh で起動する。git は実行ビットを保つため正本作成時に +x を付ける
- 入口コマンド名 (kickoff/next/status/decide/verify/review/task/handoff) は owox 標準が予約。同名の育成 skill とは生成時に名前衝突する (最後勝ち)。プロジェクト追加コマンドも同様
- glossary.update / remove は canon を変えず open 決定 (gate) を立て needs_human で止める。人間が glossary.md を編集し gate.approve で解決する (AI はブランドを直接書き換えない)
- openai.yaml は登録 skill ごとに常に出し allow_implicit_invocation を明示する (再生成で古い値を残さない)
- canon 読み deny は内容読みコマンド (cat/grep/head 等) が `.owox/` 非 skills を指す時のみ。列挙 (ls) や `.owox/skills/` は通す (保守的・指示の補助)

## 検証方針

- 単体: frontmatter 検証、owox.toml 解釈、テストゲート (合格/不合格で生成有無)、昇格で implicit 解禁、commands → skill 生成、SKILL.md 描画、glossary CRUD (add 即時・update/remove は open gate)、SessionStart の用語名注入
- 結合: setup で canon + skill + 入口が出る縦貫、未合格 skill が出ない、昇格前は自動起動しない
- 手動 (実機 Codex): setup 後に skill が /skills に出る、$next 等が MCP を引く、暗黙起動が入口系・未昇格で起きない、用語名が session 開始で届く、glossary.lookup で定義を引ける、第1段階の一通りが回る
- 未確認として残す: Codex skills 実験フラグの実機挙動、`.owox` 直読み deny の実機効き具合、横断 CLI での SKILL.md 流用 (Phase8)

## 見直し条件

- Codex の skills 仕様が変わった時 (配置・有効化・openai.yaml)
- owox.toml の項目が増えて SKILL.md との分担が曖昧になった時
- 入口 skill とプロジェクト追加 skill が衝突した時
- テストゲートが重く skill 作成が滞った時
- 昇格ゲートが形骸化し未承認 skill が自動発火するようになった時
- canon 直読み禁止で読み経路の穴 (過去 decision 本文等) が実作業を妨げた時
- glossary CRUD の人間ゲートが誤検出・摩擦で運用を妨げた時
