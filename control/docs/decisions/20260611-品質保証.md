# 品質保証

## 状態

採用済み。

## 背景

AI が書いたコードには固有リスクがある (通るが実は何もテストしてない嘘テスト、スコープ逸脱、規約・既存資産の無視)。
標準 CI ではこれらを捕まえきれない。
旗 (検証可能な正本) を品質へ適用する。
既存の verify (完了3区別) ・適応度関数・hook・来歴・state に載せ、テスト道具自体は再発明せず orchestrate する。

- `docs/decisions/20260611-製品戦略.md`
- `docs/decisions/20260611-設計原則.md`
- `docs/decisions/20260611-タスク管理.md`

## 立ち位置

owox は検証可能な品質契約のエンフォーサ。

- agent 実行ツール (OpenHands / Aider / Codex 等) の内側 TDD ループの上に立つ
- CLI / agent 非依存。どの agent も owox 契約内で働く
- 検証ラダーの最上段 (spec 駆動検証) を、型付き正本+品質の深さで強化する
- spec 駆動勢 (BMAD / Spec Kit / OpenSpec) より機械検証が深い (lint/test 止まりにしない)

## 採用

### コア (AI 時代の決め手)

- plan-alignment ゲート: 「規約・既存資産に沿ったか」を適応度関数+規約整合で機械検証する (PEV 由来)。テストでは見えない AI 固有の逸脱を捕まえる
- 要件↔テスト trace + 受け入れ基準: 各要件に GIVEN/WHEN/THEN の受け入れ基準と検証テストを link し、欠落を機械検出する。正しい物を作ったかを保証する

### 基盤・補強

- 品質ゲート編成: 既存の lint / test / typecheck を verify チェックにし、done に緑を必須化する。道具は orchestrate し再発明しない
- 品質の適応度関数: 依存方向 / 層境界 / 循環禁止 / 複雑度予算を機械検証する
- 回帰の自動捕捉: バグ発見時に回帰テストを来歴 link で必須記録する。保守状態で厳格
- 多視点+敵対レビュー: merge 前に複数視点で指摘 → 事実性確認 → 別観点で再確認する owox 標準 skill (企画書 6.4)

## 検証は2系統 (定義場所が違う。2026-06-13 追記)

検証を一律に config.toml へ書かせない。性質で2系統に分ける。

- 決定論チェック: 通る/落ちるがコマンドで決まるもの (lint / test / typecheck / build、コマンド化できる E2E)。人間が config.toml の `[[verify.checks]]` に列挙する。verify.run がこれを実行し検証完了を機械判定する (`docs/decisions/20260613-Phase4-tool記録層.md`)
- 判断型レビュー: 合否一発で決まらないもの (セキュリティ / アーキ整合 / 規約逸脱 / plan-alignment)。多視点+敵対レビューを担う subagent が行う。観点・進め方の枠組みは owox 標準が持つ。プロジェクトは具体 (層境界・複雑度予算・許す依存・規約) だけを quality.toml で宣言する

役割分担: レビューの枠組み・観点は owox (堅牢さの担保)、プロジェクト固有の具体は人間。多視点レビューをユーザー任せにしないのはこの分担のため。完了3区別が「機械検証」と「判断が要るもの」を分けているので、両系統は素直に乗る。

## 前提

- state 適応の厳格度: 初期=緩い、保守=厳格。ただしコアゲート (test/build/typecheck) は常に維持する
- 人間が品質バー (必須チェック・受け入れ基準・複雑度予算・規約) を宣言し、owox が verify / 適応度関数 / hook で機械強制し、done に紐付け、来歴で追跡する
- 品質バー、依存方向、層境界、循環禁止、複雑度予算は quality.toml に置く
- rules.toml は「いつ止めるか」、quality.toml は「何を品質制約として守るか」を持つ

## 実装

- すべて verify / 適応度関数 / hook / skill に載せる。新サブシステムは作らない
- 重い検証は変更箇所・merge 前・CI で走らせ、state で厳格度を変える
- 専用増分で実装する

### 機械検証

- 型検証: quality.toml の必須項目、値の種類、閾値の範囲
- 実行検証: lint / test / typecheck / build の成功
- 構造検証: 依存方向、層境界、循環禁止、禁止 import
- 予算検証: 複雑度、ファイル肥大、差分量が上限内か
- trace 検証: requirement の受け入れ基準と verification が link されているか
- 回帰検証: bug 修正に回帰テストまたは人間判断の来歴があるか

## 見送り (将来再検討)

- ミューテーションテスト (嘘テスト検出。cargo-mutants)
- 不変条件宣言+property test (proptest)
- 決定論的シミュレーションテスト (DST。stateful / 並行 / 分散向け)

理由: コアの確立を優先する。いずれも重く、適応度関数・verify の枠組みが固まってから載せる方が安全。

## 採用理由

- plan-alignment は、テストで見えない AI の逸脱を機械で捕まえる
- 受け入れ基準の trace は、AI が正しい物を作ったかを要件と結ぶ
- 品質ゲート編成は、既存資産を再発明せず done に縛れる
- 適応度関数は、アーキ制約を文章でなく機械で守る
- 回帰捕捉は、再発を防ぐ
- 多視点レビューは、単一視点の見落としと曖昧ループを減らす

## 捨てた案

- 標準 CI だけに頼る
- テスト道具を owox が独自実装する
- 重い検証 (mutation / DST) を最初から必須にする
- 品質バーを owox が勝手に決める

## 捨てた理由

- 標準 CI は AI 固有の問題を捕まえきれない
- 独自実装は再発明で、対象外の汎用道具化に近づく
- 重い検証の即時必須化は初期で詰まる
- 品質バーは人間が決める領域 (判断は人間)

## 見直し条件

- 見送った検証 (mutation / property / DST) を載せる判断がついた時
- plan-alignment の規約整合が誤検出で作業を妨げた時
- 受け入れ基準の必須化が重く、運用が滞った時
- 適応度関数が増えすぎて検証が重くなった時
- レビュー skill のループが収束しなくなった時
