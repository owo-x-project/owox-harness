# Phase7 スライス4: 経験 export/import + 二層ルール + 状態適応の設計

## 状態

採用・設計確定。実装はこれから。Phase7 スライス4 (`docs/requirements/20260614-Phase7-腐らず成長する.md`)。
スライス1・2・3 は実装・機械確証・コミット済み。
本スライスは当初の「経験 export/import + 状態適応」へ、人間判断で「二層ルール + ブランド変更提案ツール」を統合する (scope は人間が統合を選択)。

## 背景

経験を別プロジェクトへ持ち込めるようにし、状態適応が再起動後も効くことを実機で確かめる。
設計中に人間が経験資産モデルの中核を提起した: ブランド価値として固定し変更に人間判断が要るルールと、経験から育つルールを別管理にする。加えて現状 AI がブランドを変更提案する経路が無い欠落を埋める。

人間判断 (plan の論点) を確認し確定した:

- 経験 export の汎用範囲は SKILL.md + scripts (経験で磨かれた instructions は SKILL.md に育つ)。memory.md・tests・owox.toml はドメイン固有・ライフサイクル状態として型で除外
- export の人間ゲートは「クリーンなら書出 + needs_human」。秘密検出時は failed で書かない
- 経験から成長するルールは固定 rules.md とは別ファイルの成長層で持つ (AI 可追記・鮮度 decay・export 可)
- 二層ルール + canon.propose を本スライスへ統合する

## 採用設計

### 二層ルール

正本のルールを二層に分ける。固定層は人間が握り、成長層は AI が育てる。

#### 固定層 (人間が握る)

brand.md (価値・方針) + rules.md (変更方針・安全・不可逆) + glossary 用語。プロジェクトの個性。変更は人間判断。
glossary は既に add (AI 可) と update/remove (人間ゲート) を持つ。brand.md/rules.md は変更経路が無かった。

#### 成長層 (AI が育てる) = practices.md

`.owox/practices.md` を新設する。経験から蒸留した運用指針を固定 rules.md とは別に持つ。

- 形式は memory.md と同じ `- <日付>: <指針>`。日付で鮮度を測る (古さは見直し合図・捨てる根拠にしない。`docs/decisions/20260611-設計原則.md`)
- model は `Practices { entries: Vec<Practice{date, text}> }`。load_canon が読む (任意・無ければ空)。Canon へ practices フィールドを足す
- 追記は practice.add (AI 可・低リスク・来歴記録)。glossary.add と同じ思想 (語彙を増やすだけは AI 可)
- 届け方は session_start_context のライブ注入 (静的 AGENTS.md へ列挙しない)。glossary 用語名・phase 案内と同じ live 経路で、再生成なしに育ち古びない。最小コンテキストは人間の手入れと鮮度 decay で保つ (肥大化したら要約圧縮は将来。`docs/decisions/20260611-設計原則.md`)

### ブランド変更提案ツール canon.propose (欠落を埋める)

AI が固定層 (brand.md/rules.md) の変更を提案する経路を足す。glossary.update と同じ型。

- `canon.propose { target, change }`。target は brand / rules。change は提案本文
- open 決定 (gate) を立て needs_human で止める。canon は変えない。人間が canon を編集し gate.approve で解決する (ブランドは人間が握る)
- これで「固定だが経験で見直しを促せる」が成立し、鮮度 decay の見直し合図 (古い来歴・古い practice) と繋がる
- core の `proposal` モジュールに置く (glossary::propose_change が手本)

### 経験 export/import (経験束 = 複数 kind)

core の `experience` モジュールに export/import を置く。経験束は複数 kind を持てる形にし、skill kind と practices kind を運ぶ。

#### 束のレイアウト

- `<bundle>/skills/<id>/SKILL.md` と `<bundle>/skills/<id>/scripts/*` (汎用: 手順 + 道具)
- `<bundle>/practices.md` (汎用: 成長層)
- 除外 (型で混入を防ぐ): memory.md (このプロジェクトの経験・固有名詞)・tests (このプロジェクトの検証)・owox.toml (昇格状態)・brand/rules/glossary (固定層 = 個性)

#### export

- `experience.export { out_path }`。skill (登録済) の SKILL.md + scripts と practices.md を束へ集める
- 本文を secret::scan (スライス3 の再利用) で走査。秘密検出時は failed で書かない (持ち出しに秘密を載せない)
- クリーンなら out_path へ束を書き出し、needs_human で「外部共有前に人間レビュー」を促す + 来歴記録 (持ち出しは人間ゲート。`docs/decisions/20260611-MCP設計.md`)。書出は out_path のローカルで可逆

#### import

- `experience.import { in_path }`。束の SKILL.md + scripts と practices.md を読む
- 本文を secret::scan。秘密検出時は needs_human で止め import しない (open gate を立てる)
- クリーンなら skill を `.owox/skills/<id>/` へ draft として書く (owox.toml を持たせない = 昇格状態は移植先で改めて人間が握る・tests 無しなので登録は移植先で再評価)。practices.md は既存へ追記マージ。imported 一覧を返す

### 状態適応の回帰防止

maintenance の振る舞い案内が再起動後も効くことを実機で確かめる。

- session_start_context は canon.state.phase を読み phase_guidance を注入する (実装済)。phase は `.owox/state.toml` に永続し session 開始ごとに再読込される
- 本スライスは回帰テストを足し、state.toml が maintenance の時に session-start hook が maintenance 案内を返すことを機械確認する (新規プロダクションコードは無し・既存挙動の固定)

### 鮮度 decay の拡張

practices の鮮度を decay へ載せる。

- core に `run_practice_decay(&practices, days, today) -> Vec<DecayFinding>` を足す (kind "stale-practice")。review_decision_days を閾値に古い practice を見直し合図として報告する (run_code_decay と同じく別関数・run_decay 署名は不変)
- mcp が verify.run と next で run_decay と run_practice_decay を結合する。is_structural()=false なので commit は止めない (鮮度は助言)

### 責務分担

- core: Practices モデルと load、practices::add、proposal::propose_canon_change、experience::export/import、run_practice_decay、session_start_context への practices 注入
- mcp: practice.add / canon.propose / experience.export / experience.import の各 tool、verify.run と next で practice decay を結合

## 危険

- 成長層 practices が肥大化し session を汚す。最小に手入れ + 鮮度 decay の見直し合図で抑える。要約圧縮は将来 (`docs/decisions/20260611-設計原則.md`)
- export の秘密走査の見逃し。組込みパターンは保守的で完全でない。クリーンでも needs_human で人間レビューを必ず通す
- import した skill が tests 無しで登録扱いになりうる。owox.toml を持たせず draft とし、移植先で人間が tests 追加・昇格する
- canon.propose の乱発で open gate が溜まる。glossary.update と同じく来歴に残り gate.list で見える。保守 phase は open gate が commit を止めるので滞留は表面化する
- 二層の境界が運用で曖昧化 (固定 rules と成長 practices の使い分け)。固定 = 人間が握る個性・成長 = AI が経験で育てる運用指針、と用語集と本設計で定義する

## 検証方針

- 単体: Practices の from_markdown (日付つき項目の収集)・practices::add の追記と来歴・proposal の open gate と canon 不変・experience::export の秘密検出で書かない/クリーンで書く+needs_human・import の秘密検出で止める/クリーンで draft 書き+practices マージ・run_practice_decay の古さ判定・session_start_context の maintenance 案内
- 結合: practice.add → practices.md 追記 → session_start で注入。canon.propose → open gate → gate.list。export → out_path に束 (skills + practices.md、memory/tests/owox.toml 除外)。import → .owox/skills へ draft + practices マージ。verify.run/next に stale-practice が出る
- 手動 (後の一括対話検証へ): 状態適応の再起動実見 (state.toml=maintenance → 別 session で maintenance 案内)、経験を別サンドボックスへ export→import、秘密入り skill の export が止まるか
- cargo test / clippy / fmt
- 未確認として残す: practices の要約圧縮 (肥大時)・rules/instructions kind 以外の経験 kind・glossary 用語照合による domain 漏れ検査 (秘密走査で代替・将来)・export の item 選択

## 捨てた案

- 成長ルールを skill に寄せる → テスト無しの軽い指針を毎回 skill 化するのは重い。別ファイルの成長層が「別管理」の意図に忠実
- 成長層を持たず canon.propose だけ → 「経験で育つルール」が成立しない。AI が握れる成長層を分ける
- practices を静的 AGENTS.md へ列挙 → 成長で肥大し最小コンテキストに反する・再生成が要る。session_start のライブ注入にする (glossary と同じ)
- run_decay へ practices を足し署名変更 → 呼出元 (テスト含む) が多く波及。run_code_decay と同じく別関数にする
- export に brand/rules/glossary を含める → 固定層は個性で持ち出さない。汎用は skill + practices のみ
- export で常に open gate → 承認 → 再実行で書く 2 段 → 手数が多い。クリーンは書出 + needs_human の 1 回・秘密は failed
- import した skill を即登録 → 移植先の検証を飛ばす。draft で入れ移植先で人間が昇格

## 追補: 統一 canon 編集モデル (スライス4 拡張)

人間判断で、固定層・成長層をまたぐ canon 編集を 1 つのモデルへ統一した (glossary を雛形に一般化)。
brand / rules / practices / glossary すべてが同じ作法で編集できる。

### 一律の作法

- add (追加): AI 直接 + 来歴記録。追加は既存の方向を壊さない低リスク。glossary.add・practice.add の現状と一致
- change / remove (変更・削除): AI 提案 → open gate → needs_human。人間が canon を編集し gate.approve。canon は直接書かない

二層 (固定 / 成長) は権限差でなく内容差になる: 固定 = アイデンティティ・制約・語彙、成長 (practices) = 経験で育つ運用指針 (鮮度 decay つき)。編集の作法は 4 つとも同一。

### ツール面の集約

scattered な glossary.add/update/remove・practice.add・旧 canon.propose を 2 つへ畳む (最小コンテキスト・一貫性)。

- canon.add { target, section?, text }: AI 直接追加 + 来歴。target = brand / rules / practices / glossary
  - brand: section = values / principles / non-goals / success-criteria / style、text = 項目
  - rules: section = change-policy / dependency-policy / deletion-policy / safety、text = 項目 (irreversible / human_gate は機械強制・構造化のため add でなく propose 経由 = 安全側の例外)
  - glossary: text = "用語: 定義"
  - practices: text = 指針 (日付は自動)
- canon.propose { target, change }: 変更・削除を提案 → 人間ゲート。target = 4 種すべて
- glossary.lookup は読みとして残す

内部は既存の per-target 関数 (glossary::add・practices::add・brand/rules への追記・各 propose) を再利用し、MCP のツール面だけ統一する。core は `canon` モジュールへ集約 (proposal.rs を畳む)。

### この追補での変更

- crates/core/src/canon.rs (新設・proposal.rs を畳む): canon_add (4 種へ dispatch・brand/rules は見出し配下へ追記)・canon_propose (4 種の変更/削除提案)
- crates/core/src/lib.rs: canon_add / canon_propose を公開、proposal の公開を畳む
- crates/mcp/src/serve.rs: canon.add / canon.propose の 2 tool へ集約、glossary.add/update/remove・practice.add・旧 canon.propose を撤去 (glossary.lookup は残す)、案内文を更新

## 見直し条件

- 成長層 practices が肥大し要約圧縮の機構が要る時
- 経験 kind を rules/instructions 以外へ広げる判断がついた時
- canon.propose を brand/rules 以外 (例: quality・config) へ広げる必要が出た時
- 秘密走査の見逃し/誤検出で組込みパターンの調整・target 拡張が要る時
- 二層の使い分けが運用で曖昧になり境界の再定義が要る時
