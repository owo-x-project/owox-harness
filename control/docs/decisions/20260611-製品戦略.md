# 製品戦略

## 状態

採用済み。

## 決めること

- 対象ユーザへ価値を届ける形
- 競合に対し前面に立てる旗
- 旗を補強する差別化軸

## 判断基準

- 企画書・要件の核と一貫する
- 競合との差が明確
- 旗を1本に絞り、薄まらない
- 補強軸が旗を裏打ちする

## 採用案

### 実現アプローチ

ハイブリッド。

- 静的: 共通正本から型付き生成し、各 AI CLI 向け target harness を出す。少ない文脈で正しく動く土台。どの CLI でも読める
- 動的: MCP で、判断記録、検証実行、経験蓄積、ルール成長を扱う。機械記録・検証が要る部分だけ動的にする
- 初回導入: setup で配置と設定補助だけ行い、常駐しない

### 中心の旗

検証可能な正本。

ルール・判断・経験が、機械検証つきで腐らず成長する。

### 補強する差別化軸

- マルチ AI CLI 生成
- 判断・経緯の機械記録
- 型付き正本による構造保証
- 完了の3区別 (作業完了 / 要件完了 / 検証完了)
- 常に最小のコンテキスト消費
- プロジェクトに合わせた自己成長
- 経験の export / import (汎用のみ、ドメイン固有情報を混ぜない)

## 理由

- 企画書の核 (少ない文脈で正しく動く / 判断と記憶を残す / 検証つき成長) を、静的と動的へ素直に割り振れる
- 静的は CLI 非依存、動的は規律を機械化、で両取りできる
- 競合は「生成だけ」か「検証なし」か「特定 IDE 寄り」のどれか。検証つき+移植可能で独自性が出る
- 旗を「検証可能な正本」1本に絞ると、他の軸が全部その裏打ちになり、説明が一貫する
- 経験の export / import は競合にほぼ無く、複数プロジェクト横断の価値を生む

## 競合の位置

詳細な競合調査は `docs/competitive/` に置く。

- 単一 CLI 正本ファイル: 1つの CLI に閉じる、自由記述、検証なし
- マルチ CLI 配布ツール: 生成は被るが、判断記録・検証・成長が無い
- spec 駆動・ワークフロー: 要件から実装は強いが、特定 IDE 寄り、正本の CLI 非依存性が弱い
- 記憶・メモリ系: 記憶は残すが、検証つき・ドメイン分離が弱い

## 捨てた案

- 静的生成のみ
- MCP 常駐仲介中心
- 旗を複数同時に立てる
- マルチ CLI 生成を旗にする

## 捨てた理由

- 静的生成のみは、検証つき成長が AI の記述頼みで規律が緩む
- MCP 常駐仲介中心は、MCP 対応 CLI に依存し、CLI 非依存の理想と矛盾する
- 旗を複数立てると主張が薄まり刺さらない
- マルチ CLI 生成は必須機能だが既存ツールと同土俵で、差別化にならない

## 見直し条件

- 競合が検証つき正本へ追随した時
- 静的と動的の役割分担が複雑化した時
- 経験の export / import が想定ほど使われない時
- 旗のメッセージが現物の機能とずれた時
