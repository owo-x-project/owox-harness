# 設計原則

## 状態

採用済み。

## 背景

製品戦略の差別化軸を、具体的な仕組みへ落とす設計原則。
`docs/decisions/20260611-製品戦略.md` を補強する。
仕組みの置き場所・型は `docs/decisions/20260611-開発アーキテクチャ.md` に従う。

## 採用する原則

### メモリ階層化

- 記憶を3層に分ける
  - 会話 (短期): 正本化しない
  - 作業 (タスク中): 引き継ぎ用、git 管理対象外
  - プロジェクト (長期): .owox/ で正本化、git 管理
- 長期が肥大化したら要約して圧縮する
- スキルごとに経験メモリを持つ (スキル・ライフサイクル参照)
- 記憶に時間と鮮度を持たせる (Graphiti の知識グラフから採用)
  - 来歴・経験に「いつの判断・経験か」を残す (ID が 日付+slug なので時間は既にある)
  - 古さを腐敗検知へ連動し、鮮度の落ちた判断・経験を警告する (タスク管理の腐敗検知と同じ機構)
  - 時間軸は記憶を捨てる根拠でなく、見直しを促す合図にする
- 目的: コンテキストを賢く節約し、精度と持続性を上げる。古い前提のまま作業する事故を防ぐ

### マルチエージェント

- target harness に役割別 subagent 定義を生成する (調査 / 実装 / レビュー / 検証)
- 各 subagent は限定された文脈だけ受け取る (節約)
- レビューは複数視点で指摘 → 事実性確認 → 別観点で再確認
- 目的: コンテキスト節約、精度向上、持続性
- 詳細は target harness 内容の設計で具体化する

### 可逆性で自律境界を決める

- 操作を可逆 / 不可逆で分類する
  - 可逆 (取り返しがつく): AI 自律
  - 不可逆 (本番削除、公開、破壊的変更): 人間ゲート
- 不可逆操作の定義を .owox/ に型で持つ
- 初期定義は保守的な最小集合に絞る: 削除系 / push・公開系 / 秘密情報・外部送信系
- 判定できない操作は不可逆として扱う (安全側既定)
- MCP は不可逆操作を人間判断が必要な点として返す
- 目的: 人間判断ゲートに、機械化できる明快な基準を与える

### 適応度関数

- アーキ・プロジェクト制約 (依存方向、層の境界など) を機械検証ルールにする
- 機械検証の一種として core の検証に乗せる
- プロジェクト状態の判定にも使う
- 目的: 制約を文章でなく機械で守る

### 機械検証できる判断記録 (来歴)

- 判断に ID、状態、根拠、リンクを持たせる
- リンクで 要件 ↔ 作業 ↔ 検証 を繋ぐ
- 型で構造化し、MCP から追記する
- 機械で追跡・検証できる
- 目的: 判断・経緯を追跡可能・機械記録可能・検証可能にする

### 回帰防止

- プロジェクト状態 (初期 / 安定 / 保守) を .owox/ に持つ
- 状態に応じ AI の作業姿勢を変える
  - 初期: 大胆な再設計を許す
  - 安定: 小さく安全な変更
  - 保守: 回帰防止優先、既存検証を必ず通す
- 状態は人間が宣言する (state.set、来歴に記録)。宣言が正
- 適応度関数は現物と宣言の乖離を検知して警告する。状態は変えない
- 目的: プロジェクト状態に合わせ、回帰を防ぐ

### スキル・ライフサイクル

- 経験の具体形をスキルにする (MUSE-Autoskill を owox 流に採用)
- スキルは純テキストのまとまり: SKILL.md (用途・入出力・手順) + tests/ + scripts/ + 経験メモリ
- .owox/skills/ に置き、git 管理する
- ライフサイクル: 作成 → 評価 → 登録 → 改善
  - 評価: 同梱テスト合格を登録ゲートにする (機械検証)
  - 改善: テスト失敗・実行誤りで再生成、経験メモリが改善を導く
- owox 流の上乗せ:
  - 人間ゲート: 正本への昇格、不可逆操作を含むスキルは人間確認
  - 型検証: 純テキスト+型で構造保証
  - ドメイン分離: export 型にドメイン固有項を入れない
- カタログは frontmatter だけ読む進行的開示で出す (最小コンテキスト)
- 目的: 経験を寿命ある・テスト可能・移植可能な資産にする

### ブランド固定

- 対象プロジェクトの個性を target harness に固定し、AI の方向ぶれを防ぐ
- 含める要素: 価値・方針 / 用語集 (用語+定義+禁止語) / 表記・文体規則
- .owox/ に型付き定義で持つ
- 機械検証する: 用語違反、禁止語、表記・文体違反を適応度関数で検出
- このルールの実例は control harness の AGENTS.md 自体 (用語規則・表記規則)
- 目的: 価値・方向・言葉がぶれない

## 解決する未決事項

- 判断・経緯の機械記録形式 → 来歴 (ID + 状態 + 根拠 + リンク) の型
- 作業経験の保存形式 → メモリ階層化 + スキル・ライフサイクル。長期・スキルを .owox/ で正本化
- ルール改善の検証方式 → 機械検証 (適応度関数・スキルのテスト合格ゲートを含む) を本筋。eval は段階導入
- ブランド価値の定義形式 → 型付き定義 + 機械検証 (適応度関数)

## 捨てた案

- 会話ログを正本にする
- 文脈を全部 target harness へ詰める
- 自律境界を文章の注意書きで示す
- 制約をレビュー任せで守る
- 判断記録を自由記述 doc にする
- プロジェクト状態を考えず一律の作業姿勢にする
- ルール検証を最初から eval 中心にする

## 捨てた理由

- 会話ログ正本化は、検証不能な情報が正本へ混ざる
- 文脈の詰め込みは、最小コンテキストの旗と矛盾する
- 文章の注意書きは、機械で止まらず境界が曖昧になる
- レビュー任せの制約は、漏れで崩れる
- 自由記述の判断記録は、機械で追跡・検証できない
- 一律の作業姿勢は、保守段階で回帰を招く
- eval 中心の即時導入は、評価基盤の整備が重く初期で詰まる

## 見直し条件

- スキルが全自律で増え、人間ゲートが形骸化した時
- スキルのテストゲートが重く、作成が滞った時
- ブランドの機械検証が誤検出で作業を妨げた時
- メモリ階層の境界が運用で曖昧になった時
- subagent 定義が複雑化し、コンテキスト節約に効かなくなった時
- 可逆 / 不可逆の分類で判定しにくい操作が増えた時
- 適応度関数が増えすぎて検証が重くなった時
- 来歴のリンクが運用で維持されなくなった時
- プロジェクト状態の判定が現物とずれた時
- eval を本格導入する判断がついた時
