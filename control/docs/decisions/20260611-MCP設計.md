要件・設計原則・target harness 内容を MCP の形へ落とす。

## 構成

MCP は resource (読み取り) と tool (操作) を持つ。
読み取りは resource へ寄せ、tool は操作・計算に絞る (重複排除)。

### Resources (進行的開示)

AI は必要分だけ読む。全部は読ませない (最小コンテキスト)。

- 文脈地図 (作業 → 読むもの)
- skill カタログ (frontmatter のみ)
- brand
- rules / 安全性
- quality
- requirements
- 来歴一覧
- 検証状況
- プロジェクト状態

### Tools (ドメイン別 + action)

- harness: generate / regenerate / inspect / diff
- decision: record (来歴を残す。読みは来歴 resource)
- requirement: list / create / update / link (受け入れ基準と検証 link。深い仕様管理は対象外)
- task: list (ready) / create / update / close (検証必須) / drop (理由必須) / link
- verify: run (完了3区別を返す)
- skill: create / evaluate / register / refine
- experience: export / import
- state: set (読みは状態 resource)
- gate: list (未承認の判断点) / approve (承認を来歴へ記録し解除)

### 共通返り値 (全 tool で統一)

- status: ok / needs_human / failed
- reason: なぜそうなったか
- next_actions: 次の手
- 関連来歴 ID

これで「失敗の構造化」「人間判断点を返す」を機械化する。

### 人間ゲート

MCP は AI の呼び出しを直接ブロックできない。次の流れで止める。

- 不可逆操作・正本昇格・ブランド違反などで tool が needs_human を返す
- AI はそれを人間へ提示し止まる
- 人間が承認 → gate.approve または decision.record で承認の来歴を残す
- 承認後に再実行する
- 承認なしに不可逆操作は実行しない

### 経験の入出力

- export: 汎用経験のファイルまたはディレクトリを生成する
- それを別プロジェクトへコピーして持ち込む
- import: コピーされたファイルまたはディレクトリを読む
- 構造の混入は型で防ぐ (export 型にドメイン固有項を入れない)
- 本文 (自由テキスト) の意味的な混入は型で防げない。用語集照合・秘密情報走査で検査する
- export はプロジェクト外への持ち出しなので人間ゲートを通す (検査結果を添えて needs_human)
- 複雑な転送プロトコルは持たない (純テキスト移植)

## 詳細仕様

具体フィールドの正本は core の Rust 型。ここは形の決定を残す。

### 共通返り値 (封筒)

- status: ok / needs_human / failed
- reason: 説明
- next_actions: 次の手 (配列)
- decision_ids: 関連来歴 ID (配列)
- data: ok 時の本体 (tool 別)
- gate: needs_human 時の判断点 (kind / subject / 要承認内容)

### Resource URI

- owox://context
- owox://skills、owox://skills/{name} (管理・ライフサイクル・経験メモリ閲覧用。普段 AI が見るカタログは各 CLI ネイティブの自動注入。skill 正本 .owox/skills/ から各 CLI ネイティブ位置へ生成する)
- owox://brand、owox://rules、owox://safety、owox://quality
- owox://requirements、owox://requirements/{id}
- owox://decisions、owox://decisions/{id}
- owox://tasks、owox://tasks/{id}
- owox://verification
- owox://state
- owox://next (次に決めること = open 未決 + 次にやるタスク = ready)

### Tool 引数と data

- harness.generate { targets?, dry_run? } → { generated:[{path, action: created|updated|unchanged}] }。既存破壊上書きは needs_human
- harness.regenerate { targets? } → 同上 (差分提示後)
- harness.inspect { target? } → { files:[{path, kind}] }
- harness.diff { target? } → { diffs:[{path, before, after}] }
- decision.record { title, status: open|adopted|rejected|superseded, rationale, links:{requirement?, work?, verification?}, supersedes? } → { id }
- requirement.list { status? } → { requirements:[{id, title, status, acceptance, verification_links}] }
- requirement.create { title, acceptance?, links? } → { id }
- requirement.update { id, status?, acceptance?, links? } → { id }
- requirement.link { id, target:{type:task|decision|verification, id} } → { id }
- task.list { ready?: bool, status? } → { tasks:[{id, title, status, deps}] }
- task.create { title, links?, deps? } → { id }
- task.update { id, status?, links?, deps? } → { id }
- task.close { id } → { id, closed }。検証を通らないと failed
- task.drop { id, reason } → { id, dropped }。理由を来歴に残す
- task.link { id, dep:{type:blocks|parent-child|related|discovered-from, target} } → { id }
- verify.run { scope?: work|requirement|all, target? } → { results:[{check, passed, detail}], completion:{work, requirement, verification} }。要件↔テスト trace 導入前、requirement は needs_human
- skill.create { name, description, purpose, inputs, outputs, body?, scripts?, tests? } → { name, path }
- skill.evaluate { name } → { tests:[{name, passed}], all_passed }
- skill.register { name } → { name, registered }
- skill.refine { name, error?, feedback? } → { name, changes }
- experience.export { items?, out_path } → { path }。検査結果を添えて needs_human、承認後に出力
- experience.import { in_path } → { imported:[...] }。ドメイン混入検出は failed または needs_human
- state.set { phase: initial|stable|maintenance } → { phase } (来歴に記録)
- gate.list {} → { pending:[{id, kind, subject}] }
- gate.approve { id, note? } → { id, approved } (承認来歴を残す)

### ID 形式

- 来歴・gate・タスクの ID は 日付+slug
- 例 20260611-可逆性境界
- slug は同日内で一意。重複は verify で failed にする
- 人が読め、git で並ぶ
- 連番は使わない。並行ブランチ・複数人で同番が生まれ衝突するため

### skill 登録ゲート

- テスト合格で .owox/skills へ自動登録する (可逆)
- rules / brand への昇格、不可逆操作を含む skill は needs_human

## hook との関係

- hook が呼ぶ owox サブコマンドと一部重複する (verify など)
- 両方とも core を呼ぶので、重複ロジックは持たない

## 採用理由

- tool をドメイン別 + action にすると数が少なく、ツール一覧の文脈消費が小さい
- 読みを resource へ寄せると、進行的開示と最小コンテキストが一貫する
- 共通返り値で、失敗・人間判断点・完了を一様に扱える
- needs_human 方式は、ブロックできない MCP でも人間ゲートを成立させ、承認を来歴に残せる
- 経験をファイル・ディレクトリのコピーにすると、移植が単純で純テキストの利点が出る

## 捨てた案

- フラット細粒度の多数 tool
- intent 主導の極少 tool
- 読み取りを tool で返す
- 人間ゲートを事前承認設定だけにする
- 経験 IO を専用転送プロトコルにする
- ID に連番を含める (日付+連番+slug)

## 捨てた理由

- 多数 tool はツール一覧が肥大し文脈を食う
- 極少 tool は引数が複雑化し振る舞いが読みにくい
- 読み取りを tool で返すと進行的開示が弱まる
- 事前承認だけでは不可逆のその場判断が効かず、設定ミスで事故る
- 専用転送プロトコルは依存と複雑さを増やす
- 連番は分散環境 (並行ブランチ・チーム) の衝突源になる。チーム拡張は成功条件なので構造から排除する

## 見直し条件

- action 数が増え、1 tool の振る舞いが読みにくくなった時
- resource 未対応の AI CLI を本格対応する時
- needs_human の往復が多すぎて作業が滞った時
- 経験のコピー方式で取り違えや混入が起きた時
- 共通返り値の型が用途に合わなくなった時
