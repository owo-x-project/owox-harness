# Phase7 コンテキスト配信の再設計 (AGENTS.md 廃止・床は hook 注入・意図ルーティング)

## 状態

採用済み・機械確証済み (217 passed・clippy/fmt クリーン)・実機 Codex 検証済み (6 観点)。
当初 PostCompact で圧縮後の床を戻す設計だったが、実機で PostCompact は context 注入不可と判明し、SessionStart の compact/resume matcher へ切替えた (下記)。

## 背景

Phase7 対話検証で 3 つの懸念が残った (`docs/handoff/20260616-コンテキスト配信の再設計.md`)。

1. AI が「rules」を注入文と結びつけられない (注入に `## Rules` ラベルが無く方針がバラけていた)
2. 常時注入が多いと従われない。要る物に絞り足りない物を足す
3. AGENTS.md は hook 注入で代替できるのでは

加えてユーザ要望: コンテキストとツールを工夫し、ユーザーが明示的にツール名・用語を言わなくても、意味が十分なら曖昧なリクエストで動くようにする。

参照: `docs/decisions/20260612-段階的開示.md` (3層配信)、`20260616-Phase7-canon変更の自動適用と承認の機械強制.md` (brand/rules を SessionStart 注入へ移した直前の判断)。

## 判断基準

- 常時読む量を最小に保つ (測定可能な最小コンテキスト)
- 重要情報を AI が見落とさない (関連時に確実に届く)
- ユーザーが owox の語彙 (tool 名・用語名・rules ラベル) を覚えなくても、意図から正しい capability へ AI が飛べる
- 硬い安全は注入の有無に依らず守られる (hook 機械強制)

## 採用案

床を「消えにくさ」で分け、本文は関連時にオンデマンドで届ける。

### AGENTS.md 廃止・床は hook 注入のみ

- ルート指示ファイル AGENTS.md を生成しない。床コンテキスト (向き付け・意図ルーティング・Vision・全体スタイル・応答言語・状態・用語名・practices・Read next) は core `floor_context` が組み、SessionStart が `additionalContext` で注入する
- 圧縮・再開後の床復元は SessionStart の matcher を `startup|resume|compact` へ広げて担う。SessionStart は source=compact / resume でも再発火する (Codex 公式)。当初は PostCompact で戻す設計にしたが、実機で PostCompact は additionalContext 非対応・plain text も無視で context を注入できないと判明 (`https://developers.openai.com/codex/hooks`)。additionalContext 対応は SessionStart / PreToolUse / PostToolUse / UserPromptSubmit / SubagentStart のみ
- 編集可能な複製を置かないことで、AI が生成物を直接編集して canon を変える経路を断つ (直前判断の延長)

### 意図ルーティング表 (曖昧リクエスト対応の中核)

- 床の安定部分に「要望 → 使う tool」の対応表を常時置く。ルール/価値/原則/定義の変更 → canon.propose、教訓 → canon.add practices、次の一手 → next、読む先 → context、用語 → glossary.lookup、rules → rules.lookup、不要コード → review.lenses + deletion policy、完了/commit 前 → verify.run
- tool 説明も意図志向に保ち (Codex が説明から自動選択)、get_info にも「tool 名を言われなくても canon.propose へ寄せる」旨を明記
- 注入文に括弧補足・ツール名は使わない (生成物のツール非依存・1文で分かる。memory 準拠)

### rules / brand のオンデマンド配信

- rules 本文は `## Rules` 見出しへ集約し各方針を `###` の小節にする (懸念1の是正)。`render_rules_block` が lookup・語トリガ push で共用
- 3 経路すべて採用:
  - (a) UserPromptSubmit でプロンプトに rules/policy/delete/dependency/safety 等の語が出たら `## Rules` を、brand 関連語で `## Brand` を push (`policy_injection`・glossary_injection と同型・語幹部分一致)
  - (b) `rules.lookup` ツール (AI が能動的に引く backstop・glossary.lookup と対称)
  - (c) PreToolUse の編集直前に change/deletion/safety policy を `force_rules` で一度届ける (編集という行為が合図)
- brand は values/principles/non-goals/success-criteria を語トリガへ。Vision と全体スタイルは床に常時残す (全体スタイルは作業全般に効く・段階的開示の方針)
- 重複排除は glossary と同じ session キャッシュへ `policy:rules` / `policy:brand` の区分キーで載せる (用語名はコロンを含まず衝突しない)

## 捨てた案

- 最小 AGENTS.md を安定床として残す (向き付け+ルーティング+Vision)。圧縮で消えず PostCompact 未確認リスクを避けられるが、ユーザ判断で「単一経路・重複なし」を優先し不採用 (`docs/handoff/20260616-...` で確認)
- brand.lookup の追加。Vision 常時 + 語トリガで足り tool 増設は過剰
- 全体スタイルも on-demand へ。スタイルは作業全般に効くため push 漏れ時に文体がぶれる

## 検証後の追補 (2026-06-16・実機検証で炙り出た2点)

- 非英語プロンプトの rules push: 語トリガは英語語幹のため日本語プロンプトでは prompt 時点の rules push が発火しない。ハーデコードは i18n しない方針に反するので採らず、意図ルーティング表を言語非依存に強化 — canon.propose の前に rules.lookup / glossary.lookup で現行 wording を読む旨と、rules/policy/deletion/dependency/safety に関わる依頼ではどの言語でも rules.lookup を先に呼ぶ旨を明記。安全核は元々言語非依存 (不可逆 deny はコマンド照合・編集前 force_rules push は語に依らず change/deletion/safety を届ける)
- Stop の毎ターン催促: open gate が 1 件でもあると Stop が毎ターン継続催促していた (人間待ちゲートの蒸し返し)。dirty と同じ署名方式を open gate にも入れ、未承認 gate の顔ぶれ (open 来歴 ID の整列連結) が変わった時だけ一度表面化するよう `stop_decision` へ `gates_changed` を追加。保守 phase の commit ゲートが別途 open gate を block するので機械強制は不変

## 見直し条件

- SessionStart の compact/resume 再発火が実機で床を戻さない時 (最小 AGENTS.md の再導入を再検討)
- 語トリガの過不足が実運用で目立つ時 (語リストの調整・lookup 依存度の見直し)。日本語等の非英語プロンプトでは語トリガが発火しないが、意図ルーティング表 (英語・常時) が言語に依らず正しい tool へ導くため intent は成立する。content は rules.lookup で補える (実機で確認)
- 意図ルーティング表が増えて床が重くなった時 (オンデマンド化)
