# Phase5 実機検証の是正

## 状態

採用。Phase5 を実機 Codex の対話で検証して見つけた粗を是正する設計。
`docs/validation/20260613-Phase5-スキルと入口.md` の続き (対話検証) で発覚。
`docs/decisions/20260613-Phase5-スキルと入口.md` を一部是正し、`docs/decisions/20260611-MCP設計.md`・`docs/decisions/20260613-Phase4-MCP実装.md` の「読みは resource へ寄せる」を転換する。

## 背景 (実機で見つけた粗)

対話 Codex で入口 skill を起動して判明:

- `$next` が機能しない。入口 skill 本文「Read owox://next」に対し、モデルは resource を取りに来ず (read_mcp_resource を使わず) canon ファイル直読みへ退避し deny され、CLI を手探りして失敗。未決判断も ready タスクも返せない
- `$verify` が verify.run を呼ばず生の検査スクリプトを直接実行。owox の検証層 (検査再実行・gate 連動) を通らない
- 生成 skill の `agents/openai.yaml` が Codex の正式スキーマと不一致。`allow_implicit_invocation` をトップレベルに出すが、正式は `policy.allow_implicit_invocation`。さらに `interface` ブロック (display_name 等) が無い
- 生成物が英語固定のため、第一言語が日本語のプロジェクトでもモデルが英語で応答する

確実に動いたのは tool 呼び出し (glossary.lookup・skill.list は明示で完走) と hook 群 (session 注入・直読み deny・Stop 継続)。resource はモデルが能動的に取りに来ず不安定。

## 採用設計

### 読みを tool へ一本化 (resource 撤去)

`owox://context`・`owox://next` resource を撤去し、同じデータを返す read 系 tool を新設する。

- 根拠: Codex で resource はモデルが read_mcp_resource で能動取得しないと届かない。入口 skill のような「確実にデータを取る」用途で不安定。tool は確実に動く面 (検証で実証)。同データを resource と tool で二重に持つ重複も解消する
- Phase4 は「読みは resource へ寄せる」と決め resource 読取の成立を確認したが、同時に「モデルはファイル読取権がある限り直読みする」とも判明していた。実機の入口 skill でこれが顕在化したため転換する
- 命名・封筒は既存 tool に揃える (検証で動いた面)

### 入口 skill 本文を強束ね

入口 skill の本文を、1 つの owox tool を命令形で呼び代替を禁じる形にする。

- 散文「Read owox://next」「Run verify.run」はモデルが別経路へ逸れる。命令形 + 代替禁止で 1 つの owox 面に束ねる
- 例 (verify): verify.run を呼び完了3区別を報告する。プロジェクトの検査スクリプトを自分で走らせない。verify.run がそれを実行・記録する
- 読み系入口 (kickoff/next/status/handoff) は新 read tool を呼ぶ

### skill ライフサイクルの入口を足す

skill の register / promote / list / remember は tool だけで人間が叩く入口が無く、「skill を登録して」が codex 内蔵 skill-creator の暗黙起動に化け owox の skill.register に届かなかった。標準入口に `skill` を足し (task 入口と同型)、skill tool へ命令形で束ねる。これで `$skill` から確実に owox の skill ライフサイクルへ入れる。第1段階の標準入口は 9 個になる (kickoff/next/status/decide/verify/review/task/skill/handoff)。

### openai.yaml を正式スキーマへ

- `allow_implicit_invocation` を `policy.allow_implicit_invocation` へ入れ子化する (トップレベルは Codex に無視され、明示専用のはずの入口が自動起動しうる)
- `interface` を出す: display_name・short_description・default_prompt。default_prompt は `$名前` を含める (UI 表示と $ 起動のため)

### 出力言語の追従

- 正本に応答言語の設定を持つ (machine 設定の config.toml)
- SessionStart 注入に「この言語で人間へ応答せよ」を 1 文加える
- 指示文 (生成物・英語) と応答言語を分離する。生成物を多言語化せず、応答だけ追従させる (正本 1 つ・安定)

## 不採用

- resource と tool の両建て — 同データ重複・面が二つに。codex 第1では tool が確実。他 CLI で resource 自動添付が活きる時に将来再考
- resource 維持 + 入口本文の指示強化のみ — read_mcp_resource は Codex 固有でツール非依存制約に反し名指しできず、不安定が残る
- 生成物自体を母語生成 — 実装重く翻訳維持コスト。応答言語注入で足りる

## 変更境界

触る:
- crates/mcp/src/serve.rs: owox:// resource 撤去・read 系 tool 新設・get_info の resource 案内除去
- crates/core/src/commands.rs: 標準入口本文の強束ね・tool 参照化
- crates/core/src/hook.rs: SessionStart の読む先を tool 参照へ・応答言語注入・Stop 文言
- crates/core/src/targets/codex.rs: openai.yaml スキーマ修正・AGENTS.md の読み案内を tool へ
- crates/core/src/model.rs ・ load.rs: 応答言語設定の読込
- 各テスト

触らない:
- canon 直読み deny・glossary/skill ライフサイクル・task/decision/gate tool・不可逆 deny
- 記録層・封筒の振る舞い

守る振る舞い:
- 直読み deny・用語注入・skill ライフサイクル・commit/stop ゲート・config.toml の MergeToml

## 危険

- resource 撤去は Phase4 決定の転換。関連判断文書を是正・相互参照する
- 入口本文を強束ねしても `$verify` 迂回が完全に止まる保証は無い。実機再検証で確かめる
- 応答言語注入にモデルが従うかは実機確認要
- tool 名と入口 skill 名 ($next 等) の混同を避ける命名にする

## 検証方針

- 単体: read 系 tool が封筒で未決・ready を返す、openai.yaml が policy/interface を出す、応答言語設定の読込、入口本文の生成
- 結合: setup 縦貫で resource が消え read tool が出る、openai.yaml スキーマ妥当
- 手動 (実機 Codex): `$next` が read tool を引いて未決・ready を返す、`$verify` が verify.run を通す、入口が自動発火しない、母語で応答する
- 未確認として残す: 強束ね後の迂回有無、言語追従の実機効き

## 後回し (別途検討)

- Stop block のノイズ: 作業ツリーに未コミット変更があると、AI がそのターンで何も触らなくても毎回 Stop が verify/記録を促す。Stop hook はターン差分でなく git のツリー状態を見るため (安全側だが複数ターン作業でうるさい)。ターン開始時のツリー状態を基準にする等は別途設計。実運用ではコミットして終えるので鳴り止むため M1 では保留

## 見直し条件

- Codex の skill / openai.yaml スキーマが変わった時
- 他 CLI 対応で resource 自動添付が有用になった時
- 応答言語注入が効かない・摩擦になった時
- skill 入口とプロジェクト追加 skill / 内蔵 skill が衝突した時
