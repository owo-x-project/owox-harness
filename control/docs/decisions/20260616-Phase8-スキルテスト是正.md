# Phase8 スキルテスト是正

## 状態

採用 (実装済・検証済 `docs/validation/20260617-Phase8-対話検証と粗の是正.md`)。Phase8 (`docs/requirements/20260616-Phase8-経験を厚くする.md`) の設計。
Phase5 のスキルライフサイクル (`docs/decisions/20260613-Phase5-スキルと入口.md`・`crates/core/src/skill.rs`) を是正する。

## 背景

スキルは SKILL.md = 自然言語の手順。テストは tests/ の各ファイルが終了コード 0 で合格。
弱点が2つある:

- 散文の振る舞いを単体テストできない。「テストで何を確かめるか」が不明で書けない
- `exit 0` するだけの空テストでも合格する。ゲートが「合格テストが在る」ことしか保証しない

owox が保証するのは「pass するテストが存在する」で「テストが中身を覆っている」ではない。
ここを正直な形へ是正する (マネージャ問いで A+C 採用)。

## 採用設計

### C: 契約 lint を全スキル共通の最低ゲートにする

owox が自動で走らせる決定的検査。人間が書かない。registrable_shape を拡張する。

- frontmatter の name / description 必須 (既存)
- SKILL.md 本文が参照する `scripts/<name>` が実在し実行可能か。本文から `scripts/...` のパス形を機械抽出し、対応する ScriptFile が無ければ違反。散文でもパス形は決定的に拾える
- tests/ の各ファイルが実行ビットを持つか。持たないと sh 起動が誤る (運用前提 `docs/decisions/20260613-Phase5-スキルと入口.md`)
- owox.toml のスキーマ妥当 (既存 parse で担保)

lint 違反は問題として draft に留める (skill_status の problem へ。skill.register が理由を返す)。
これで「テストを書けない」が消える。最低限の健全性は owox 提供の lint が担い、人間は script があるときだけ振る舞いテストを書く。

### A: テストの定義を正直にする

- 「テストはスキルが依存する script / tool の振る舞いを確かめるもの」と SKILL.md・get_info・本設計で明示する。テスト可能面は scripts/ にある
- 決定的にテストできる対象が無い純散文スキル (scripts/ 無し) は自動起動にできない方針を維持・明確化する。現状 implicit=true → tests/ 必須 で半分実現済み。純散文で自動起動したいなら、テスト可能な script に手順を落とすのが筋
- 振る舞いの自然言語評価 (LLM 判定) は Phase10 の opt-in・助言にする。ゲートにしない (決定論の核を汚す・非決定的)。`docs/decisions/20260611-eval方針.md` の延長

### 入口系免除の据わり

入口系 (commands.toml 生成・implicit=false) は明示起動のみで自動発火しないため tests/ 免除を維持する。
プロジェクト追加の implicit=false スキルも免除だが、明示起動なので自動発火リスクは無い。
契約 lint は全スキルにかかるので、免除スキルも script 参照・実行ビットの健全性は担保される。

## 危険

- scripts 参照抽出が過検出・未検出 → パス形 (`scripts/` 前置) に限定し保守的に。曖昧な散文中の tool 名は対象にしない (false positive を避ける)
- 既存スキルが新 lint で draft へ落ちる → 正本側の例を先に直し、回帰テストで確認
- 「純散文は自動起動不可」がスキル作成の摩擦になる → 明示起動は使えるので実害小。摩擦が出たら見直し

## 検証方針

- 単体: scripts 参照欠落で draft・problem に理由、tests/ 実行ビット欠落で draft、正常スキルは登録、implicit=true + 合格テストはこれまで通り登録、implicit=false は lint のみで登録
- 結合: skill.register が lint 違反を封筒で返す
- 手動 (実機 Codex): script 付きスキルを作り lint → テスト → 登録 → 昇格まで回す

## 捨てた案

- 振る舞いの LLM 評価をゲートにする → 非決定的。決定論の核を汚す。Phase10 opt-in・助言へ
- 散文中の tool 名参照まで lint する → 自然言語で過検出。パス形に絞る
- 空テスト検出のヒューリスティック (内容を見て trivial 判定) → 当てにならない。契約 lint + 「script があるときだけ振る舞いテスト」で代替

## 見直し条件

- 契約 lint の過検出でスキル作成が滞った時
- 純散文スキルの自動起動需要が強く出た時 (テスト可能化の別手段を検討)
- Phase10 の eval が入り、振る舞い評価を助言として足す時
