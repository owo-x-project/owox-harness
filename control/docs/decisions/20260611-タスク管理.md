# タスク管理

## 状態

採用済み。検証可能タスク層を Phase 4 で実装 (`docs/decisions/20260613-Phase4-tool記録層.md`)。
done 検証必須・来歴連動 (drop 理由) ・ready (前提 done) を実装・実機確認。
腐敗検知の適応度関数 (stale / 孤立 / 重複 / ゾンビ / done 未検証 の機械検出) は重いため Phase 7。
GitHub Issues sync・Beads 移行 import は未着手 (任意機能)。

## 背景

タスク管理が無いと、AI はやることを見失う。
一方、中途半端なタスク管理は腐る (放置・重複・嘘 done が溜まる。実体験)。
最小実装やアダプタ依存では腐敗を防げない。
そこで、旗 (検証可能な正本) をタスクに適用し、Beads を超える。

- `docs/decisions/20260611-製品戦略.md`
- `docs/decisions/20260611-設計原則.md`

## 位置づけ

一級の検証可能タスク層。owox が正本を所有する。
アダプタ従属ではなく source of truth。

## なぜ Beads を超えられるか

Beads が腐る構造的弱点は、検証エンジンも判断層も持たないこと。

- done がエージェントの自己申告で、機械で確かめない
- タスクと判断 (なぜ) の結びつきが薄い
- 古い・孤立・ゾンビ タスクを検知して掃除する仕組みが無い

owox は完了3区別の検証と来歴 (判断) と適応度関数を持つ。これが武器。

## タスクモデル

型付き正本。.owox/tasks/ に置き、git 管理、機械可読。

- id: 日付+slug (来歴と同形式。連番なし、並行作業の衝突防止。`docs/decisions/20260611-MCP設計.md` の ID 形式)
- title
- status: todo / doing / done / blocked / dropped
- links: requirement / 来歴 / verification
- deps: blocks / parent-child / related / discovered-from (型付き)

## 腐敗を防ぐ中核 (すべて必須)

後付けオプションにすると結局腐るので、中核必須にする。

- done に検証必須: 完了3区別 (作業 / 要件 / 検証) を通らないと閉じれない。自己申告 done を排除
- 来歴連動: 存在・完了・破棄の理由を来歴に残す。捨てるにも理由。silent rot 禁止
- 適応度関数で腐敗検知: 放置 (stale) / 孤立 (リンク無) / 重複 / ゾンビ (永久 block) / done 未検証 を機械検出し、掃除を促す
- 状態適応の厳格度: 保守=放置ゼロで厳格、初期=緩い (回帰防止と一貫)

## 依存と ready

- ready = 依存解決済 + 前提タスクが検証済
- 自己申告でなく検証済を前提条件にする点が Beads の ready を超える

## 人間ゲート

- 範囲外・不可逆を含むタスクは needs_human

## MCP と制御の連動

- task tool: list (ready) / create / update / close (検証必須) / drop (理由必須) / link
- resource: owox://tasks、owox://tasks/{id}
- session 開始 hook → ready を文脈に出す
- 完了前 hook → 該当タスクの close に verify、腐敗検知を表面化
- 作業中の気づき → discovered-from でタスク化 (誘導)

## 既存ツールとの関係

- owox が正本 (source of truth)
- GitHub Issues へ可視化 sync (任意、チーム可視化)
- Beads から移行 import (任意)
- アダプタ従属にはしない

## 実装順

- 設計は今決める
- 実装は中核生成 (第1段階) とは別の専用増分にする
- 腐敗防止の中核を最初から入れる (後付けにしない)

## 採用理由

- 検証ゲートが嘘 done を断ち、腐敗の主因を消す
- 来歴連動で、放置・破棄が理由つきになり追跡できる
- 適応度関数が腐敗を機械検出し、人手の掃除に頼らない
- 状態適応で、初期の試行錯誤と保守の厳格を両立する
- owox 正本にすると、検証・来歴・skill と一体で扱える

## 捨てた案

- 最小の組み込みタスク層
- Beads などへのアダプタ従属
- タスク管理をスコープ外にする
- 腐敗防止を後付けオプションにする
- 来歴・work をタスクへ転用する

## 捨てた理由

- 最小実装は腐敗を防げず、結局腐る
- アダプタ従属は腐敗を外部依存に委ね、超えられない
- スコープ外はプロジェクト進行の背骨を失う
- 後付けオプションは使われず腐る
- 来歴・work の転用は軸が混ざり、やること一覧が埋まらない

## 見直し条件

- 腐敗検知が誤検出で作業を妨げた時
- タスクの型が用途に合わなくなった時
- 検証必須が重く、タスク運用が滞った時
- GitHub Issues sync の往復で不整合が出た時
- 依存の型が増えすぎた時
