# Phase 3 hook 実装

## 状態

採用済み (2026-06-12)。Phase 3 の機械強制 hook を作る際の方針。

## 背景

Phase 1 で SessionStart hook の発火・additionalContext 注入を実機確証。Phase 2 で正本の型を確定。
Phase 3 は「危ない時・完了時に必ず止まる」を機械で効かせる。完了の目安は「止めるべき所だけ確実に止まる」。

着手前に未確定だった点を、人間判断と一次情報で確定する。

参照:

- `docs/decisions/20260611-制御方針.md`
- `docs/decisions/20260611-第1対象CLI.md`
- `docs/validation/20260612-Phase1-Codex実機.md`

## hook の入出力契約 (一次情報)

Codex hooks の一次情報 (developers.openai.com/codex/hooks.md) で確認。手元キャッシュのマニュアル本体には出力契約が載っていなかったため、別途取得して確定した。

入力 (stdin、スネークケース):

- 共通: session_id / transcript_path / cwd / hook_event_name / model / permission_mode
- PreToolUse: tool_name / tool_use_id / tool_input / turn_id。Bash のコマンドは tool_input.command
- SessionStart: source (startup / resume / clear / compact)
- Stop: turn_id / stop_hook_active / last_assistant_message

出力 (stdout、キャメルケース):

- ブロック: hookSpecificOutput に hookEventName=PreToolUse / permissionDecision=deny / permissionDecisionReason を入れる
- ブロック (代替): 終了コード 2 + stderr に理由
- 許可+文脈: hookSpecificOutput に additionalContext
- 終了コード 0 で出力無し: 素通り

SessionStart で既に使う hookSpecificOutput.hookEventName 構造を PreToolUse でもそのまま使える。

## 判断 1: 進め方は安全核を縦貫

5 種の hook を横並びに揃えるのでなく、安全核 (不可逆操作の検出 → PreToolUse で deny → 実機で本当に止まるか確認) を先に一気通貫する。差分要約・作業メモ (PostToolUse 系) は後回し。

理由:

- 実機 deny が唯一の未確認点。Phase 1 と同じく「推測部分を実機で確証」を最優先にする
- 完了の目安「止めるべき所だけ確実に止まる」の核がまず立つ

## 判断 2: 不可逆操作は owox 同梱の既定 + target 追記

owox core が危険コマンドの既定検出パターン (git push の force / git reset --hard / rm の再帰強制削除 / git clean の強制 / git branch の強制削除 等) を持ち、箱から出してすぐ効く。
target は rules.md の Irreversible エントリに検出フィールド (コマンド正規表現) を非破壊追加して、プロジェクト固有の不可逆操作を足せる。

検出には正規表現が要る (既定も target 追記もコマンド文字列の照合)。標準ライブラリに無く、target 機能の前提でもあるため regex を core の依存へ加える (依存方針の例外として正当)。

理由:

- 箱から出してすぐ安全。利用者が危険定義をゼロから書く負担が無い
- target が固有の不可逆操作を足せる。owox 既定のみ固定だとプロジェクト固有を守れない
- 制御方針の「機械検証できる規律は hook で強制」と一致する

段階 (いずれも実装済み):

- スライス 1: core 同梱の既定検出器。実機 deny を確証
- スライス 2: Irreversible へ任意の `detect:` (コマンド正規表現) を非破壊追加。rules.md で target 固有の不可逆操作を機械検出。正規表現は読込時に妥当性検証。PreToolUse は既定 ∪ target で照合

## 判断 3: hook は薄いシェルを廃止し owox を直接呼ぶ

hooks.json の command を `.codex/hooks/session-start` 経由でなく `owox hook <event>` の直接呼び出しにする。シェルファイルと実行ビットが要らなくなる。

理由:

- 5 hook へ増える前に整理すれば、5 個のシェルファイルを作らずに済む (清潔・将来性)
- owox は PATH 上のシステムバイナリ。マニュアルが repo ローカルスクリプトに勧める git ルート基準パスは、PATH 解決の owox には当てはまらない (相対パス問題自体が消える)

## commit / stop の完了検証は Phase 3 で誘導まで

制御方針の「完了・commit 前検証」の中身 (verify 済みか・未記録判断があるか) は、verify の結果や `.owox/decisions/` の判断記録という Phase 4 (MCP) の状態に依存する。状態が無い Phase 3 で機械判定はできない。

そこで Phase 3 は配管と誘導までを作り、機械ブロックは状態が揃う Phase 4 へ送る。

- PreToolUse が git commit を認識し、非ブロックの additionalContext で完了確認を促す (allow + 一行の気づき)
- Stop は完了前に verify・判断記録を促す。stop_hook_active が false の時だけ 1 度継続させ、既継続なら受理してループを避ける

これは制御方針の「機械検証できる規律は hook で強制、できないものは誘導」の段階適用。Phase 4 で verify 状態・判断記録が揃えば、同じ配管を機械ゲート (未検証なら block) へ格上げする。その場しのぎでなく、検証済みの配管の上に Phase 4 が方針を載せる。

### Phase 4 で機械ゲートへ格上げ済み (2026-06-13)

予告どおり格上げした (`docs/decisions/20260613-Phase4-tool記録層.md` の「commit/stop の機械ゲート化」)。

- commit: PreToolUse で config.toml の検査を再実行し、verify 失敗なら deny (AI が verify.run をサボっても効く)。未承認 open gate は警告。再実行は古い結果で通さないための選択 (状態を持たず漏れない)
- stop: 誘導のまま (毎ターン発火で再実行は重い)。未承認 open gate の件数を表面化
- 未承認 gate を保守状態で block へ格上げする state 適応は state.toml と後続

### 差分要約は push せず Phase 4 の pull へ

制御方針は「commit 前: 差分要約」を挙げるが、commit リマインダへ diff を常時 push するのは旗「最小コンテキスト」と段階的開示 (中身はオンデマンド) に反する。一度試作し、以下の根拠で push をやめた:

- コンテキスト汚染 (context rot): 冗長トークンは費用増だけでなく信号を薄め推論精度を下げる。窓を広げても解決しない (Chroma 等の研究)
- hook の additionalContext は毎ターン読み直され再課金される (一度入れると以後ずっと運ぶ)
- AI は通常たった今その変更をした → diff の多くは既知=汚染寄り
- 公式ガイダンスも「高頻度 PreToolUse に大きな diff を注入するな」「生 diff より必要時に引く」

よって差分の閲覧は Phase 4 の owox:// (中身を必要時に引く pull) で提供する。commit 前は一行の気づきだけ常時、diff はオンデマンド、という分担にする。

## 作業メモ (work-note) は Phase 4 へ

作業メモはメモの保存先と取り出しの仕組みが要る。それは Phase 4 (MCP・記録層) が設計する `.owox/` の記録構造に属する。Phase 3 で保存先を仮置きすると Phase 4 の設計と衝突する捨て実装になるため、記録層と一体で Phase 4 で作る。

## human_gate と不可逆ブロックの関係

- 不可逆操作: コマンドパターンで機械検出でき、PreToolUse で決定的に deny する (機械強制)
- human_gate (人間ゲート): 「アーキ判断」等ファジーで機械検出しにくい状況。コマンド正規表現では捕まえられない。指示 (AGENTS.md の Hand back to a human) と、session 開始・完了時の表面化で誘導する

両者は重なるが強制手段が違う。機械検出できるものは不可逆ブロックへ、できないものは human_gate の誘導へ寄せる。

## 捨てた案

- 5 hook を横展開してから実機検証
- 不可逆定義を target 記述のみ / owox 既定のみ固定
- 薄いシェルを維持

## 捨てた理由

- 横展開は実機 deny の未確認を最後まで残す
- target 記述のみは初期が無防備、owox 既定のみ固定はプロジェクト固有を守れない
- 薄いシェル維持は 5 hook でシェルファイルが増え不潔

## 見直し条件

- 既定検出器が誤発火で正常作業を妨げた時 (パターンを保守的化)
- target の正規表現照合だけでは不可逆操作を捕まえきれないと判明した時
- `owox hook <event>` の PATH 解決が実機 Codex で安定しないと判明した時
- Stop / commit gate の完了検証を加える時に契約の追補が要る時
