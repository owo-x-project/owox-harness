# 承認と自動改善ループ

採用判断。2026-06-19。元議論 `docs/handoff/20260618-handoff-承認と自動改善.md`。
人間の元発言4点 (gate.approve 修正・自動承認+後追い確認・製法テンプレ・自動改善ループ) を詰めて確定。

## 背景

- gate.approve が AI に「人間が先に承認 action する」と誤解させ、AI が人間へ実行依頼 → MCP tool なので人間は呼べず詰む
- 寝てる間など、たまに自動で進めたい。ただし後から確認・訂正できること
- よく使うパターンを積極使用する成長型の仕組みの価値
- 人間が都度「ルール化して」と言わずに済む自動改善

## 採用した案

### 点1 gate.approve の文言是正 (バグ・挙動不変)

- 詰みの主犯は1行でなく約7か所の注入文。特に「人間がこのゲートを gate.approve で解決/承認する」という言い回し
- 全箇所を「人間が承認と判断したら AI が gate.approve を呼ぶ。CLI の確認プロンプトが人間の承認ゲート。人間にツール実行や事前 action を頼まない」へ統一
- 対象: crates/mcp/src/serve.rs の gate.approve 説明、crates/core/src/canon.rs の propose 返り (構造化・自由文の両 requires と本文)、crates/core/src/commands.rs の decide 入口文、crates/core/src/hook.rs の commit 前 open gate 文 (保守 block と助言)
- skill.promote は destructive_hint 無で「人間がチャットで昇格と言う→AI が呼ぶ」が正しく詰まないため変更しない (CLI 確認を足すかは別途・後述見直し)
- コード挙動は不変。文言のみ

### 点2 自動承認 (既存自律度勾配で線引き)

> supersede。点2 は `docs/decisions/20260620-自律度根本方針と自動承認パス再設計.md` が上書きした。
> 旧案「auto 既定オフ・人間が明示オン・セッション限り」は、根本方針 (最初を厚く握りあとは任せる) に合わせ
> 「architecture 軸から既定導出・profile 由来は永続/session 由来は限りの2系統・対象は層×不可逆の積・後追いキューは来歴一本」へ変わった。
> 以下は旧記述として残す。点3・点4・集約点は有効。

- 機構の壁: destructive_hint は tool 単位の静的注釈。gate.approve は呼ぶと必ず CLI 人間確認が出るので、自動承認は gate.approve を通せない (寝てる間も人間待ちで止まる)
- 境界は新たな可逆性軸を足さず、既存の自律度勾配 (guarded/supervised/free。crates/core/src/gate.rs) に乗せる
  - guarded 層ゲート → 常に人間 (gate.approve のまま・auto 不可)
  - supervised/free 層ゲート → owox 所有の auto 承認パス (destructive_hint 無) で自動承認、`approval:auto` 印を付け後追いキューへ
- auto はオフ既定。人間が CLI 確認付きでオンにした時だけ働く。寿命はセッション限り (次の session_start で自動解除・人間が毎回明示再オン)
- オン操作自体が人間ゲート (CLI 確認) なので、AI が無断で自己承認窓を開けない

### 点3 製法テンプレ (既存拡張で能動提示)

- 新語「製法」「パターン」は造語禁止に触れるため導入しない。practices=経験則 / skills=手順 のまま
- 隙間は「能動提示」。今 practices は受動注入・skills は呼び出し待ちで、着手前に該当を先出しする駆動が弱い
- 着手前の能動提示を既存の kickoff 束導線に相乗りさせ、現タスクへ該当する practices/skills を owox が先出しする

### 点4 自動改善ループ (訂正検知→草案自動起草)

- 人間が AI を訂正した事実を owox が検知 → practice 草案を自動起草
- 検知は決定論コア (人間が AI 直後出力ファイルを手編集) + AI 自己申告 (ファイルを触らない会話だけの訂正)
- 草案は固定でなく後追いキュー/open gate へ積む。人間承認で初めて practices へ固定
- 検知=自動・固定=人間で理念 (人間が重要判断を握る) と両立

### 集約点 (点2 と点4 を一本化)

- 点2 の auto 承認分と点4 の auto 草案は同じ後追い確認面に積む
- session_start / next で「未確認 N 件」を表面化し続け、人間が確認 or 差し戻し (canon 履歴の逆操作)
- 確認面・ツールは1系統で済み断片化を避ける

## 採用理由

- 点1: 詰みの実害が出る純バグ。挙動を変えず文言で直せる
- 点2 境界を既存勾配に: guarded/supervised/free は既に「変更権限の所在」を表す。可逆性という別軸を足すと既存層と二重管理・造語問題。最小コンテキストと清潔を優先
- 点2 セッション限り: 切り忘れで意図せず auto が続くのを構造で防ぐ。最も安全
- 点3 既存拡張: 新独立機構は practices/skills/routine と役割重複・断片化し、造語も要る。能動提示だけ足すのが清潔
- 点4 訂正検知→草案: 都度「ルール化して」が消える。検知は決定論、固定は人間承認で理念を守る
- 集約: 後追い面を1系統にし、人間の確認動線を散らさない

## 捨てた案

- 全ゲート auto + 後追い確認 (削除/不可逆も自動): destructive_hint 強制を全面的に外し、不可逆操作が無確認で通るリスク
- 可逆性で新軸を導入: 表現は精密だが既存自律度勾配と二重管理・新次元の保守コスト
- auto 承認なし (点1 のみ): 寝てる間に進めたい要求を満たさない
- 製法の独立機構を新設: 既存3機構と重複・造語必要
- 完全自動ループ (検知→起草→可逆なら自動固定): 人間の判断機会が減り理念と緊張
- auto ウィンドウをオフまで永続: 切り忘れリスク
- 訂正検知をファイル差分のみ: 会話だけの訂正を取りこぼす

## 見直し条件

- 点1 是正後も実機で AI が人間へ実行依頼する挙動が残る → 注入経路を再点検
- supervised/free でも不可逆な操作が auto を通って事故 → 層判定に不可逆性の補助条件を足すか再検討
- セッション限りが運用で煩雑すぎる → 操作回数上限など別寿命を検討
- 自己申告検知が AI 依存で機能しない / 誤起草が多い → 検知信号の重み付けを見直し
- skill.promote の自動起動解禁にも CLI 確認 (destructive_hint) を要すべきか → 別途判断
