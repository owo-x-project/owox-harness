# 自律度の根本方針と自動承認パス再設計

採用判断。2026-06-20。
元議論 `docs/handoff/20260620-自律度方針とコンテキスト効率の深掘り.md`。
`docs/decisions/20260619-承認と自動改善ループ.md` の点2 (自動承認) を supersede する。

## 背景

人間の方針表明:

- git 等バージョン管理が前提 → push しない限りほぼ全操作は可逆・戻せる
- よって「何をどう作るかの最初の方針」だけ人間が負担少なく強く握れれば、動き方は AI に任せてよい
- 実作業は devcontainer / worktree でサンドボックス化し破壊を最小化できる

この立場が成立する鍵は「握るべき核判断が最初に出尽くす」こと。
ここを3つの穴で検証し、根本方針を決め、その具体路として点2 を再設計した。

## 採用した案

### 根本方針 = 最初を厚く握りあとは任せる + 3つの安全弁

人間の立場に基本同意する。新しい同等権限の層は作らない (理念「人間が核を握り AI 自律性最大化」と衝突)。
既存の自律度勾配 (guarded/supervised/free。`crates/core/src/gate.rs`) の閾値調整で対応する。
任せた先の安全弁を3つ置く。

穴A 外部副作用 → 独立最優先ガード:

- push/deploy/メール/課金/外部送信/秘密漏洩は git で戻らず、可逆性が成立しない唯一の領域
- 不可逆操作は層・auto 設定と直交させ常にゲートする (`rules.irreversible`・`crates/mcp/src/hook.rs` pre_tool_use)
- 自動承認はこれを絶対に迂回しない配線を保証する

穴B 核判断の湧出 → 二段ゲート / 物質化トリガ限定:

- 一段目 = 最初の方針 (canon/requirements の人間承認)
- 二段目 = 作業中に核が guarded 資産への操作として物質化したら機械検知で止める (`layer_pre_action_gate`)。AI は `decision.record` の authorizes で触りたいパスを宣言し人間 `gate.approve` で 1 回解凍 (`docs/decisions/20260619-Phase9-guarded承認で解凍.md`)
    - 2 軸での位置づけ (`docs/decisions/20260627-判断2軸と対話kickoff.md`): authorizes へのパス判定・記録は owox(機械) の行動軸。不可逆/核操作の安全境界ゲートゆえ人間が確認して解凍するのは製品意図軸
- 「止まって人間に問え」は散文で課せないので、停止は機械トリガ (ファイルパス) でのみ課す
- guarded 資産に物質化しない純設計判断は可逆コードに留まる限り AI に委ね、後追いレビューで人間が見る

穴C 時間コスト → 最初を厚く + 後追い早期:

- 可逆でも手戻りの作業時間は戻らない。サンドボックスは破壊範囲を限定するが手戻りコストは限定しない
- 一段目ゲート (最初の方針) を厚く握り設計判断の余地を最初に減らす
- 後追いレビューを早く回し手戻りが膨らむ前に止める。手戻りゼロは諦め予防に投資する

### 点2 自動承認パスの再設計

`20260619-承認と自動改善ループ.md` の点2 (auto は既定オフ・人間が CLI 確認付きオン・セッション限り) を、根本方針に合わせて緩める。

既定 = プロジェクト性質で可変 (architecture 軸から導出):

- profile の architecture 軸 (`crates/core/src/profile.rs`) から auto 既定を純関数で導く。layered = オフ・flat = オン
- 組込み5束では architecture と auto 適性が完全相関する (script/research = flat = オン・clean-arch-app/library/data-platform = layered = オフ)。preset 無し (素の新規 = 既定 layered) は慎重側 = オフで自然
- 新しい軸 (5軸目) は足さない。`20260619-承認と自動改善ループ.md` が「可逆性で新軸」を捨てた精神 (二重管理・新次元の保守コスト回避) と整合する

同意源と寿命 = 2系統:

- profile 由来 (性質オン) = profile が固定の間ずっと auto 有効。人間の profile 宣言が同意源。寿命は永続 (profile 固定の間)
- session 由来 (性質オフのプロジェクトで一時的に上げる) = 人間が session で明示オン。寿命は session 限り (切り忘れ防止)

対象操作の既定集合 = 層 × 不可逆の積で機械決定:

- supervised/free 層 かつ 不可逆ガードを通らない操作を自動で auto 対象にする
- 外部副作用は穴A の独立ガードで除外されるので、auto 対象は「supervised/free 層の可逆操作」だけに自動で絞られる
- 別途の許可リストは持たない。層判定と不可逆ガードという既存2機構の合成で済む

後追いキューの保存形式 = 来歴 (Decision) 一本:

- auto 承認した操作と auto 草案 (点4) を `approval:auto` 印 + 未確認状態の Decision として記録する
- session_start / next で未確認件数を表面化し、人間が確認 or 差し戻し (canon 履歴の逆操作)
- `20260619-Phase9-guarded承認で解凍.md` の「来歴を単一の真実源にし cache へ書かない」と同じ精神。確認面は1系統・承認証跡が散らない

## 採用理由

- 根本方針を層追加でなく既存勾配の閾値調整で実現 → 理念を保ちつつ可逆性観を取り込む。新次元を足さず清潔
- 穴A を独立ガードに → 可逆性が成立しない唯一の領域を層と直交させ、auto を緩めても不可逆が漏れない
- 穴B を物質化トリガ限定に → 散文の停止指示に頼らず機械で確実に捕る。捕れない純設計判断は可逆なので任せられる
- 穴C を最初厚く+後追い早期に → 人間の「最初の方針は強く握る」と一致。手戻りは予防で抑える
- 点2 既定を architecture 軸導出に → 新フィールドゼロで最清潔。組込み束では正しく、将来 flat で慎重 / layered で auto が要れば autonomy 軸へ昇格できる
- 寿命2系統に → profile 宣言は人間の明示同意なので永続でよい。一時的な引き上げだけ切り忘れ防止で session 限り
- 対象集合を層×不可逆の積に → 列挙を保守しない。既存2機構の合成で網羅性問題が出ない
- キューを来歴一本に → guarded 解凍と一貫。単一真実源で確認動線を散らさない

## 捨てた案

- 人間と同等権限の新層を追加 → 理念と衝突・二重管理
- auto 既定を一律オン → 慎重なプロジェクトで危険・性質を無視
- auto 既定を一律オフ維持 (旧点2) → 「もう任せてよい」観より保守的すぎる
- autonomy 軸 (5軸目) を新設 → 組込み束では architecture と同結果なのに軸が増えコンテキスト・保守増。実需が出るまで保留
- preset ごとに auto 既定を定義 → 独自束にフィールドが要り実質5軸目に滑る
- auto 寿命を全て session 限り → 固定 profile に対し毎 session 切り直しは冗長
- auto 寿命を全て永続 → session 由来の一時引き上げで切り忘れリスク
- 対象操作の明示許可リスト → 網羅性問題・保守コスト
- 後追いキューを専用ファイル / cache → 二重管理・承認証跡が散る・寿命曖昧 (guarded 解凍で既に却下)
- 純設計判断も AI 自己申告で二段ゲートへ → 散文依存で信頼性が低い
- 着手前チェックポイント (大きな設計判断前に方向性確認) → 「任せてよい」観に逆行・人間負担増

## 見直し条件

- flat で慎重 / layered で auto が実需になった → architecture 導出をやめ autonomy 軸 (5軸目) へ昇格
- 物質化しない純設計判断の手戻りが頻発し後追い早期で吸収しきれない → 着手前チェックポイントを再検討
- profile 由来の永続 auto が運用で広すぎる → profile 性質オンでも操作回数上限など別寿命を検討
- supervised/free でも不可逆操作が auto を通って事故 → 不可逆ガードと auto 承認パスの順序・配線を再点検
- 後追いキューの未確認が溜まり人間が捌けない → 表面化の頻度・粒度を見直し

## 実装余地 (未確定・実装時に詰める)

- architecture 軸 → auto 既定の純関数の置き場 (profile.rs に従属させる)
- auto 承認パスが不可逆ガードを迂回しない配線の具体 (pre_tool_use の順序保証)
- 来歴に `approval:auto` 印 + 未確認状態を持たせる型と、確認 / 差し戻しツール
- session 由来 auto の明示オンと profile 由来 auto の共存配線
