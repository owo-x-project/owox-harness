# 要件の種類分類と PRFAQ の正本の置き場

## 問題

owox-harness の要件管理像で2点が未定義だった。

1. 要件が「機能要件か非機能要件か技術要件か設計要件か」「技術者向けかステークホルダー向けか」が
   モデル上どこにも現れず、何の要件かを問えない。要件正本層 (`20260614-Phase6-requirements正本層.md`) は
   ライフサイクルと受け入れ基準しか持たない
2. PRFAQ で起草した時の正本がどこに置かれるか。`20260618-Phase9-性質軸適応機構.md` は
   「PRFAQ は要件の高度のみ決める・要件起草スキル req の中の起草方法を切り替える」とだけ決め、
   プレスリリース風文書そのものを残すのか・どこに置くのかは未定義だった

## 採用判断

### 正本は要件正本層の一本

PRFAQ でも lightweight でも、正本は `.owox/requirements/<id>.md` のみ。起草方法は保存先を変えない。

- PRFAQ は考え方 (誰が便益を得るか・主要な問い) を引き出す起草層。出力は要件正本へ蒸留する
- プレスリリース風文書は別ファイルとして残さない。二重正本を作らない (どちらが真かの曖昧さと腐敗を断つ)
- 便益・なぜ (PRFAQ の語る部分) は来歴 (decisions) へ残し、要件へ link する。来歴と link は既存機構
- 正本は機械可読で AI・技術者向け。ステークホルダー向けの語りは起草層の役割で、保存形ではない

owox の「検証可能な正本・canon は tool 経由」方針と一致し、将来 owlspec への橋渡しも素直
(型付き・1件1ファイル・充足は導出)。

### 種類は機能/非機能の任意タグ

要件1件に任意の種類タグを足す。値は `functional` / `non-functional` の2値。

- 任意。性質軸 (requirements-shape など) や起草方法と独立で、どの要件にも付けられる
- 機能 = 系がする振る舞い。受け入れ基準のテスト link で検証しやすく met 判定向き
- 非機能 = 性能・安全・可用性などの品質特性。quality.toml の適応度関数や検査委譲で守る側
- 種類で対象者は分けない (本文を技術者向け/ステークホルダー向けに二重化しない)
- 技術要件・設計要件はタグに含めない。技術選定・設計上の制約は要件でなく来歴へ置く
  (core 要件で「技術選定の採用理由・捨てた案・見直し条件は来歴」と正本化済み)。
  要件正本へ同じものを置くと二重管理になる

機能/非機能の2値に絞るのは、4語 (機能/非機能/技術/設計) が直交せず技術・設計が来歴と重なるため。
軸を混ぜず、後で「これは要件か来歴か」と迷わない最小の語彙にする。

## 実装

- core (crates/core/src/requirement.rs): `RequirementKind` (Functional / NonFunctional) を追加。
  Requirement・入力2種へ `kind: Option<_>` を足し、`## Kind` 節で描画・読込。未知値は弾く。
  種類は性質軸でゲートしない (どの要件にも常時付けられる)
- MCP (crates/mcp/src/serve.rs): requirement.create / requirement.update に kind 引数。
  空文字で消す。create のツール説明へ任意 kind を明記
- 要件起草スキル req (crates/core/src/commands.rs): PRFAQ も lightweight も要件正本へ蒸留し
  プレスリリース文書を別に残さない・便益となぜは来歴へ link・種類を機能/非機能でタグ付け・
  技術/設計の制約は来歴へ、を本文へ追加
- cargo 317 passed・clippy 0

## 捨てた案

- 種類タグを足さない: 受け入れ基準の検証 link の有無で機能/非機能を暗に区別できるが、
  種類を問えるようにしたい人間判断 (種類タグ追加) を採った
- 機能/非機能/制約の3値や4値そのまま: 制約 (技術・設計) は来歴の領分で、要件へ置くと二重管理
- PRFAQ 文書を別正本として併存: 来歴性は高いが正本が2層になり、どちらが真かの曖昧さと腐敗リスク
- PRFAQ 生成をやめ lightweight 一本化: 4方法論の出し入れという旗を弱める

## 実機検証と到達性の是正

実機 Codex 対話で req を検証したところ、決めた方法論が一切適用されなかった。エージェントは
profile.get も PRFAQ 起草もせず、要件全体を kind=functional 一括にし、技術前提 (既存ストレージ層を使う)
を受け入れ基準に書き、ideal-first なのに自分で priority=1 を設定し、便益・なぜの来歴も残さなかった。

根本原因: 床ルーティング (crates/core/src/hook.rs) が要件登録の要望を req スキルでなく
requirement.create ツールへ直接飛ばしていた。req スキルは明示起動のみで本文 (方法論) が読み込まれず、
エージェントは方法論を通らず生ツールを叩いた。床は axes (prioritization=ideal-first 等) を注入済みだが、
行動の地点 (ツール) に振る舞いが縛られていないため無視された。生成物・要件は正しいが、
エージェントが実際に通る経路から到達不能だった。

是正1 (3点を組み合わせ):

- 床ルーティングの要件行を req スキルへ向け、非交渉事項 (kind 分類・技術/設計の制約は来歴へ・
  ideal-first は優先度を人間へ) を1行で行動の地点に置く
- requirement.create のツール説明へ同じ非交渉事項を移し、ツールを叩く地点でも見せる
- 機械ガード: ideal-first の時に requirement.create で AI が priority を付けたら弾く
  (mcp serve.rs の ai_priority_blocked・profile 未解決は素通り)。起草時のランク付けを止め、
  ランクは人間が決めた後に requirement.update で残す導線へ寄せる

是正1後の再検証で、中核は成立した — 機能/非機能で要件を分けタグ付け・技術前提を adopted な決定へ分離・
ideal-first で priority を自己設定せず人間へ次手提示・両基準を実在検査へ trace link。しかし PRFAQ 起草
(プレスリリースの枠組みと着手前の人間合意) は依然スキップされた。床1行で req スキルを名指ししても、
PRFAQ の対話フローは req スキル本文にしか無く、エージェントがスキルを実起動しないため発火しなかった
(床行に直書きした非交渉事項だけが効くという同じ構図)。

是正2 (PRFAQ を行動の地点へ直書き):

- 床ルーティングを性質軸連動にし (crates/core/src/hook.rs render_requirement_routing)、
  requirements-shape=prfaq の時だけ PRFAQ 起草 (プレスリリースの枠組みと着手前の人間合意) を1行へ直書きする。
  優先度を人間へは prioritization=ideal-first の時だけ。lightweight や incremental では出さず床を最小に保つ
- 方法論は req スキル本文にも残す (深さの参照)。床は発火の引き金、スキルは詳細、と役割を分ける

ガードの限界: create のランク付けは止まるが update は人間判断の正規チャネルとして許す
(ツールは「人間が決めたランクか AI の自己判断か」を見分けられない)。優先順位付けの完全な人間ゲートは
将来スライス。cargo 319 passed・clippy 0。

是正2後の3回目検証でも PRFAQ のプレスリリース風の語りと着手前確認は不発だった。床へ直書きできる指示は
「AI が止まらず自分でできる」もの (種類分類・制約回し・優先度) に限られ、「止まって人間へ問い返事を待つ」
語りは散文では強制できないと判明。ただし Phase9 決定が定める PRFAQ の荷重部分は語りでなく人間ゲート
(人間が what/why を承認するまで固めない) であり、それは実際に効いていた — 3回とも AI は draft で作り
accepted へ上げず実装に入らず人間へ次手を渡した。

結論: PRFAQ の語りは advisory とし散文で強制しない (低価値で drift を招く)。床の PRFAQ 合図は残すが
強制はしない。AI の自己 accepted は未観測ゆえ機械ゲートは足さない (投機的機構を避ける)。要件分類・
正本一本化・優先度ガードは実機成立として閉じる (`docs/validation/20260620-要件分類とreq到達性.md`)。

## 追補: PRFAQ をツール契約へ昇格 (再開)

上の結論は PRFAQ を advisory で閉じたが、それだと prfaq を設定しても lightweight と行動が変わらない
(draft 据え置きはどちらでも起きる汎用挙動)。設定の意味がなくなる、と人間が再提起した。

再判断: PRFAQ の本質を2つに割る。

- 逆算の蒸留 (誰が・なぜ便益を得るか): AI が止まらず自分でできる obligation。よってツール契約で強制できる
- 着手前の人間合意 (accepted 昇格): 止まって聞く obligation。機械ゲートが要るが自己昇格は未観測ゆえ据え置き

前者をツール層へ昇格する。これは run-2 で実機成立した ai_priority_blocked と同じ機構種別 (床・スキルの
語りでなく、AI が必ず通るツールでの拒否) なので効く見込みが高い。これで prfaq が初めて lightweight と
異なる強制された行動差を生む。

実装:

- core (crates/core/src/requirement.rs): CreateRequirementInput に benefit を追加。指定時は採用済み
  decision として来歴へ記録し要件へ link する (便益・なぜは来歴の領分という上の正本と一致)
- mcp (crates/mcp/src/serve.rs): requirement.create に benefit 引数。prfaq_benefit_missing で
  requirements-shape=prfaq の時に benefit 欠落を弾く (ai_priority_blocked と同型・profile 未解決は素通り)。
  ツール説明へ「逆算・prfaq では benefit 必須」を明記
- 床 (crates/core/src/hook.rs render_requirement_routing): prfaq 行を「逆算しプレスリリースの便益を
  create の benefit へ渡す」に揃え、強制点 (ツール) と一致させた
- cargo 322 passed・clippy 0

実機成立: 同じ統制プロンプトで実機 Codex 対話を再検証 (run-4)。requirement.create 2件とも benefit を
初手で充足し、便益・なぜが採用済み decision として来歴へ残り要件へ双方向 link された。3回目まで空だった
便益の来歴が埋まり、PRFAQ が初めて lightweight と異なる強制された行動差を生んだ。中核 (種類分割・制約を
来歴へ・priority 自己設定せず・draft 据え置き) は4回連続維持 (`docs/validation/20260620-要件分類とreq到達性.md`)。

据え置き: accepted 昇格の人間ゲートは自己昇格を観測してから (投機的機構を避ける owox 方針)。

## 見直し条件

- owlspec が要件仕様管理を引き取り、種類分類を owlspec 側で深く扱うようになった時 (owox の kind を畳む)
- 機能/非機能の2値が現場で粗すぎると分かった時 (制約など値を増やす)
- 種類タグが next 報告など機械処理で使われ始め、ゲートや報告へ載せる必要が出た時
- req スキルへ向けても実機で PRFAQ 起草を通らない時 (床がスキルを起動させる強さが足りない)
- 優先順位付けの完全な人間ゲートを入れる時 (create ガードを update まで広げ承認経路へ載せる)
