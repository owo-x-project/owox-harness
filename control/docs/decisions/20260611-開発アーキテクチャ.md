# 開発アーキテクチャ

## 状態

採用済み。

## 採用案

- Cargo workspace を使う
- crate は core、mcp、setup
- 共通正本+変換の構造にする
- マルチ CLI 変換は Target トレイト+登録表、まず core 内モジュール
- データは target repo 内 .owox/ へ集約し git 管理
- 生成 target harness は各 AI CLI 規定位置へ出力
- 経験の export 型にドメイン固有項を入れず、混入を型で防ぐ
- MCP を主入口にする
- setup は一時実行だけにする
- 常駐 CLI は作らない

## 責務

### core

- TOML を読む
- 設定・正本を検証する
- 正本モデルを型で持つ
- 型付き生成で各 CLI 向けへ変換する
- 機械検証を実行し、完了の3区別を返す
- 経験を汎用型で扱い、export / import する

### mcp

- core を呼ぶ
- AI から使う操作を出す (生成、確認、判断記録、検証実行、ルール成長、経験出入)
- 人間判断が必要な点を AI へ返す
- 失敗理由を構造化して返す

### setup

- 初回導入だけを扱う
- MCP 実行ファイルを配置する
- AI CLI 側の MCP 設定を補助する
- 必要な検査を行う
- 完了後、常駐しない
- core を呼ぶ

## マルチ CLI 変換

- 各 AI CLI の生成方法を Target トレイトで表す
- 登録表で Target をまとめ、べた分岐を避ける
- 初期は core 内モジュール、CLI が増えたら別 crate 化が安い形にする

## データ配置

- 正本、判断記録、経験、検証結果: target repo 内 .owox/ へ集約し git 管理
- 生成 target harness: 各 AI CLI 規定位置 (例 AGENTS.md、.codex/、CLAUDE.md)
- 作業引き継ぎ: git 管理対象外

## 経験分離

- 汎用経験とドメイン固有情報を別の型にする
- export 対象は汎用経験の型のみ
- ドメイン固有項を export 型へ入れず、構造の混入を型で防ぐ
- 本文 (自由テキスト) の意味的な混入は型で防げない。用語集照合・秘密情報走査で検査し、export は人間ゲートを通す

## ディレクトリ構成

```text
crates/
  core/
  mcp/
  setup/
docs/
  concept/
  decisions/
  requirements/
  validation/
tests/
  fixtures/
  integration/
```

## 理由

- 共通判断を core に集める
- MCP と一時セットアップ実行ファイルが同じ中核を使える
- Target トレイトで CLI ごとの差分を変換側へ閉じ込め、core 内のべた分岐を防ぐ
- .owox/ 集約と git 管理で、判断・経験がプロジェクト側に残り、人間が確認・引き継げる
- 生成物と正本を分けることで、再生成で正本が壊れない
- 経験の型分離で、ドメイン固有情報の構造混入をレビュー頼みでなく機械的に防ぐ。本文の意味的混入は検査+人間ゲートで補う
- MCP 主入口にすると、人間が AI を通して使う形になる

## 捨てた案

- 単一 crate
- 最初から CLI ごとに別 crate
- core 内にべた分岐
- データを MCP の外部ストアへ置く
- 生成 harness へ判断・経験を直接混ぜる
- 経験 export 時にフィルタで除去
- 常駐 CLI

## 捨てた理由

- 単一 crate は責務が混ざりやすい
- 最初から CLI ごとに別 crate は、CLI が少ない初期では過剰分割で共通判断が散る
- core 内べた分岐は CLI が増えると腐る
- 外部ストアは git と離れ、プロジェクト側に残る理想と矛盾する
- 生成 harness へ混ぜると正本と生成物が混ざり、再生成で壊れる
- フィルタ除去は漏れで混入リスクが残る
- 常駐 CLI は AI 経由で使う思想を弱くする

## 見直し条件

- core が生成・検証・設定・経験を抱えすぎた時
- CLI ごとの差分が core 内で分岐しすぎた時
- setup が常駐 CLI のように肥大化した時
- MCP で扱う操作が多すぎて人間判断が埋もれた時
- .owox/ の内容が肥大化し、必要情報を探しにくくなった時
- 生成内容が文書中心から構造化データ中心へ変わった時
