# 段階的開示

## 状態

採用済み。旗「最小コンテキスト」を、主張でなく仕組みにする設計。

## 背景

正本を AGENTS.md へ集約した結果、用語集やルールが増えると常時読む量が膨らむ懸念が出た。
特に用語集は重要で「必要時だけ読む」では AI が自分の無知に気づけず誤読する。スタイル等のルールも作業や場所で変わる。

参照: `docs/decisions/20260611-MCP設計.md` (resource の進行的開示)、`docs/decisions/20260611-制御方針.md` (状態適応・トリガ)、`docs/decisions/20260611-target-harness内容.md` (context.toml)。

## 判断基準

- 常時読む量を小さく保つ (測定可能な最小コンテキスト)
- 重要情報を AI が見落とさない (気づきは常に与える)
- 作業・場所でルールが変わるのに対応する

## 採用案

正本の各情報を 3 層に置き、関連する時だけ重い中身を届ける。

- 常時 (気づき): Goal・対象外・安全境界・全体スタイル・用語名 (定義は載せない)
- 関連時 (relevance): 用語定義・スコープ付きルール/スタイル・作業別必読。context 地図と hook / MCP が「その時」に届ける
- オンデマンド: それ以外の詳細

用語集:

- 常時は用語名だけ (「専用の意味を持つ」という気づき)。名前だけなら大量でも軽い
- 定義本体は必要時に owox:// で引く
- 自動注入: プロンプトや編集対象に定義済み用語が現れたら hook がその語の定義だけ差し込む
- 極大時は用語名リスト自体を作業領域でスコープする

見直し (Phase5。`docs/decisions/20260613-Phase5-スキルと入口.md`):

- 用語名の常時掲示は AGENTS.md 静的列挙をやめ SessionStart hook の注入へ移す (動的 CRUD で古びないため)
- 定義の取得は owox:// resource でなく glossary.lookup tool (出現時 push は維持)。用語を tool で CRUD できるようにする

ルール (スタイル含む):

- ルールはスコープ (適用範囲) を持てる
- 全体ルールは常時 (brand)。スコープ付きは context 地図に「パス/作業種別 → 適用ルール+必読」で持ち、その範囲に入った時だけ届ける
- 届ける手段: 編集前 hook (対象ファイルが分かる) / MCP の文脈 tool (作業が分かる)

スコープの照合方式 (context の kind):

- path: 編集対象ファイルのパスを glob と機械的・決定的に照合する (hook が対象を知る)
- task: 作業はファジーで正規表現にできない。作業の宣言 (kickoff / 状態) か AI 判断で照合する
- 決定的にできる path は機械へ、曖昧な task は宣言・判断へ寄せる

## 実装段階

- Phase 2: context の型確定 (作業/パス → 必読 + 適用ルール、スコープ種別)。届ける実装は持たない。用語集・スタイルは小規模なら当面 AGENTS.md へ常時のまま許容
- Phase 3: 編集前 hook が対象に応じてスコープ付きルールを届ける
- Phase 4: owox:// resource で用語定義・詳細を必要時配信。用語名は常時/定義は必要時の分離を実装

## 理由

- 「気づきは常時・中身は関連時」で、重要性と最小コンテキストを両立できる
- スコープを一般概念にすると、スタイルに限らず全ルールへ効く
- 段階を分けると、仕組み (hook / MCP) の前提が揃ってから届ける実装に入れる

## 捨てた案

- 全部常時 (AGENTS.md へ全集約のまま)
- 全部オンデマンド (重要情報も必要時のみ)
- 用語集を定義ごと常時注入

## 捨てた理由

- 全部常時は規模で破綻し、旗に反する
- 全部オンデマンドは AI が無知に気づけず誤読する
- 定義ごと常時は用語集が大きいほど膨らむ

## 見直し条件

- 用語名リストすら常時で重くなった時 (スコープ必須化)
- スコープの粒度 (パス glob / 作業種別) が実運用で足りない時
- hook / MCP の届け方が複雑になりすぎた時
