# canon 編集の統一維持

## 状態

採用 (再確認)。`docs/decisions/20260614-Phase7-経験IOと二層ルール.md` の統一 canon 編集モデルを維持する判断を、マネージャ問いを受けて明文化する。

## 背景

問い: canon.add / canon.propose を brand.* / glossary.* などへ型別分割した方が AI が迷わないのでは。

現状の表面 (`crates/mcp/src/serve.rs`):

- 書き: canon.add (追加・AI 直接 + 来歴)、canon.propose (変更 / 削除・人間ゲート)。どちらも target = brand / rules / practices / glossary を引数で取る
- 読み: glossary.lookup (1語)、rules.lookup (rules 全文)

Phase7 で glossary.add / update / remove・practice.add・旧 canon.propose を canon.add / canon.propose の2つへ畳んだ経緯がある。問いは、その逆 (分割) の是非。

## 判断

統一を維持する。分割しない。

理由:

- 編集のセマンティクスが全 canon 種で同一。add = 追加・AI 直接・来歴、propose = 変更/削除・人間ゲート。brand でも glossary でも practices でも振る舞いが同じ
- 型で割ると、中身が全く同じ tool が4つ並ぶ。AI から見て「どれも同じことをするのにどれを呼ぶ?」となり、むしろ迷う
- tool は振る舞いで割る・データ型で割らない。読みを glossary.lookup (1語) と rules.lookup (全文) に分けたのは返る形が違うから (正しい分割)。書きは形が同じなので統一が正しい
- 実機 Codex 対話で canon.add / canon.propose は迷わず回ると確証済み (`docs/validation/20260616-Phase7-対話検証と粗の是正.md`)

## 是正 (分割でなく)

迷いの実体は canon.add の説明文が全 target の section 語彙を詰めて長い点。直す先は分割でない:

- canon.add / canon.propose の説明を簡潔化する (target ごとの section 語彙を詰め込みすぎない)
- 不一致時に有効な section / item を返すフィードバックを効かせる (canon.propose は既に現項目を data.items で返す。canon.add でも未知 section 時に有効値を返す)

優先度は低い (実害は確認されていない)。Phase8 の説明文整理のついでに行う。

## 見直し条件

- 実機で target / section 引数の取り違えが頻発した時。その場合も分割でなく、まずフィードバック改善 (有効値の提示) を試す
