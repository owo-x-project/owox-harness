# マルチCLI生成

採用判断。2026-06-21。
Phase9 の差別化軸「特定 AI に閉じない」を、第1対象 Codex に続く第2対象 Claude Code の生成で実現する。
本書は Target 抽象の据え方と、各 CLI の hook / MCP / skills 写像の判断を束ねる。
第3対象は当初 Gemini CLI を経て OpenCode へ変更したが、2026-06-21 の実機検証でクラッシュ・OS 停止を引き起こしたため今回スコープから除外した。将来の追加対象として残す (同じ Target 抽象に乗る)。

## 背景: 3 CLI の機構差 (一次情報)

owox は subagent や hook を自前実行せず、各 CLI ネイティブ機構へ生成する側。CLI ごとの差は配置先と設定形式に閉じ、決定論ロジックは core (`owox hook <event>` バイナリ) に集約する。

- Codex: hook は `.codex/hooks.json` に event→command 登録。MCP は `.codex/config.toml` の mcp_servers。skills は `.agents/skills/<id>/` + 固有メタ `agents/openai.yaml`
- Claude Code: hook は `.claude/settings.json` の hooks で event→command 登録 (Codex と同形)。MCP は `.mcp.json` の mcpServers。skills は `.claude/skills/<id>/`
- OpenCode: hook 相当は `.opencode/plugins/` の JS/TS プラグイン (event→関数。command 登録は無い)。MCP は `opencode.json` の mcp。skills は Agent Skills (別途調査で確定)

Codex と Claude Code は hook が command 登録で同形 → `owox hook <event>` バイナリをそのまま流用できる。OpenCode だけ hook がプラグイン (関数) で、owox バイナリを直接登録できず JS/TS の薄い橋渡しが要る (第3対象の節で扱う)。

## 採用した案

### Target 抽象は既存のまま流用

CLI 追加は `crates/core/src/target.rs` の registry へ 1 行・`crates/core/src/targets/<cli>.rs` を 1 ファイル足すだけ。`Target` trait (name / generate / generate_skills) と `write_all` の書込み方法 (Overwrite / MergeToml / MergeJson) で CLI 差を吸収する。core 本体にべた分岐を作らない。

### 書込み方法に MergeJson を追加

人間と共有する JSON 設定 (Claude Code の settings.json・OpenCode の opencode.json) へ owox 管理ブロックだけ差し込み、人間の他設定 (permissions・env・別 MCP サーバ) を壊さない。セマンティクスは既存 MergeToml と同型: オブジェクト同士は深く潜り、配列・スカラは断片で置換。owox 断片はトップレベルで自分のキーだけ持つので人間の他キーは残り、owox が管理する配列だけ断片で置換され再生成で冪等。実装は `crates/core/src/target.rs` merge_json / merge_json_value。

### 第2対象 Claude Code (実装済)

- 生成物 = hook 登録 (`.claude/settings.json` の hooks・MergeJson) + MCP 登録 (`.mcp.json` の mcpServers.owox・MergeJson) + skills (`.claude/skills/<id>/`)。`crates/core/src/targets/claude.rs`
- ルート指示ファイル (CLAUDE.md) は生成しない。床コンテキストは SessionStart hook が live 注入する (Codex と同じ・編集可能な複製を置かず canon 直編集経路を断つ)
- hook は薄いシェルを介さず owox バイナリを直接呼ぶ。Claude Code の hook 入出力契約は Codex 踏襲の契約と同形 (snake_case 入力・hookSpecificOutput / permissionDecision=deny / decision=block 出力) なので `owox hook <event>` を流用
- 登録 event は4種 (SessionStart matcher=startup|resume|compact・PreToolUse matcher=Bash|Edit|Write|MultiEdit・UserPromptSubmit・Stop)。Codex と同じ4契約
- 編集 tool の被覆: Claude Code は専用 tool で編集する。Edit / Write は Codex・Claude Code 共通、MultiEdit は Claude Code 固有の複数編集 tool (どれも file_path を持つ) なので層ゲート (`crates/core/src/hook.rs`) と PreToolUse 判定 (`crates/mcp/src/hook.rs`) の編集 tool 集合へ MultiEdit を追加
- NotebookEdit は notebook_path で入力形が別 (file_path でない) ため v1 は対象外。Jupyter 編集を捕捉する必要が出たら入力正規化で別途対応
- SKILL.md は横断標準なので文面を verbatim 出力。Codex の openai.yaml のような自動起動可否の明示制御は Claude Code の skill frontmatter に対応フィールドが無いため出さない (v1)


## 採用理由

- Target 抽象流用: CLI 追加が registry 1 行 + 1 ファイルで済む既存設計が効いた。core にべた分岐を増やさず清潔
- MergeJson: 共有設定を壊さず冪等に owox ブロックだけ管理。MergeToml と同型で実装・理解コストが小さい
- 床は hook 注入で複製を置かない: Codex で確立した方針 (canon 直編集経路を断つ・圧縮再開後も床が戻る) を CLI 非依存で貫く。マルチ CLI でも canon が単一真実源
- `owox hook <event>` 流用: command 登録が同形の Codex・Claude Code は配管だけで核ロジックを共有。OpenCode はプラグインで橋渡しが要るが、橋は I/O 翻訳に徹し決定論は core に残す

## 捨てた案

- ルート指示ファイル (CLAUDE.md) を生成 → 編集可能な複製ができ canon 直編集経路が開く・圧縮注入は hook が担う (Codex で廃止済の再来)
- settings.json / opencode.json を Overwrite で丸ごと持つ → 人間の他設定を壊す。MergeJson で共存
- 各 CLI ごとに hook ロジックを再実装 → 決定論が分散し腐る。`owox hook` 一本へ集約
- NotebookEdit を v1 で無理に被覆 → 入力形が別で正規化が要る。需要が出てから

## 要実機確認 (target 検証で潰す)

- Claude Code: 生成した settings.json hooks が4 event で発火するか・PreToolUse の deny / Stop の block が効くか・SessionStart の additionalContext で床が届くか・MultiEdit が層ゲートに載るか・.claude/skills/<id>/ が読まれ $起動するか

## 見直し条件

- Claude Code の hook / settings スキーマが変わった → `targets/claude.rs` を追従
- NotebookEdit 等 file_path を持たない編集 tool の被覆が要る → hook 入力正規化を設計
- Gemini CLI / OpenCode 等を本格対象化する → registry へ追加 (同じ Target 抽象に乗る)
