# Phase6 スライス4: plan-alignment + 多視点レビュー枠組み (routable) の設計

## 状態

採用 (実装前)。Phase6 スライス4 (`docs/requirements/20260614-Phase6-検証を閉じる.md`)。
判断型レビューの観点・進め方を owox 標準として持ち、変更内容で振り分け可能 (routable) にする。subagent 実行・ティア振り分けは Phase8。

## 背景

検証は2系統 (`docs/decisions/20260611-品質保証.md`)。決定論チェックは verify.run と quality 適応度関数 (スライス1〜3) で機械判定した。残る判断型レビュー (セキュリティ・アーキ整合・規約逸脱・plan-alignment) は多視点+敵対レビューが担う。入口=薄い skill の review は Phase5 で配置済みだが本文は最小の仮置き。

人間との確認で2点を決めた:

- 観点は owox 標準のベストプラクティス。各観点の具体バーと観点の追加はプロジェクト (層で分ける)
- 変更内容でレビューを振り分ける。枠組みを routable に作り、subagent 実行・ティア振り分けは Phase8 が同じ契約を消費する

## 採用設計

### 観点は owox 標準・具体はプロジェクト (層で分ける)

役割分担 (`docs/decisions/20260611-品質保証.md`) どおり:

- 観点 (lenses) 自体は owox 標準。「どう厳しくレビューするか」を再発明させない
- 各観点の具体バー (規約・層境界・予算) は rules.md / quality.toml が持つ
- 観点の追加・上書きはプロジェクト (review.toml と commands.toml)

### 標準の観点 (owox ベストプラクティス・離散単位)

各観点を独立に走る離散単位にする (Phase8 が観点 → subagent へ 1 対 1 で割り当てられる):

- correctness: 意図どおり動くか、境界・失敗時を含む
- design: 責務分割・依存方向・層境界に沿うか
- security: 秘密・危険操作・外部送信・退行リスク
- plan-alignment: 既存の命名・構造・流儀に沿うか、車輪の再発明をしていないか
- requirement: 変更が要件と受け入れ基準に結びつくか
- pruning (庭師): 変更が残した不要 (死んだコード・未使用・重複・取り残した足場) を見つける

### 振り分け可能 (routable): 観点に適用トリガを持たせる

各観点は適用条件 (applicability) を持つ。変更が触る内容で適用観点を機械選択する。

- 適用条件は Always か Paths (glob の並び) のどちらか。Paths は変更が触ったファイルがその glob に当たる時だけ適用する
- glob 照合は quality 適応度関数 (スライス3) の glob→regex を再利用する (新規依存なし)
- owox 標準の既定: correctness / design / security / plan-alignment / requirement / pruning は Always (どの変更にも普遍に効く核)。加えて言語横断で既知の依存マニフェスト (Cargo.toml / package.json / go.mod / pyproject.toml / Gemfile 等) が変わった時に plan-alignment を強める Paths トリガを既定で持つ (依存追加は規約・既存資産レビューの要)
- 変更が触ったファイルは mcp が git diff で集めて core の選択へ渡す (core は git を持たず決定論。today / known_checks と同じ流儀)

### 機械選択は tool・進め方は枠組み (skill 本文)

- review.lenses tool: 今の変更 (git diff) に適用される観点を機械選択して返す。決定論部分を tool に置く
- review 入口 skill の本文: 進め方の枠組み。review.lenses で適用観点を取り、verify.run の機械検証済みの事実を取り込み、各観点で 指摘 → 事実性確認 → 敵対的再確認 を回す。pruning は庭師の安全規則を通す
- 決定論 (どの観点が効くか) は tool、判断 (実際のレビュー) は AI、という分担。完了3区別が機械と判断を分けているのと一貫

### 枠組みは実行モデル非依存 (Phase8 がそのまま消費)

観点と進め方を「何エージェントで回すか」と切り離して契約化する。

- 本スライス (Phase6): 単一エージェントが review.lenses の適用観点を順に回す
- Phase8: 同じ select_lenses 契約をルータが読み、適用観点ごとに subagent を 1 対 1 で立て、変更のリスク/規模でモデルのティアと数を機械的に振り分ける (`docs/decisions/20260611-オーケストレーション.md`)。枠組みを書き換えず実行だけ載せる

枠組み=契約、実行=差し替え可能。これが「枠組みは owox 標準・実行は Phase8」を設計で担保する。

### 剪定は庭師の安全規則を必ず通す (盲目削除しない)

育てる (農家) と対の刈り込む (庭師) を一級概念としてレビューにも持たせる (`docs/decisions/20260613-庭師-剪定.md`)。

- レビューは「これは不要では」を指摘・提案するだけ。その場で消さない
- 実際の削除は削除基準 (rules) を確認し、剪定後に verify を通して振る舞い不変を機械で確かめ、何を・なぜ剪定したかを来歴に残す。不可逆な削除は人間ゲートへ (既存 hook の不可逆 deny)
- 対象は本レビューの変更が残した不要に絞る。repo 全体の腐敗検知は Phase7、庭師 subagent / 掃除 skill の実行は Phase8

### plan-alignment の決定論部分との接続

plan-alignment の機械検証できる部分 (層境界・依存方向・複雑度予算) は quality 適応度関数 (スライス3) が弾く。レビューは「機械で出ない規約逸脱・既存資産の無視」へ集中し、機械判定済みを二重に裁かない。

### プロジェクト拡張: review.toml

owox 標準の観点に、プロジェクトが review.toml で観点を追加・上書きできる (commands.toml と同じ upsert)。

- `[[lens]]` = id / description / when (always か paths)
- プロジェクト固有の sensitive パスへ観点を当てる (例: src/auth/** に security を強める、migrations/** に専用観点)。owox は言語非依存で sensitive パスを知れないため、ここが per-project の振り分け点
- 無ければ owox 標準のみ (opt-in)

## 変更境界

触るもの:

- crates/core/src/review.rs (新規): Lens / Applicability 型、standard_lenses、select_lenses、review.toml 読込
- crates/core/src/commands.rs: review 入口の本文を枠組み (review.lenses を取る + 進め方 + 庭師安全) へ書き換え
- crates/mcp/src/serve.rs: review.lenses tool、get_info 追記
- crates/mcp/src/files.rs: 変更ファイルの列挙 (git diff)
- crates/core/src/lib.rs、docs

触らないもの:

- subagent オーケストレーション・役割別権限・モデルのティア振り分け (Phase8)
- ルータ本体 (適用観点 → subagent 並列・敵対実行) (Phase8)
- repo 全体の腐敗検知 (Phase7)
- verify / quality / requirements (スライス1〜3)

守る振る舞い:

- review 入口は明示起動のみ・テスト不要
- 生成文はツール非依存・CLI 名なし・括弧の補足なし
- core は git を持たず決定論 (変更ファイルは mcp が渡す)
- review.toml 無しなら owox 標準のみ (opt-in)

## 危険と最善解 (緩和でなく設計で消す)

- Phase8 との重複・単一で回らない: 最善解は枠組みを実行モデル非依存にすること。観点を離散単位にし select_lenses を契約化する。Phase6 は単一で順に回し、Phase8 は契約を読んで subagent へ割り当てるだけ。書き換えが要らず重複が原理的に生じない
- 機械判定済みを二重に裁く: 最善解は事実の層を分けること。verify.run / quality / 要件 trace が base 層、レビューはその補集合だけ。進め方の第1ステップに「機械検証済みは再判断しない」を固定する
- 剪定の盲目削除: 最善解は庭師の安全規則を枠組みへ焼くこと。提案のみ・削除基準 + 剪定後 verify + 来歴・不可逆は人間ゲート
- 振り分けの誤り (観点の付け過ぎ/漏れ): 最善解は既定を安全側 (普遍観点は Always) にし、絞り込みは Paths トリガの追加 (opt-in) にすること。既定で漏れない。トリガは glob で機械的・決定論
- 常時コンテキスト圧迫: 最善解は入口 skill が明示起動 (オンデマンド) であること。常時コストは構造的にゼロ。事実は tool から取り本文へ再掲しない

## 検証方針

- 単体: standard_lenses に 6 観点が揃う、select_lenses が Always を常に含み Paths を変更ファイルで絞る、依存マニフェスト変更で plan-alignment が選ばれる、review.toml の追加・上書き、glob 照合、review 本文に進め方 (事実取り込み・事実性確認・敵対的再確認) と庭師安全が含まれツール非依存
- 結合: review.lenses tool が今の変更の適用観点を返す、command_skills が review を implicit=false で生成
- 手動 (後の一括対話検証へ): 実機 Codex で review が適用観点を取り多視点+敵対レビューを回す、機械検証済みを二重に裁かない、剪定が盲目削除へ走らない、依存追加で plan-alignment が効く
- 未確認として残す: subagent による並列・敵対実行とティア振り分け (Phase8)・repo 全体の腐敗検知 (Phase7)

## 捨てた案

- レビュー実行 (多視点 subagent 並列・ティア振り分け) を本スライスで作る → Phase8 のオーケストレーションと重複。routable な契約までを Phase6 に置き、実行は Phase8
- 観点をプロジェクト任せにする → ベストプラクティスを再発明させ堅牢さが落ちる。観点は owox 標準・具体と追加はプロジェクト
- 適用トリガに content スキャン (秘密検出等) を入れる → 言語/内容依存で誤検出。Paths (glob) の決定論トリガに絞り、内容判断は AI レビューへ
- 機械判定済み (quality/trace) もレビューで再判断 → 二重で非効率。verify.run の事実を取り込み補集合へ集中
- 振り分けを skill 本文の散文に委ねる → 機械選択にならない。決定論は review.lenses tool・select_lenses に置く

## 見直し条件

- Phase8 で subagent オーケストレーションを作る時、select_lenses 契約を実行側が消費する形へ寄せる
- 標準観点の既定トリガ (依存マニフェスト等) が言語横断で過不足と分かった時
- review.toml のトリガ表現 (glob) が sensitive パスの表現に足りない時
- レビュー本文が肥大し簡潔さを欠いた時
