# 判断 2 軸と対話 kickoff

## 状態

採用済み。`docs/decisions/20260611-方向付け.md` を精緻化。
kickoff の質問形を定める部分は `docs/requirements/20260625-mission-kickoff.md` を改訂する根拠。

## 背景

2026-06-27 の対話検証（`docs/validation/20260627-要件群対話検証.md`）で人間所見。

- kickoff が機械的。grill-me のような共同決定がなく AI が考えを押し付ける
- kickoff 外でも AI が雑に人間へ判断を仰ぐ

理想は「行動の判断は全て owox が示す。人間は何をどう作りたいかだけ示す。AI が必要に応じて助言し実作業する」。

理念正本（`docs/concept/企画書.md`）の「人間が重要判断を握る」と理想を合わせると、これまで一語「判断」で扱っていたものが 2 種類に割れる。今の tool・skill はこの 2 軸を混同し、行動軸まで人間へ質問し、製品意図軸を引き出さず即推奨を出していた。

## 採用: 判断 2 軸

判断を 2 軸に分け、全 tool・skill はこの軸で振る舞いを決める。

### 製品意図軸（人間が握る）

何を作るか / 誰のため / 成功条件 / リスク許容 / 止める操作 / 優先順位。
owox はこれを引き出し、確定後に正本へ残す。owox が代わりに決めない。

### 行動軸（owox が握る）

どの tool をいつ使うか / verify の実行可否 / skill 化 / profile / rules 下地 / 索引更新 / 庭師。
owox が断定的に示す。AI は人間へ打診せず、即興もしない。

### 人間 gate の掛け先

人間確認は製品意図正本（requirements / decisions）・安全境界・不可逆操作のみ。
行動軸の生成物（profile / 索引 / setup 書込など）は owox が決めて書いてよい。

## 採用: 対話 kickoff

`docs/decisions/20260611-方向付け.md` の grill-me 原則「各問に推奨案」を、製品意図軸の質問に限定する形へ精緻化。固定段階チェックリストの自販機型を、質問種別で出し分ける対話型へ変える。

### 質問 3 種

- 引き出し: 開いた質問。推奨なし可。AI は理解を反映し鋭い追撃を 1 問
- 確認: AI が決めた構成・理解を提示。人間は否認 / 修正だけ
- 判断: 人間しか決められない意図 / 安全の分岐のみ。推奨+理由+選択肢

「全質問に推奨必須」は撤回。引き出しに推奨を貼ると押し付けになる。

### 軸での段階仕分け

harness 構成（rules / skill / tool / profile）は行動軸。人間質問でなく owox 決定 → 確認へ格下げ。
目的 / 成功条件 / 安全境界 / リスク許容は製品意図軸。引き出し対話で詰める。

### 進め方

人間は自然発話でよい。AI が owox を代理操作（`mission.start` / 記録）する。
人間が tool を直接叩く前提を外す。

## 採用: 行動軸の権威化（kickoff 外）

`next` は 2 つを分けて返す。

- 次に人間が決める製品意図（gate）
- AI が今すぐやる行動（owox が断定）

これで「雑に仰ぐ」も「指示外の即興」も同時に塞ぐ。検証の entry-routing 常時 advisory・指示外 tool 実行・余分な verify.run は、行動方針が権威的でないことの症状なので、ここで解く。

## 捨てた案

- 全判断を一語で扱い続ける → 軸混同が温存され押し付けと過剰打診が残る
- 全質問に推奨必須を維持 → 引き出しが押し付けになる
- harness 構成を人間質問のまま残す → 行動軸を人間に背負わせ理想と逆行
- P 群を個別バグとして対症療法 → その場しのぎ。軸違反として直せば P1 と余分書込が同時に解ける

## 見直し条件

- 行動軸を owox が断定した結果、人間が握りたい判断まで owox が決めてしまった時
- 引き出し対話が冗長で疲労を生んだ時
- 確認の格下げで人間が構成を把握できなくなった時
- owlspec が要件仕様を扱い始め、製品意図軸の境界調整が要る時
