# 設定の統一 canon 化

## 状態

採用済み（未実装）。`docs/decisions/20260616-canon編集の統一維持.md` と `docs/decisions/20260614-Phase7-経験IOと二層ルール.md`（統一 canon）を拡張。実装計画は `docs/requirements/20260628-設定の統一canon化.md`。

## 背景

target harness の設定のうち prose 正本（brand / rules / practices / glossary）は canon.add / canon.propose で AI が変更できる。一方 TOML 設定（config / quality / release / agents）と context.md は AI が書く口が無く手動編集しかない（`crates/mcp/src/serve.rs` に書込 tool 不在・`crates/core/src/canon.rs` の target_file が4種だけ写像）。

owox の理念「AI の自律性を最大限に活かす」に対し、設定が AI 経由で完結しないのは矛盾。2026-06-28 の人間判断で穴を埋める方針を確定。

## 採用: canon を正本変更の統一窓口へ

canon を「prose 追記」から「正本変更の統一窓口」へ抽象化する。窓口 tool（canon.add / canon.propose / gate.approve 適用 / canon.revert）は据え置き、内部を backend で分岐する。

### prose backend（既存維持）

brand / rules / practices / glossary / context.md。追加は即時（canon.add）・変更削除は人間ゲート（canon.propose）。context.md は文脈マップで副作用が小さく prose と同質なのでここに置く。

### structured backend（新規）

config.toml / quality.toml / release.toml / agents.toml。全操作が人間ゲート（canon.propose のみ・canon.add では受けない）。エントリは構造化引数で受け、スキーマ検証 → 既存 TOML を読み → 構造体操作 → 書き戻し。変更削除の照合は target ごとの識別子フィールドで行う（verify.checks は name / layers は name / boundaries は paths / targets は cli 名 / release.artifacts は path / 単一値はキー直指定）。

## 採用: 設定値は全て人間ゲート

structured backend の追加・変更・削除を全て canon.propose 経由の人間ゲートにする。理由は判断2軸（`docs/decisions/20260627-判断2軸と対話kickoff.md`）。TOML 設定値は副作用が大きく、応答言語・層境界・配布方針・オーケストレーション編成など製品意図軸と安全境界に触れるため、人間が握る。prose の「追加即時」は副作用が小さい成長層だから許せた緩和で、設定値には広げない。

## 捨てた案

- 項目別の専用 tool（config.set / quality.set 等）→ 同一振る舞いの tool が並び発見性が落ちる。`docs/decisions/20260616-canon編集の統一維持.md` の「tool は振る舞いで割る」に反する
- 設定値も prose と同じく追加即時 → 副作用の大きい設定を AI が無確認で変える。判断2軸に反する
- 文字列追記の延長で TOML を書く → 入れ子テーブル・複数フィールド・スキーマ検証に対応できず壊れる
- mode 統合で窓口を1 tool へ畳む → `docs/decisions/20260628-tool整理仕上げと仕様管理取り込み.md` で見送り済。destructive_hint と必須引数の出し分けが効かず安全設計を壊す

## 見直し条件

- TOML 書き戻しで人間のコメント・キー順序が失われる実害が出た時（toml_edit 採用で回避可・依存追加とのつり合いで判断）
- 識別子フィールドの照合が誤り、変更削除を取り違えた時
- 設定の全ゲート化が実運用で重く、行動軸へ寄せたい設定が出た時
- agents.toml など構造化エントリのスキーマが複雑すぎ、1 引数に収まらない時
