## 採用案

- 実装言語: Rust
- 正本・設定形式: 型構造体を中核。出入り口の形式は Markdown へ差し替え済み (`docs/decisions/20260612-正本フォーマット.md`。当初 TOML)
- 配布方法: GitHub Releases + setup.sh + 一時セットアップ実行ファイル。checksum を同梱し、setup.sh がダウンロード後に検証する
- 生成方式: 型付き生成 + 薄いテンプレート
- 検証方法: Rust 統合検証
- 利用入口: MCP 主入口、常駐 CLI なし

## 理由

- Rust は単一実行ファイル配布に向く
- Rust の型 (enum 網羅、serde) が最も厳しく、型付き正本の構造保証を裏打ちする
- 中核を型構造体に置き、TOML はその出入り口にすると、人間が編集しやすく曖昧さが無い
- GitHub Releases は MCP 実行ファイルと一時セットアップ実行ファイルを配布しやすい
- 型付き生成は依存を増やさず検証しやすい。薄いテンプレートは出力文の整形だけ担う
- Rust 統合検証は一時 target repo 生成、再生成、差分・内容確認を一体で扱える
- MCP 主入口にすると、人間が AI を通して owox-harness を使える
- setup.sh は初回導入だけ担い、複雑な処理は一時セットアップ実行ファイルへ逃がせる
- 安全性を要件領域に持つ以上、自分の配布物の完全性確認 (checksum 検証) は前提になる
- 常駐 CLI を置かないことで、AI 経由の利用を基本にできる

## 捨てた案

- Go
- TypeScript
- YAML
- JSON
- テンプレートエンジン全面採用
- 静的ファイルコピー
- cargo install 中心の配布
- 常駐 CLI

## 捨てた理由

- Go は単一バイナリに向くが、型表現が Rust に劣り構造保証の旗が弱まる
- TypeScript は MCP 公式 SDK が厚く開発は速いが、単一バイナリ配布が弱く、型保証が実行時に剥がれる
- YAML は値解釈が曖昧で事故が起きやすい
- JSON は人間が手で編集しにくい
- テンプレートエンジン全面採用は依存と規則が初期から増える
- 静的ファイルコピーは設定反映とマルチ CLI 差分に弱い
- cargo install 中心は開発者向けに寄り、利用者導入の説明が増える
- 常駐 CLI は AI 経由で使う前提を弱くする

## 見直し条件

- 初期実装で依存が増えすぎた時
- 生成規則が複雑化した時
- マルチ CLI 対応で構造が崩れた時
- GitHub Releases の配布自動化が重くなりすぎた時
- Rust 実装で MCP 対応が遅くなりすぎた時
- 一時セットアップ実行ファイルが常駐 CLI のように肥大化した時
- brand / rules の長文を TOML で人間が編集するのが苦痛になった時
- checksum で足りず署名配布が必要になった時
