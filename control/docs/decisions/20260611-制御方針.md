# 制御方針

## 状態

採用済み。

## 背景

MCP は存在するだけでは使われない。
「大量指示して AI に守らせる」方式は owox が否定した前提 (読み落とす)。
だから機械で効かせるのを軸にする。

- `docs/decisions/20260611-MCP設計.md`
- `docs/decisions/20260611-設計原則.md`

## 3層で制御する

### 機械強制 (hook、決定論)

CLI イベントで必ず発火する。AI の遵守に依存しない。

- session 開始: 文脈地図を出す (必読 + 作業に応じた読み物)。次の一手 (owox://next = 次に決めること + ready タスク) と open 未決を表面化する
- 編集・危険操作の前: 安全性・可逆性境界をチェック。不可逆なら gate (needs_human) で止める
- 完了・commit 前: verify 必須、完了条件、差分要約、未記録判断の検出

### 指示 (ルート指示ファイル、最小)

- 「いつ・何の tool / resource を使うか」を短く書く
- 詳細は rules 正本から生成する。長文ルールにしない (最小コンテキスト)

### 誘導 (tool 返り値の next_actions)

- 各 tool 結果が次の手を示し、ワークフローを繋ぐ (record → verify → …)

## 強制と誘導の境界

- 機械強制: 安全性 / 不可逆操作 / 完了検証
- 誘導: 判断記録 / skill 作成・改善

機械検証できる規律は hook で強制し、できないものは指示+誘導にする。

## 判断 2 軸との対応

`docs/decisions/20260627-判断2軸と対話kickoff.md` の 2 軸を、この 3 層へ次のように写す。

- 製品意図軸（人間が握る）: 機械強制で守る。人間 gate は製品意図正本・安全境界・不可逆操作のみ
- 行動軸（owox が握る）: 誘導で示す。tool の next_actions は次の行動を断定形で示し、AI は行動軸を人間へ打診しない（「雑に人間へ判断を仰ぐ」を禁ずる）
- 行動軸の生成物（profile / 索引 / setup 書込など）は owox が決めて書いてよい。人間 gate へ回さない

## 非準拠時 (AI が MCP を使わず進む)

- 機械強制点 (危険操作前、commit / stop) は常にブロックする
- それ以外はプロジェクト状態で変える
  - 初期: 警告
  - 安定: 中間
  - 保守: ブロック
- 状態は state.toml の人間宣言が正。適応度関数は現物との乖離を警告するのみ (回帰防止と一貫)

## トリガ定義

- rules.toml に「いつ → 何を (必須 / 推奨)」を型で定義する
- 指示と hook へ生成する
- CLI 差分は Target が吸収する

## hook イベント対応 (例)

- session 開始 → 文脈地図
- 編集・危険操作の前 → 安全性・可逆性チェック、不可逆は gate
- commit / stop の前 → verify 必須、完了条件、差分要約、未記録判断の検出

## 採用理由

- 機械強制は読み落としに強く、owox の前提と一貫する
- 安全・不可逆・完了だけ強制すると、止めるべき所だけ確実に止まる
- 判断記録・skill を誘導にすると、作業が重くならない
- 非準拠の状態適応は、初期の試行錯誤と保守の回帰防止を両立する
- rules.toml 集約は、指示と hook の二重管理を避け、CLI 差分を Target へ閉じ込める

## 捨てた案

- できる限り全部を機械強制
- 全部を指示+誘導 (hook 最小)
- 非準拠を常にブロック
- 非準拠を警告のみ
- トリガをルート指示ファイルや hook へ直書き

## 捨てた理由

- 全部強制は作業が重く、起動回数が増える
- 指示+誘導だけは読み落とし問題が残る
- 常にブロックは初期の試行錯誤を妨げる
- 警告のみはゲートの意味が薄れる
- 直書きは二重管理になり、指示と機械強制がずれる

## 見直し条件

- 機械強制が誤発火で作業を妨げた時
- 状態適応の境界が運用で曖昧になった時
- rules.toml のトリガ定義が複雑化した時
- hook を持たない AI CLI を本格対応する時
