# Stop で検査を自動実行し失敗を block

## 状態

採用・実装済み。`docs/decisions/20260613-Phase4-tool記録層.md` の「stop は誘導のまま (毎ターン発火するため再実行しない)」を supersede。

## 背景

旧 Stop は完了前に「verify.run を走らせて」とプロンプトで促すだけ。実行は AI 任せ。
AI が促しを無視すると壊れたまま停止し、人間が気づくか次の commit ゲートまで漏れる。
促しに頼ると AI が無駄に往復する (促し→自分で verify→また促し)。

旧判断の理由は「stop は毎ターン発火するので検査再実行は重い」。だが直前 verify.run / commit と
作業ツリーが同一なら結果を再利用する署名キャッシュが既にある (commit ゲートが利用)。これを
Stop も使えば、検査は内容が変わった時だけ走り、毎ターンの重複実行は起きない。前提が変わった。

## 採用: Stop で検査を自動実行

変更があるターンの終わりに検査を機械実行し、結果で振る舞いを変える。

- 失敗 → 停止を保留 (block) し、失敗した検査名を添えて修正へ向ける。prom だけにせず必ず走らせる
- 通過 / 検査未設定 → 新規に検証した瞬間だけ判断記録を一度促し、再利用 (既に見た内容) は黙る
- クリーン or 未決 gate ゼロ → 黙って終わる (ノイズを出さない)

検査は直前 verify.run / commit と作業ツリーが同一なら記録を再利用、無ければフレッシュ実行して
記録する (commit ゲートと共用の署名キャッシュ。二重実行を避ける)。Stop でフレッシュ実行した結果も
記録するので、続く commit ゲートが再利用できる。

## 採用: 1 ターン高々 1 回 (スタック防止)

block・継続は `stop_hook_active` が false の時だけ。既に継続済みなら受理する。
直せない検査 (既存失敗・flaky) で AI が延々スタックしトークンを浪費するのを防ぐ。
直らないまま停止しても commit ゲートが最終防壁として commit を block するので、壊れたコードは
確定しない (機械強制は不変。`docs/decisions/20260613-tool記録層.md` の commit ゲート)。

## 責務

検査の実行・記録の読み書きは mcp (hook バイナリ・cached_verify_outcome)。
合否と gate から block / 促し / 受理を決める純判定は core (stop_decision)。
commit ゲートと Stop が同じ署名キャッシュ・同じ検査実行ヘルパを共用する。

## 捨てた案

- 促しのまま AI 任せ → 壊れたまま停止が漏れる・無駄な往復が残る (今回の動機そのもの)
- commit ゲート相当まで Stop で見る (検査+品質+ブランド+構造腐敗) → 毎ターン発火には重い。
  検査の合否だけが「壊れたまま停止」の核心。品質・腐敗は verify.run tool と commit ゲートが担う
- 直るまで毎ターン block → 直せない検査でセッションがスタックしトークンを浪費 (目的と逆行)

## 見直し条件

- 速い検査でも毎エピソード実行が体感の停止遅延を生んだ時
- 1 回 block で直らず停止する事例が頻発し、commit ゲートまで漏れる手戻りが目立つ時
- 検査が flaky で誤 block が頻発した時 (検査側の安定化が先だが、Stop の扱いも再考)
