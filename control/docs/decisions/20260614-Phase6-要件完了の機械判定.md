# Phase6 スライス2: 要件↔テスト trace と要件完了の機械判定の設計

## 状態

採用 (実装前)。Phase6 スライス2 (`docs/requirements/20260614-Phase6-検証を閉じる.md`)。
完了3区別を完成させ、旗「検証可能な正本」を現物で立てる核 (M2)。

## 背景

スライス1 で requirements 正本層を作った。各受け入れ基準は検証 link (verify) を任意で持てる。
verify.run は今、要件完了を一律 needs_human で返す (`crates/core/src/verify.rs`)。
スライス2 で、検証 link のある要件を機械判定 met へ移し、link が欠ける要件は従来どおり needs_human で残す (ハイブリッド)。

## 採用設計

### 検証 link は config の検査名を指す

受け入れ基準の verify は config.toml の `[[verify.checks]]` の name を指す。owox が実際に実行できる単位は検査コマンドなので、link をそこへ解決する。

- 粒度はプロジェクトが検査の切り方で決める (1テストだけ走る検査を定義すれば基準単位、全テスト一括なら粗い)。owox は test runner の粒度を強制しない (tool 非依存)
- スライス1 で verify を「テスト名・検査名」としたのを「検査名」に締める。互換 (既存の自由文字列のうち検査名に一致するものが機械判定対象)

### 検証 link は書き込み時に config の検査名へ照合する (dangling を作らせない)

`requirement.link_verification` と `requirement.create` の inline 基準は、verify を config.toml の検査名へ照合し、未知なら利用可能な検査名を添えて弾く。task.create が実在しない依存先を弾くのと同型 (`crates/core/src/task.rs` の unknown_dep_target)。

- これで dangling link (実行できない link) は書き込み時に発生不能になる。最大の危険を設計で消す
- spec-first を阻害しない: 「テストはこれから」は unlinked (link 無し) で表す。dangling link は要らない。流れは「要件+基準を書く (unlinked) → next が trace 欠落で先回り → 実装+検査を config へ足す → link (照合通過)」。unlinked がそのまま spec-first の中間状態
- hard-reject にする理由: link は機械判定を成立させる trace。存在しない検査への link は「link 済みに見えて実行できない」偽の trace になる。検査を先に config へ足させ、エラー文でその流れを教える方が清潔
- 照合に使う検査名は mcp が config から読んで core へ渡す (core は config 読込へ結合しない。today / work_dir と同じく外から与える)
- 後から検査を消した端ケースの runtime dangling だけが残り、verify.run が needs_human の trace 欠落として穏当に扱う

### 要件完了は per-requirement の判定

verify.run は accepted な要件ごとに met を判定する (draft は未確定・superseded は置換済みのため対象外)。

各要件の判定 (受け入れ基準 → 検査結果):

- 受け入れ基準が無い → needs_human (機械判定する対象が無い)
- 検証 link が欠ける基準がある (unlinked) → needs_human (trace 未完成。どの基準が欠けるか reason に出す)
- 検証 link が config に無い検査を指す (runtime dangling。link 後に検査を消した端ケース) → needs_human (実行できない。どの基準のどの検査か reason に出す)。書き込み時照合で通常は発生しない
- 全基準が実在検査へ link され、その検査が全通過 → met (機械判定)
- 全基準が link 済みだが link 先の検査が落ちた → failed

検査は config の全検査を一度実行し、結果を name→passed の表にして判定へ渡す。要件が link する検査はその部分集合なので結果が必ずある。

### 集約と封筒

完了3区別の requirement 値を、accepted 要件の判定から集約する:

- accepted 要件が無い (または基準を持たない) → needs_human (第1段階の挙動を温存)
- いずれか failed → failed
- いずれか needs_human → needs_human
- すべて met → met

封筒 status: 検査失敗 → failed (要件 failed は必ず検査失敗を含むのでここに吸収される)、それ以外は needs_human (作業完了は機械判定できないため。要件が met でも work は人間確認)。
旗の達成は data の `completion.requirement` が needs_human から met へ動くことで成立する。

requirement は「満たされた (met)」、verification は「検査が通った (passed)」と語を分ける (完了3区別の意味の違いを残す)。

### verify.run の返り (data)

- results: 検査ごとの {check, passed, detail} (既存)
- completion: {work, requirement, verification}。requirement が met / needs_human / failed を取る (既存は一律 needs_human)
- requirements: accepted 要件ごとの {id, title, met, unlinked, reason} (新規。どの要件がなぜ未判定かを開示)

### trace 欠落は next で先回り・verify.run で確定

trace 欠落 (accepted だが検証 link が欠ける) の機械検出を2か所で出す。役割を分ける:

- next: accepted かつ「基準が無い or unlinked>0」の要件を「受け入れ基準の trace が要る」として出す。検査を実行せず静的に分かるので軽い (next は頻繁に呼ばれる。検査は回さない)
- verify.run: 検査を実行し met / failed まで確定する (重い・正確)

これで plan 未決「owox://next に accepted かつ未充足の要件を出すか」に答える: 静的に分かる trace 欠落だけ next に出し、充足の確定は verify.run に置く。

### commit ゲートは変えない

commit ゲートは従来どおり検査失敗で deny する。要件 met を commit の歯にはしない。

- 要件 met は report (verify.run) と先回り (next) で効かせ、commit を要件未充足で止めない。初期 phase で trace 未整備のプロジェクトを罠にかけない
- trace の厳格度を state (phase) で上げる (maintenance で accepted 要件の trace 欠落を block 等) のは将来。スライス2 は報告に留める

## 変更境界

触るもの:

- crates/core/src/requirement.rs: 要件1件の met 判定関数 (受け入れ基準 → 検査結果表) を足す。link_verification と create_requirement に既知検査名の照合を足す (未知 link を弾く)。要件の構造を知る側に判定と照合を置く
- crates/core/src/verify.rs: run_verify が requirements を受け、検査結果から per-requirement と集約を組み立てる。signature に requirements を追加
- crates/mcp/src/serve.rs: verify_run が list_requirements で要件を読み run_verify へ渡す。next が要件を読み render_next へ渡す。link_verification / requirement.create が config の検査名を読んで照合へ渡す
- crates/core/src/lib.rs: 必要なエクスポート
- docs (本設計・検証)

触らないもの:

- commit/stop ゲート (hook.rs)。要件 met を歯にしない
- requirement の CRUD tool (スライス1 で確定)
- quality.toml (スライス3)・多視点レビュー (スライス4)
- task.close の検査実行 (run_checks 利用。run_verify とは別経路)

守る振る舞い:

- run_verify を呼ぶのは verify.run tool のみ (signature 変更の波及は serve.rs に閉じる)
- 検証 link が無い要件の verify.run は従来どおり needs_human (第1段階互換)
- core は時計を読まない・封筒は core の型

## 危険 (いずれも設計で解消または受容)

- dangling link: 書き込み時照合で発生不能にした (上記)。最大の危険を設計で消した。残るのは検査を後で消した端ケースのみで needs_human で穏当に扱う
- 意味の締め (link=検査名) を get_info とツール説明に明記し、未知 link のエラー文へ利用可能な検査名を添えて教える
- next が要件を毎回読む。tasks/decisions と同クラスの読込で、状態/キャッシュを足すと無状態の思想を壊すため解さず受容する
- per-requirement の reason 文面が長くなりすぎないか。欠落基準の番号列挙に留め簡潔にする

## 検証方針

- 単体: met 判定 (基準なし / unlinked / dangling / 全通過 met / link 先 failed)、集約 (accepted のみ・failed/needs_human/met 優先順)、run_verify の data 形、accepted 以外は判定外
- 結合: serve 経由で verify.run が completion.requirement を met で返す経路、next が trace 欠落要件を出す経路
- 手動 (後の一括対話検証へ): config の検査名へ link した要件で verify.run が met を返すか、link が欠ける要件が needs_human で残るか、next が trace 欠落を先回りで出すか
- 未確認として残す: state 適応の trace 厳格度 (将来)、quality.toml (スライス3)

## 捨てた案

- 検証 link を自由なテスト識別子にし、別途「どの検査が実行するか」の対応表を持つ → 新サブシステム。検査名直結で足りる
- 未知の検査への link を soft-warn で許す → 「link 済みに見えて実行できない」偽の trace が残る。hard-reject + unlinked 中間状態で spec-first を満たす
- 要件 met を commit ゲートの歯にする → 初期 phase で trace 未整備を罠にする。報告に留める
- requirement 完了を単一のグローバル値のまま返す → 複数要件の met を表せない。per-requirement へ広げる
- trace 欠落を verify.run だけで出す → next で先回りできる軽い信号を捨てることになる

## 見直し条件

- 検証 link の粒度 (検査名直結) が、基準単位の細かい trace に足りないと分かった時
- 要件 met を commit/maintenance で強制したくなった時 (state 適応の trace 厳格度)
- per-requirement の判定が重くなった時 (要件件数の増大)
