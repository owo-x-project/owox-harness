# 要件・設計の grill-me 拡張と対話的質問ツール提示

## 状態

採用・実装済み。`docs/decisions/20260611-方向付け.md` の grill-me 思想を、kickoff だけでなく
要件・設計の入口へ広げる具体化。

## 背景

grill-me (AI が質問攻めで曖昧な要望を具体化する) は kickoff だけに実装され、要件定義 (req) ・設計には
無かった。req は「what と why に人間承認を取る」とだけ言い、引き出しや鋭い追撃が無い。設計には入口
skill すら無かった。

加えて、kickoff / req の質問は `next` がテキスト出力するだけで、各 CLI が持つ対話的質問ツール
(Claude Code の AskUserQuestion 等) へ寄せる指示が無く、徹底質問が AI の自由発話頼みだった。
判断型でも推奨案を先頭に置く指示が無かった。

## 採用: grill-me を req / design へ広げ、共通スニペット化

質問で具体化する入口 (kickoff / req / design) は共通の grill-me 指針 `GRILL_GUIDANCE` を持つ。
重複を避け一貫させる (一度に1問・コードで分かる事は自己解決・質問種別で形を変える)。

- 質問種別: 引き出し (推奨なし) / 確認 (owox 決定を提示し否認・修正だけ) / 判断 (推奨+理由+選択肢)。
  `docs/decisions/20260627-判断2軸と対話kickoff.md` の3種に合わせる
- req: grill-me を足し「意図が具体化するまで要件を書かない」を明示
- design: 入口 skill を新設。設計はトレードオフ判断 → decision.record。可逆な技術判断は owox が決めてよく、
  製品意図・安全の分岐だけ人間へ

## 採用: 質問ツール名を target 生成時に写像する

正本 (commands.rs の grill 指針) は CLI 非依存のプレースホルダ `{{QUESTION_TOOL}}` を持ち、CLI 名を
入れない (既存規約・テストで強制)。target 生成時に各 CLI の質問ツール名へ写像する
(`crate::target::apply_question_tool`)。

- Claude Code: `AskUserQuestion` を持つ → 「using the AskUserQuestion tool」へ
- Codex: 構造化質問ツール無し → 「by asking in plain text」へ倒す
- 判断型は推奨案を先頭の選択肢に

写像は Target trait の `question_tool()` (Claude=Some / 既定=None) で表し、CLI 追加時は1メソッド足すだけ。
生成 skill には具体ツール名が出るので、AI が自分の質問ツールを取り違えない。

MCP の kickoff/next 出力にも提示指示を1行置くが、こちらは抽象のまま。owox serve は共有プロセスで
接続中の CLI を識別できないため、runtime では具体名へ写像できない (生成 skill が具体名を担い、
MCP 出力は補強)。

## 捨てた案

- design を req に統合 → 要件 (製品意図軸) と設計 (行動軸寄り) の局面が混ざる。役割分離で別入口にする
- 正本本文へ直接ツール名を書く → 生成文ツール非依存の規約に反する。プレースホルダ写像で正本は
  CLI 非依存を保ちつつ生成物に具体名を出す
- 抽象指示のみで写像しない → AI が自分の質問ツールへ確実に向かわず平文に流れる恐れ。生成 skill では
  具体ツール名まで写像する
- 全質問に推奨必須 → 引き出しが押し付けになる (`docs/decisions/20260627-判断2軸と対話kickoff.md` で撤回済み)

## 見直し条件

- 抽象指示では AI が質問ツールを使わず平文に流れると分かった時 (target 個別の写像が要る)
- req / design / kickoff の grill 指針に差が要り、共通スニペットが窮屈になった時
- design 入口の利用が薄く req に畳んだ方が良いと分かった時
