# Vision 語彙とセッション目標

## 状態

採用済み。`docs/decisions/20260612-正本フォーマット.md` の語彙 (north_star → Goal) と `docs/decisions/20260611-方向付け.md` の North Star を Vision に統一する。

## 背景

brand の Goal を毎セッション AGENTS.md (常時読むルート指示ファイル) で挿入している。問題が2つ。

- 誤読: 「Goal」は汎用語。常時読む文書の先頭に出るため、AI も人間も「今セッションの目標」と読み違える。実体はプロジェクト永続の目的 (`docs/decisions/20260611-target-harness内容.md` の「目的 (North Star)」)
- 差分・コンフリクト: 誤読されると人がセッション毎に Goal を書き換える。brand は静的・安定のはずが毎回 git 差分を生み、多人数でコンフリクトする

加えて、現行設計文書で永続目的の呼び名が North Star / 北極星 / Goal とブレている。

根本原因は、1スロット (Goal) に2つの寿命 (永続目的・一時のセッション目標) が混ざり、永続側の経路 (常時読む AGENTS.md) 1本で挿していること。改名だけでは直らない。寿命でスロットと経路を分ける。

## 判断基準

- 一読で「永続目的・今回の作業ではない」と分かる名前
- ビジネス用語を避け平易 (North Star は専門用語気味)
- 永続目的と一時のセッション目標を別スロット・別経路に分離する
- 一時物を git 管理の正本に置かない (差分・コンフリクトを断つ)
- 旗「最小コンテキスト」と整合 (新サブシステムを足さない)

## 採用案

### 寿命で3概念を分離

- Vision (永続目的): どこへ向かうか。不変。brand に持ち、AGENTS.md で常時挿入。旗そのもので小さく基盤のため常時でよい
- セッション目標 (一時): owox-harness は持たない。人間のセッション冒頭プロンプトがそのまま目標。保存しない。これで差分・コンフリクトの根が消える
- 次の一手 (やること提案): いま何をすべきか。動的。`docs/decisions/20260611-方向付け.md` の owox://next が出す「次に決めること・次にやるタスク」。Vision ともセッション目標とも別機能

人間は owox://next の候補から選び、それをプロンプトにする。Vision (向き) → 次の一手 (候補提示) → 人間が選ぶ (プロンプト) → 実行、で流れが閉じる。だからセッション目標を保存しなくても人間は迷わない。

### 語彙統一

永続目的の呼び名を Vision に統一する。`north_star` / `Goal` / `North Star` / `北極星` を Vision に置換する。

- 対象: 現行設計を述べる文書 (requirements / decisions / roadmap) と実装 (brand の見出し・型・生成)
- 対象外: validation 文書。実機観測の当時記録のため据え置く (`北極星` 表記を残す)

## 捨てた案

- 改名のみ (寿命分離なし)
- Goal のまま注記を1行添える
- North Star に統一する
- セッション目標を canon (brand 等) に持つ
- セッション目標を git-ignore スクラッチに永続化する
- セッション目標を MCP state で動的管理する

## 捨てた理由

- 改名のみでは、セッション目標の置き場が未定で差分・コンフリクトの根が残る
- 注記は常時トークンを増やし、生成物に編集禁止注記を置かない方針と矛盾する
- North Star はビジネス用語気味で平易さに欠ける
- セッション目標を canon に持つと、一時物が git に乗り誤読・差分・コンフリクトを再発する
- git-ignore スクラッチ・MCP state は仕組みが増える。owox://next が候補提示するため、第1段階で保存は不要。将来必要なら動的層 (MCP) で足す

## 見直し条件

- owox://next が機能せず、人間がセッション目標を見失う時 (保存を再検討)
- Vision の常時挿入が最小コンテキストを圧迫する時 (Phase 4 の owox:// へ持ち上げ)
- Vision の語が別概念と衝突する時
