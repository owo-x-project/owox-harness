# Phase8 調査知識層

## 状態

採用 (実装済・検証済 `docs/validation/20260617-Phase8-対話検証と粗の是正.md`)。Phase8 (`docs/requirements/20260616-Phase8-経験を厚くする.md`) の設計。
記録層 (`crates/core/src/record.rs`・`task.rs`) と腐敗検知 (`decay.rs`) を踏襲し、新サブシステムを作らない。

## 背景

調査 (リサーチ) の成果に置き場が無く、来歴に紛れるか消える。
AI が再調査し人間が再説明する無駄が出る。調査記憶を検証つきで蓄積する層を足す。

設計判断はマネージャ問いで人間が確定済み:

- 新層を作る (practices 拡張でない)。practices は自由文の指針・常時注入・指針鮮度。調査記憶は構造化・オンデマンド・経過日数鮮度で性質が違う。混ぜると注入が膨らみ、指針か参照かが曖昧になる
- 更新は supersede 専用 (in-place 更新しない)。調査時点で何を知っていたかを残す

## 採用設計

### 新モジュール knowledge.rs

`crates/core/src/knowledge.rs`。`.owox/knowledge/<id>.md` に1調査1ファイル。
ID=日付+slug。record.rs の slugify・allocate_id を再利用する (crate 内 pub(crate) を knowledge からも使う)。

### 型

- KnowledgeStatus: Current / Superseded。supersede 専用なので Draft は持たない
- Knowledge: id / title / researched_on (調査日 YYYYMMDD) / sources (出典の並び・URL か参照) / summary (調査結果の散文) / tags (任意・lookup 補助) / status / supersedes (置き換えた旧 ID の並び)

### Markdown 符号化 (新文法を足さない)

- 1行目 `# <title>`
- `## Researched on`: 調査日 (YYYYMMDD)
- `## Sources`: 出典の箇条書き
- `## Summary`: 調査結果 (散文)
- `## Tags`: 任意の箇条書き
- `## Status`: current / superseded
- `## Supersedes`: 旧 ID の箇条書き

読み方は record.rs と同型 (`Doc::parse`・`section.list()`・`split_pair`)。

### tool (4個)

- knowledge.add { title, summary, sources[], researched_on?, tags?[], supersedes?[] } → { id }
  - researched_on 省略時は today (mcp が供給)
  - supersedes 指定時は旧エントリを superseded へ書き換え、新規を current で書く。これが supersede の実体 (専用 tool を増やさない)
- knowledge.list { status?, stale? } → { items:[{id,title,researched_on,tags,stale}] }
- knowledge.get { id } → 調査全文 (canon 直読み禁止の読み口)
- knowledge.lookup { query } → keyword が title / summary / tags に一致するエントリの要約

更新は supersede のみ。in-place 更新 tool は作らない (来歴を消すため)。

### 鮮度 (腐敗検知へ連動)

decay.rs に run_knowledge_decay を足す (run_practice_decay と同流儀・run_decay 署名は不変)。

- 経過日数 = today − researched_on。閾値超えで kind="stale-knowledge" を報告
- 閾値は quality.toml の `[decay]` に knowledge_stale_days を追加 (DecayConfig・DecayRaw へ。既定は review_decision_days と揃え 90)
- is_structural()=false。commit を止めない (鮮度は助言)
- superseded は対象外 (現役の調査だけ鮮度を見る)

### 報告と読みの分離

- 報告: next の腐敗節と verify.run の data.decay に stale-knowledge を含める (practices 鮮度と同じ扱い)
- 読み: オンデマンドのみ。床コンテキストへ注入しない (最小コンテキスト)。中身は lookup / get で引く

### 秘密走査

knowledge.add で summary + sources を secret::scan する。検出時は failed (書かない)。
外部内容を保持するため、experience の本文走査と同じ守りを入口で効かせる。

### 来歴連動の線引き

調査記憶の追加・supersede は AI 直接 (来歴を残さない)。証拠・参照であり、decision.record と同じく低リスク。
過剰記録へ戻らない (`docs/handoff/20260613-Phase4対話検証で見つけた粗の改善.md` の方針)。

### ドメイン分離 (経験 export 除外)

調査記憶はこのプロジェクトのドメイン調査で、汎用経験でない。
experience.export から型で除外する (memory / tests / 固定層と同じ除外集合に knowledge を加える)。
要件 対象外「ドメイン固有情報を汎用経験へ混ぜない」を守る。

## 変更境界

触る:

- crates/core/src/knowledge.rs (新規)、lib.rs (型・関数 pub use)
- crates/core/src/decay.rs (run_knowledge_decay)
- crates/core/src/quality.rs (DecayConfig・DecayRaw に knowledge_stale_days)
- crates/core/src/experience.rs (export 除外集合へ knowledge を加える)
- crates/mcp/src/serve.rs (knowledge.add / list / get / lookup と Params、next / verify.run の decay 集約へ run_knowledge_decay、get_info に tool 追記)
- docs (本設計・validation)、正本側の knowledge サンプル (target 検証用)

触らない:

- 既存記録層・封筒・canon の振る舞い
- run_decay / run_verify の署名 (knowledge decay は別関数)

守る振る舞い:

- canon 直読み禁止 (読みは knowledge.get / lookup 経由)
- core は時計を読まない (today は mcp が供給)

## 危険

- 出典 URL に秘密 (トークン付き URL 等) が混じる → add の secret::scan で止める
- lookup が素朴な部分一致だと雑音 → 最小は部分一致、必要なら tags 優先。実機で調整
- 調査記憶が増え1ファイル運用が重くなる → 記録層と同条件 (同じ時に見直す)
- next が knowledge ファイル読取で遅くなる → 件数は少ない想定。重ければ next から外し verify のみへ (code decay と同じ判断)

## 検証方針

- 単体: add → list → get の round-trip、supersede で旧 superseded・新 current、鮮度閾値 (古い current を stale-knowledge・superseded は対象外)、secret 検出で failed、lookup の keyword 一致、export 除外
- 結合: serve 経由の tool 呼び出し、next / verify.run に stale-knowledge が出る
- 手動 (実機 Codex): knowledge.add → lookup → get を直読みせず回す、古い調査が next で警告される

## 捨てた案

- practices 拡張で持つ → 性質が違い注入が膨らむ・指針と参照が曖昧 (人間判断で却下)
- in-place 更新 tool → 調査時点の記憶が消え鮮度の根拠 (調査日) も曖昧 (人間判断で却下)
- 床コンテキストへ常時注入 → 最小コンテキスト違反。オンデマンド lookup に寄せる
- 来歴を必ず残す → 過剰記録。証拠の追加は軽量

## 見直し条件

- 調査記憶が長文化し節符号化が窮屈になった時
- lookup の部分一致が雑音を生み、索引が要るようになった時
- 鮮度既定 (knowledge_stale_days) が実利用で合わない時
