# Phase6 対話検証の是正 (継続観察3件)

## 状態

採用。Phase6 を実機 Codex の対話で検証して残った継続観察3件を是正する設計。
`docs/validation/20260614-Phase6-対話検証.md` の続き。粗2件 (quality 走査の未追跡漏れ・changed_files 偽カウント) は既に是正済みで、本書は対話で気になった3件 (来歴 ID 長・Stop の毎ターン催促・link 未知名の事前誤推測) を扱う。

## 背景 (対話で見つけた継続観察)

- 来歴 ID が長い。requirement.update が決定タイトルへ対象要件 ID をそのまま埋め、要件 ID が既に日付つき (`20260614-...`) のため slug に日付が二重に入り、さらにタイトル全文が slug 化され肥大した (例 `20260614-Update-requirement-20260614-有効な認証情報でログインできる-the-statement`)
- Stop hook が毎ターン催促する。作業ツリーが dirty な限り、同じ変更で何も足していないターンでも STOP_CHECKLIST を出し続けた。verify.run 済みかも変更が増えたかも見ていなかった (Phase5 で「後回し」と記録した残ノイズ)
- AI が link_verification を呼ぶ前に「未知の検査名でも保存される」と誤推測した。実際は config の検査名へ照合し未知は弾くが、ツール説明が「a test or check name」だけでこの照合を述べていなかった (要件完了の機械判定.md:109 で「ツール説明に明記」と方針済みだが未実装)

## 採用設計

### 来歴 ID の短縮 (日付重複除去 + slug 上限)

- 別 ID の slug へ日付つき ID を埋める時は日付プレフィックス `YYYYMMDD-` を剥がす (record.rs の strip_date_prefix)。日付が二重に入らない
- slugify に slug の最大長 (40 文字) を設け、語境界 (`-`) で切る。ID/ファイル名だけ短く保ち、タイトル本文 (frontmatter) は full のまま。一意性は allocate_id の連番が担うので切り詰めても衝突しない
- 全 ID 種別 (decision/requirement/task) で一律。ID は識別子なので意味の完全性よりも短さを優先する

### Stop は変更が変わった時だけ促す

- 作業ツリーの署名 (HEAD との diff 内容 + 未追跡一覧のハッシュ、.owox 除外) を取り、前回促した時の署名と比べる。dirty かつ署名が変わった (初回含む) 時だけ STOP_CHECKLIST を出し、同じ変更が続くだけのターンは黙る
- 署名は用語重複排除と同じ session キャッシュ (`.owox/.cache/sessions/<id>-stop.json`) に保つ。hook が session_id を持つので hook 内で完結し、verify.run tool との連携は要らない
- core の stop_decision は dirty_changed を真偽値で受けるだけ (git 署名は mcp が判定して渡す。既存の責務分担に沿う)
- 署名を取れない時 (git 無し等) は変更扱いに倒し、見逃さない側へ寄せる

### link の照合をツール説明に明記

- link_verification の description に「検査名は config.toml の `[[verify.checks]]` に照合され、未知名は弾かれ利用可能な名前が返る」を明記
- requirement.create / link_verification の verify パラメータ doc も「config の検査名・未知は弾く」に直す
- 検査名を呼ぶ前に一覧する専用 tool や resource は増やさない (要件完了の機械判定.md の「エラー文で教える」方針を保ち、説明明記で誤推測を断つ)

## 不採用

- slug を短縮せず日付重複だけ除去 — ID 肥大が残る。長文タイトルで再発する
- 未追跡ファイルの内容まで Stop 署名に含める — 重く、作成時に一度促せば足りる。未追跡の中身編集を促し損ねる軽微な穴は許容
- verify.run 実行を session へ記録して Stop が参照 — MCP tool に session_id が届く保証がなく、hook 内完結の署名比較で足りる
- 検査名一覧 tool の新設 — 面が増える。説明明記とエラー文で足りる

## 変更境界

触る:
- crates/core/src/record.rs: slugify に上限・strip_date_prefix 追加
- crates/core/src/requirement.rs: 決定タイトルで対象 ID を strip_date_prefix
- crates/core/src/hook.rs: stop_decision に dirty_changed を追加
- crates/mcp/src/hook.rs: tree_signature・Stop 署名の session 保存/読み・stop() 配線
- crates/mcp/src/serve.rs: link_verification description と verify パラメータ doc
- 各テスト

触らない:
- 既存の来歴ファイル (新規 ID のみ短縮)・quality/review 経路・verify 判定・commit ゲート・直読み deny

守る振る舞い:
- ID の一意性 (連番)・git 無し時の安全側退避・Stop の同一ターン1回制限・用語キャッシュ

## 危険

- slug 切り詰めで ID から内容を読み取りにくくなる。タイトル本文は full なので参照性は frontmatter で保つ
- DefaultHasher は std 実装に依存。変われば署名が一致せず Stop が元の毎ターン催促に戻るだけ (機能後退せず安全側)
- 未追跡ファイルの中身だけ変えたターンは Stop が黙る (一覧不変)。作成時に一度促し済みなので許容

## 検証方針

- 単体: slugify が上限で語境界切り・strip_date_prefix の剥がし、stop_decision が dirty_changed=false で黙る (cargo 149 passed)
- 結合: target で Stop が同一変更の2回目以降黙り変更が変わると再度促す、requirement.update の decision ID が短い、link_verification description が照合を述べる
- 手動 (実機 Codex): 対話で ID が邪魔にならない・Stop が静かになる・AI が link 前に検査名照合を理解する

## 見直し条件

- slug 40 文字が短すぎ/長すぎと分かった時
- Stop 署名の取り方 (未追跡内容・rename 等) で促し漏れが目立った時
- 検査名の事前開示が必要になった時 (説明明記で足りなければ context への列挙を検討)
