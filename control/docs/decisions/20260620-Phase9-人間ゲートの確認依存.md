# Phase9 人間ゲートの確認依存と trusted の抜け穴

## 問題

guarded ゲート (rules/brand/glossary 変更・素の open 決定) の「人間が承認した」という同意は、
gate.approve 等に付けた destructive 注釈でクライアント (Codex) が確認プロンプトを出すこと **だけ** に
依存している。owox はツールの内側から「人間が承認しろと言ったのか・AI が勝手に決めたのか」を区別
できず、唯一の正直な同意信号がその確認プロンプト。

Phase7 (`20260616-Phase7-canon変更の自動適用と承認の機械強制.md`) は「destructive 注釈は
approval_policy=never でも auto でも必ず人間確認・組織の requirements.toml 以外で上書き不可」と
記録したが、Phase9 実機検証で `trust_level=trusted` のプロジェクトでは確認が省かれると判明。
trusted 下では AI が gate.approve を自分で呼んで rules 変更まで自己承認でき、人間ゲート保証が
静かに蒸発する。Phase7 前提に trusted という抜け穴があった。

## 採用判断

owox が生成する target 設定で、ゲート系ツールだけを「全面信頼下でも必ず確認を出す」へ固定する。
人間ゲート保証は「クライアントが確認を出すこと」が前提だと明文化し、生成設定でその前提を担保する。

- 対象ツール: gate.approve・gate.auto_enable・canon を書き戻すツール (確認を出さないのが目的の
  gate.auto_approve は対象外)
- 担保の置き場所: owox は既にクライアント設定を生成・マージするので、確認固定もその生成時に置く。
  AI が触れない設定面で前提を固定するのが清潔
- 同意信号は AI の外からしか来ない。ツール内に偽の人間検査を足しても保証にならない

## 技術的裏付け (Codex 0.141.0)

通常の `[mcp_servers.X]` (RawMcpServerConfig) が per-server・per-tool の承認設定を持つ。

- server 既定: `default_tools_approval_mode`・`enabled_tools`・`disabled_tools`
- tool 個別: `tools.<name>` 配下の承認モード
- 承認モード列挙: UnlessTrusted・OnFailure・OnRequest・Never

owox が書くのはプロジェクト直下 `.codex/config.toml` の owox ブロック (target/codex.rs の
render_mcp_config)。ここへゲートツールの承認固定を足せる。

## 未確定 (実機 Codex 対話で要確認)

- trusted でも必ず確認を出すモード値。候補は OnRequest (UnlessTrusted は trusted に譲るので不適・
  Never は確認なし)。列挙名だけでは断定できず実機で発火を見る
- 正確なキー構造。`[mcp_servers.owox.tools."gate.approve"]` の approval_mode か、tools がマップか
- プロジェクト `.codex/config.toml` の per-tool 設定が `~/.codex/config.toml` のグローバル
  trust_level に勝つか (設定レイヤの優先順)
- ツール名のドット (gate.approve) を TOML キーで引用する必要があるか

## 捨てた案

- owox 独自の腕外承認経路 (人間が別途 `owox approve <id>` を走らせ owox が検査): headless/trusted でも
  堅牢だが、owox が意図的に消した「人間がコマンドを走らせる」摩擦を戻し承認面が 2 つに増える
- 明記のみ (コード変更なし): 最安だが trusted で保証が黙って蒸発し何も止めない

## 実装スコープ (次スライス)

- target/codex.rs の render_mcp_config にゲートツールの承認固定ブロックを追加
- マルチ CLI: Codex 以外の生成先 (Claude 等) も各 CLI の同等設定 (常時確認・ask) へ写す
- 生成テスト: 生成 config にゲートツールの確認固定が出る回帰
- 実機 Codex 対話で、trusted プロジェクトでもゲート確認が実際に出ることを確認 (未確定のモード値を確定)

## 見直し条件

- 候補モード値が実機で確認を出さない時 (別の設定面・腕外経路へ切替)
- Codex の承認設定スキーマが変わった時
- 設定レイヤの優先順でプロジェクト設定がグローバル trust に負ける時
