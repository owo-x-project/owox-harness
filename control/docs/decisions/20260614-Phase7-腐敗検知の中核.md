# Phase7 スライス1: 腐敗検知の中核の設計

## 状態

採用・実装済み・機械確証済み。Phase7 スライス1 (`docs/requirements/20260614-Phase7-腐らず成長する.md`)。
タスクと来歴の腐敗を機械検出し `next` と verify.run へ報告する (`docs/decisions/20260611-タスク管理.md`・`20260611-設計原則.md`)。

## 背景

タスク管理と設計原則の決定で、放置 / 孤立 / 重複 / ゾンビ / done未検証 と古い記憶を適応度関数で機械検出すると決めた。Phase6 で quality の適応度関数 (行数予算・禁止パターン) を「機械検出して報告、効かせ方は commit で phase 適応」の型で実装済 (`docs/decisions/20260614-Phase6-quality適応度関数.md`)。腐敗検知はこの型に載せ、新サブシステムを作らない。

人間判断 (plan の論点) を確認し確定した:

- 閾値・規則の置き場 = quality.toml の `[decay]` 節
- 放置の基準 = ID 日付と note 日付の最新を「最後の活動」とし today との日数差で測る
- スライス1スコープ = タスク5種 + 来歴鮮度 (open 放置・見直し合図)

## 採用設計

### decay は core の判定純関数・新モジュール

core `decay.rs` (新規) に `run_decay(tasks, decisions, config, today) -> Vec<DecayFinding>`。
タスクと来歴を読み、quality.toml の `[decay]` 閾値と today を受けて腐敗を判定する純関数。core は git/時計を持たず today を引数で受ける既存方針に従う (`docs/decisions/20260613-Phase4-tool記録層.md`)。
DecayFinding = 種別 + subject (タスク/来歴 ID) + detail。`summary()` で1行サマリ (commit ゲート・封筒に使う。QualityViolation と同型)。

### 閾値は quality.toml の [decay] 節 (数値は phase で変えない)

DecayConfig は quality.rs に置き Quality 型へ足す (quality.toml から読む設定なので budgets/boundaries と同居)。判定ロジックは decay.rs。依存方向は decay → quality (config) / task / record で循環なし。

- 閾値: stale_task_days / open_decision_days / review_decision_days
- `[decay]` 無しでも owox 標準の既定値で動く (検知は既定で有効、数値だけ人間が上書き)。budgets/boundaries が opt-in なのと違い、decay は既定で検知する (タスク管理の「腐敗防止は中核必須・後付けにしない」)
- phase で閾値の数値は変えない。Phase6 quality と同じく、検出は常に同じ規則で走り、phase は commit ゲートの効かせ方 (block か警告) だけを変える。「初期=緩い」は数値でなく enforcement で実現する (per-phase の数値表を持たず単純に保つ)

### 古さは ID 日付と note 日付の最新で測る

- タスクの最後の活動 = ID 日付 (`YYYYMMDD-slug` の先頭8桁) と各 note の日付前置の最新。これと today の差が日数
- 来歴の古さ = ID 日付と today の差
- 日数計算は core の純関数 (Howard Hinnant の civil→days で両 `YYYYMMDD` を通日へ直し差を取る)。mcp の today_utc が使う days→civil の逆。git mtime は使わない (非決定論で core 方針に反する)

### タスクの検出規則 (5種)

- 放置 (stale): status が todo / doing / blocked で、最後の活動から stale_task_days 超え。done / dropped は対象外
- 孤立 (orphan): links と deps が空で、かつ最後の活動から stale_task_days 超え。新規 todo を即座に孤立扱いせず誤検出を避ける (age ゲートで「古くて繋がらない」を強い信号にする)
- 重複 (duplicate): ID から日付前置を剥がした slug が一致する別タスクがある (`strip_date_prefix` を再利用)。dropped どうし・dropped との一致は除く。曖昧一致は持たない
- ゾンビ (zombie): blocks 依存の対象が dropped で永久に done になれない、または blocks 依存に循環があり ready になれない
- done未検証: status が done だが verification link が無い (Phase6 の close は必ず付けるが過去・import タスクを拾う)

### 来歴の検出規則 (2種)

- 放置 open: status が open で ID 日付から open_decision_days 超え。未決のまま放置された判断点を表面化する
- 見直し合図: status が adopted で ID 日付から review_decision_days 超え。古い前提を警告する。削除はしない (時間軸は見直しを促す合図。`docs/decisions/20260611-設計原則.md`)

### 報告経路: next に腐敗節・verify.run は報告・commit は構造的腐敗のみ phase 適応

- next: 腐敗警告の節を足す (ready タスク・open 来歴と同じ常時報告)。context を汚さぬよう件数を要約し全件は出さない。next が canon の quality.decay と today を読んで run_decay を呼ぶ
- verify.run: data に decay 違反を載せ報告のみ。完了3区別は変えない (quality と同じ・判定は commit に集約)。次の手に1行添える
- commit ゲート: phase 適応する歯は構造的腐敗 (done未検証・ゾンビ) のみに絞る。保守は block・初期/安定は警告。放置/孤立/重複/来歴鮮度は age・cleanliness 信号で、commit を止めると狼少年化するため next と verify.run の助言に留める (block しない)。open 来歴の放置は既存の open_gates ゲートが保守で既に block しており二重化しない

### today の供給を clock へ切り出す

今 today_utc と civil_from_days は serve.rs にある。commit ゲート (hook.rs) でも today が要るため `crates/mcp/src/clock.rs` へ切り出し serve/hook で共用する (重複を作らない)。

### 責務分担

- core: 判定純関数 run_decay と日数計算。決定論
- mcp: today を供給し、tasks / decisions を読み、結果を next / verify.run の封筒と commit ゲートへ載せる

## 変更境界

触るもの:

- crates/core/src/decay.rs (新規): run_decay・DecayFinding・日数計算・検出規則
- crates/core/src/quality.rs: DecayConfig 型と `[decay]` の TOML 読込・既定値
- crates/core/src/verify.rs: run_verify が decay 違反を data へ報告 (引数追加)
- crates/core/src/hook.rs: commit_gate に構造的 decay の phase 適応を足す (引数追加)
- crates/core/src/lib.rs: decay の公開
- crates/mcp/src/clock.rs (新規): today_utc・civil_from_days を serve から移設
- crates/mcp/src/serve.rs: next・verify.run が tasks/decisions/today を読み run_decay を渡す配線
- crates/mcp/src/hook.rs: commit_gate へ decay サマリを渡す配線
- tests/fixtures・target サンドボックス・docs

触らないもの:

- タスク層の CRUD と done 検証必須・drop 理由必須・ready 判定 (Phase4 で確定)
- quality の budgets/boundaries の判定 (Phase6 で確定)
- 完了3区別の verification の意味 (config 検査のまま。decay を混ぜない)
- 来歴・タスクの記録形式 (1件1ファイル・ID=日付+slug)

守る振る舞い:

- core は git/時計を持たず決定論 (today は外から)
- `[decay]` 無しでも既定値で検知する (budgets/boundaries の opt-in と違う)
- コアゲート (検査失敗) は phase 不問で常時 deny
- next・verify.run は phase 不問で常時報告

## 危険

- 誤検出・狼少年化。age 系は commit を止めず助言に留め、構造的腐敗 (done未検証・ゾンビ) だけ保守で block する。閾値は人間が quality.toml で調整できる
- 新規 todo の孤立誤検出。orphan に age ゲートを掛け「古くて繋がらない」だけ拾う
- 重複の slug 一致が正当な分割タスクを重複扱いしうる。報告のみで block しない
- 来歴溜まりが context を汚す。next は要約し全件を出さない
- 既定の閾値の数値 (放置・見直しの日数) が phase に合わない。enforcement の phase 適応で吸収し既定は緩めに置く。具体値は未決として残し対話検証で詰める
- ゾンビの循環検出のコスト。blocks グラフは小さく DFS で足りる

## 検証方針

- 単体: 日数計算 (civil 変換・閏年)、放置/孤立/重複/ゾンビ/done未検証 の検出と非検出、来歴の放置 open・見直し合図、`[decay]` 無しの既定値、orphan の age ゲート、commit_gate の構造的 decay の phase 適応 (保守 block・初期警告)、age 系が commit を止めない
- 結合: next が腐敗節を要約して出す、verify.run が data.decay に違反を出す、commit ゲートが保守で done未検証/ゾンビを deny し放置は deny しない
- 手動 (後の一括対話検証へ): タスクを溜め放置/孤立/重複/ゾンビ/done未検証 を作り next と verify.run が報告するか、古い来歴が警告されるか、phase を初期→保守で commit の止まり方が変わるか、閾値調整で誤検出を抑えられるか
- 未確認として残す: 閾値の既定値の妥当性 (対話検証で詰める)、重複の曖昧一致 (将来)、per-kind severity を config 化するか (将来)

## 捨てた案

- 閾値を phase ごとの数値表で持つ → 設定が複雑化。Phase6 quality と同じく数値は固定し enforcement だけ phase 適応にする
- decay をコアゲート (検査失敗) と同じ phase 不問 block にする → age 系で commit を止め狼少年化。構造的腐敗だけ保守 block・age 系は助言
- git mtime で最後の活動を測る → 非決定論で core 方針に反する。ID 日付と note 日付で測る
- DecayConfig を別ファイル decay.toml にする → 適応度関数の設定が2箇所に割れる。quality.toml に集める
- 重複に曖昧一致 (編集距離) を入れる → 新規依存・誤検出増。slug 完全一致に絞る
- next で腐敗を全件出す → context 汚染。要約し open/放置中だけ見せる

## 見直し条件

- age 系の助言で腐敗が掃除されず block が要ると分かった時 (per-kind severity の config 化)
- 重複の slug 一致では足りないと分かった時
- 既定の閾値が多くのプロジェクトで合わないと分かった時
- 腐敗検知のコストが大きく対象限定が要る時
