# Phase 4 tool 層・記録層

## 状態

採用・実装・実機確証済み。記録層 (decisions) ・共通返り値 (封筒) ・tool (decision.record / gate.list / gate.approve / verify.run) ・計算 resource (owox://next) ・task tool 群 (note / update を含む) ・state.set を実装し、実機 Codex の stdio と対話で確証 (`docs/validation/20260613-Phase4-tool記録層.md`)。対話で見つけた粗 (Stop 素通り・用語重複排除・依存先検証・decision 過剰記録・tool 発見性) も改善済み。owox://verification は作らない。

## 決めたこと

### 記録層は 1 件 1 ファイル。decisions は Markdown

来歴 (decisions) は `.owox/decisions/<id>.md`、1 判断 1 ファイル。ID は 日付+slug (例 `20260613-可逆性境界`)。

- 形式は正本フォーマット決定 (`docs/decisions/20260612-正本フォーマット.md`) と一致: prose=Markdown。来歴は理由 (rationale) という散文が主。Markdown が読み書きしやすく、AI も生成しやすい
- 1 件 1 ファイルにする理由: 中央台帳 (decisions.toml 等に全件) は並行ブランチ・複数人で同じ行を奪い合い衝突する。1 ファイルなら衝突しない。ID=日付+slug は git で時系列に並び人が読める。連番を使わないのと同じ理由 (`docs/decisions/20260611-MCP設計.md`)
- パーサは既存の Markdown 自作パーサ (`crates/core/src/markdown.rs`) を再利用。新文法を足さない
- slug は title から導く。同日内で同 slug が衝突したら末尾に連番を足し一意化する (上書きでデータを失わない)。多言語 title を許すため slug は非 ascii 可

decision の見出し: Status / Rationale / Links / Supersedes / Approval。Status は open / adopted / rejected / superseded。

### state (phase) は config と別の machine 値

state.set の phase (initial / stable / maintenance) は machine 値なので `.owox/state.toml` に置く (config.toml = 生成・検証設定、state.toml = 状態宣言、で分ける)。tasks は task tool 群と一体で後続。

### gate は「status=open の来歴」。別ストアを持たない

人間ゲート (gate) を来歴と統一する。判断点 = 「まだ人間が決めていない来歴」= status=open の decision。

- gate.list = status=open の来歴一覧
- gate.approve { id, note? } = その来歴を status=adopted へ遷移し、Approval に承認注記+日付を追記
- 別の gate ストアを持たない。判断点と来歴が二重化しない。承認が来歴へ自然に残る (旗「検証可能な正本」)

捨てた案: gate を独立ストアに持つ。捨てた理由: 来歴と二重管理になり、承認の記録が来歴へ残らない。

### gate (永続の判断点) と needs_human (一時の信号) は別もの

混同しやすいので区別する。

- needs_human: 封筒の status。どの tool でも返せる「この応答は人間判断が要る」という一時の信号。その場で AI を止め人間へ提示させるだけで、永続しない
- gate: status=open の来歴。「未解決の判断点」をファイルに残したもの。gate.list で一覧でき、gate.approve で解決する。永続する

関係: 後で必ず片付ける未決として残したい時に open の来歴 (gate) を作る。verify.run は今 needs_human を返すが gate を自動では作らない (毎回作ると来歴が増える)。残すべき判断は decision.record で open にする。
将来 needs_human を返す tool が自動で open 来歴を作るべきかは見直し対象 (今は呼び出し側が明示的に記録する)。

### decision の鮮度・退避 (溜まっても汚染しない理由)

来歴は追記が前提で消さない (履歴が価値。git log と同じ)。溜まっても context は汚れない:

- 来歴は常時注入しない。読むのは必要時のみで、owox://next が見せるのは open だけ (adopted/superseded は出ない)。常時読む量は増えない
- 更新は supersedes で行う。新しい来歴が古いものを置き換え、古い方は superseded になる
- 腐れるのは放置された open・前提が変わったのに adopted のまま、というもの。鮮度の落ちた来歴を機械で見つけ警告するのは腐敗検知 (`docs/decisions/20260611-設計原則.md`・`20260611-タスク管理.md`、Phase 7)
- サイズが問題化したら superseded を読む経路の外 (別フォルダ) へ退避する。削除ではない。今は不要

### 共通返り値 (封筒) は core の型。Serialize のみ

全 tool が同じ封筒で返す。型は core (`crates/core/src/envelope.rs`):

- status: ok / needs_human / failed
- reason: なぜそうなったか
- next_actions: 次の手 (配列)
- decision_ids: 関連来歴 ID (配列)
- data: ok 時の本体 (tool 別。serde_json::Value)
- gate: needs_human 時の判断点 (kind / subject / requires)

core の封筒は Serialize だけ持つ。schemars (JsonSchema) は付けない。理由: schemars は MCP 境界の都合であり、core を MCP へ結合させない。mcp 側で封筒を JSON 文字列へ直列化し、CallToolResult のテキストとして返す (structured_content は Codex の対応が不確実なためテキストにする)。tool 引数の型は mcp 側に置き schemars を付ける。

### verify.run は完了を3区別。検証完了だけ機械判定

検査コマンドは config.toml の `[[verify.checks]]` (name / command) に人間が明示列挙する (tool 非依存・明示的・machine 設定は config.toml 集約の方針と一致)。

完了3区別 (`docs/requirements/20260611-core-要件.md`):

- 検証完了: config の検査コマンドを実行し全通過なら passed、失敗で failed (機械判定)
- 要件完了: 第1段階は要件↔テスト trace が無いため needs_human (`docs/decisions/20260611-品質保証.md` 導入後に機械判定へ)
- 作業完了: 与えた範囲が終わったかは機械で判定できないため needs_human

封筒 status: 検査失敗→failed、検査通過→needs_human (検証は機械で済むが作業・要件完了は人間確認)、検査未設定→needs_human。これで「verify が要件完了を人間判断として返す」を実装する。

### 日付は呼び出し側 (mcp) が与える。core は決定論

ID の日付は mcp が実行時に求め (UTC、SystemTime からの civil-date 変換。新依存を足さない)、core の record_decision へ文字列で渡す。core は時計を読まず、日付を引数で受けるため決定論でテストしやすい。

## tool 引数と data (本スライス実装分)

- decision.record { title, status, rationale?, links?{requirement?,work?,verification?}, supersedes? } → { id }
- gate.list {} → { pending:[{id, title, subject}] }
- gate.approve { id, note? } → { id, approved }
- verify.run { scope? } → { results:[{check, passed, detail}], completion:{work, requirement, verification} }
- state.set { phase } → { phase } (変更を来歴へ)
- task.create / task.list / task.update / task.close / task.drop / task.link (下記タスク層)

resource: owox://next = status=open の来歴 (未決) + ready タスク。phase は resource でなく session push で届ける。

## commit/stop の機械ゲート化 (2026-06-13)

Phase 3 で誘導までだった commit/stop を機械ゲートへ格上げした
(`docs/decisions/20260612-Phase3-hook実装.md` の予告どおり、検証済み配管の上に方針を載せる)。

### commit ゲートは検査を再実行する (古い結果で通さない)

git commit の PreToolUse で、config.toml の検査を hook がその場で再実行する。

- verify 失敗 → deny (完了検証は機械強制。`docs/decisions/20260611-制御方針.md`)。AI が verify.run をサボっても、commit 時に検査が走り落ちれば止まる (これが歯)
- 検査未設定 → 警告 (機械検証できないと伝える。deny はしない。verify.run の no-checks と整合)
- 全通過 + 未承認 open gate あり → 警告 (判断記録は誘導)
- 全通過 + open gate 無し → 素通り
- 不可逆操作は従来どおり最優先で deny

採用理由 (再実行 vs 永続+鮮度照合):

- 信頼できないゲートは無いより悪い (誤った安心)。永続+鮮度照合は「編集検出」に依存し、既に許容済みの穴 (Bash リダイレクト書き込みが編集検出をすり抜ける) が「未検証なのに通る」安全の穴に化ける。再実行はその瞬間の実コードを検証し漏れない
- 状態を持たない: 鮮度フラグ・git フィンガープリント・セッション状態ファイルが要らず `.owox/` を汚さない。no-repo / 未追跡ファイル等の端ケースも無い
- hook 無状態・判定は core・決定論、という既存思想に乗る
- コストは限定的: commit は頻度が低い (毎ターンの stop と違う)。重い検査は CI、速い検査を commit ゲートへ、とユーザが範囲を選べる

捨てた案: verify 結果を永続し鮮度照合。捨てた理由: 上記の安全の穴と、状態・編集検出の仕組みが増える。owox://verification (見える化) は将来足すが、ゲートの根拠にはしない。

### stop は誘導のまま (毎ターン発火するため再実行しない)

stop は完了前に verify・判断記録を促し、未承認 open gate があれば件数を表面化する。検査の再実行はしない (毎ターン末に走ると重い)。機械強制の完了検証は commit ゲートが担う。

### 厳しさ (phase) の state 適応 (2026-06-13 実装)

state.toml + state.set で状態適応を実装した (`docs/decisions/20260611-制御方針.md`)。

- `.owox/state.toml` に phase (initial / stable / maintenance)。人間宣言が正。既定 initial
- state.set { phase } → state.toml を書き、変更を adopted 来歴へ残す (監査)
- commit ゲートの格上げ: verify 失敗は phase 不問で常時 deny (コアゲート常時維持)。未承認 open gate は maintenance で deny、initial/stable では警告。stable の「中間」は当面 initial と同じ警告 (専用機構を作らない)

### phase は振る舞い案内も変える。session 開始で push (2026-06-13)

phase はゲートの厳しさだけでなく AI の動き方も変える。これが phase を「読めるだけの値」でなく「行動に効く値」にする。

- session 開始 hook が今の phase を読み、phase 向けの振る舞い案内を注入する: initial=大規模・構造的な改修を歓迎/推奨 (清潔さ優先、現状の振る舞い維持にこだわらない)、stable=均衡、maintenance=変更を小さく可逆に・振る舞いを保ち修正に回帰テスト
- AGENTS.md は生成時に固定される静的ファイルで phase は実行中に変わるため、静的に焼かず session 開始の動的注入にする
- owox://state は撤去した。「読めるだけで行動に効かない」resource を持たない。phase は session push とゲートで効き、人間は state.toml を直読みできる

### state の分割は per-scope を将来に (今は単一 phase)

state を直交軸 (リスク・モード等) で増やさない。phase とほぼ相関し設定負担だけ増えるため。
将来の自然な分割はディレクトリ/モジュール単位の phase (安定した中核 + 実験中モジュール)。
既存 context.md のパス単位スコープに乗せられる。「成熟 repo だが今だけ大改修したい」衝突は、新軸でなく人間 gate (一時承認) で扱う。

## 検証可能タスク層 (2026-06-13 実装)

`.owox/tasks/<id>.md` に 1 タスク 1 ファイル (来歴と同形式 Markdown・ID=日付+slug)。
正本フォーマット・並行ブランチ衝突回避は来歴と同方針 (`docs/decisions/20260611-タスク管理.md`)。

- モデル: title / status (todo/doing/done/blocked/dropped) / links (requirement/decision/verification) / deps (型付き: blocks/parent-child/related/discovered-from)
- 腐敗を防ぐ中核を最初から入れた:
  - done に検証必須: task.close は config の検査を実行し通過しないと閉じれない (自己申告 done を排除)。検査未設定は needs_human、失敗は failed
  - 来歴連動: task.drop は理由必須で、破棄理由を adopted 来歴へ残し task に decision link を張る (silent rot 禁止)
  - ready: todo かつ前提 (blocks) の対象がすべて done (検証済) のものだけ。done=検証済なので「前提タスクが検証済」を満たす
- task.update は done への遷移を拒む (close を通させ検証必須を守る)
- owox://next に ready タスクを足した (未決の来歴 + ready タスク)
- 腐敗検知の適応度関数 (stale / 孤立 / 重複 / ゾンビ / done 未検証 の機械検出) は重いため Phase 7。検証必須・来歴連動という腐敗の主因対策は本スライスで入れた

tool: task.create / task.list (ready?) / task.update / task.close / task.drop / task.link。
owox://tasks/{id} の pull resource は task.list が一覧を返すため当面持たない。

## owox://verification は作らない (2026-06-13)

owox://state 撤去と同じ理由で作らない。

- 検証の真実は verify.run (再実行・新鮮・正確) が持つ。読みたければ呼べばよい
- 保存した結果は編集後に古くなる。古い passed を信じて動くのは commit ゲートで「永続+鮮度照合」を却下した罠そのもの
- 「読めるが再実行で済む or 古くて誤誘導する」resource は持たない (owox://state と一貫)

将来ほしければ「検証の履歴」(いつ何が通った/落ちたの追記式・監査用) として別途。これは「今 verified か」とは別目的で優先度は低い (任意)。

## 対話検証で見つけた粗の改善 (2026-06-13)

実機 Codex の対話検証 (`docs/validation/20260613-Phase4-tool記録層.md`) で振る舞いの粗が出た。
機構と誘導を分けて直した。設計の要点:

### Stop は何もしてないターンで黙る

stop_decision に「作業ツリーが汚れているか」を引数で渡し、クリーン かつ 未承認 open gate ゼロ
なら受理 (出力なし) にした。どちらか有る時だけ 1 度継続する。

- 汚れ判定は mcp が git で行い真偽値だけ core へ渡す。core は時計も外部状態も読まない (決定論維持)
- `.owox` 配下は汚れ判定から除外する。decision / task を作っただけのターンは verify 対象でないので黙る
- porcelain の entry 行 (status 2 文字 + 空白) だけを数える。空行やラッパが足すノイズ行を汚れとみなさない
- git が無い・失敗時は汚れ扱いで継続へ退避し、変更の見逃しを避ける (安全側)

### 用語注入は session 単位で重複排除

同じ用語を同 session で何度も注入していた。session_id をキーに注入済み語を覚え、既出語は再注入しない。

- 注入済み語は `.owox/.cache/sessions/<id>.json` に置き、`.owox/.gitignore` の `.cache/` で履歴から外す
- core の glossary_injection は注入文と一致語を返し除外集合を受ける。mcp がキャッシュを読み書きする
- session_id が無い・読めない時は従来どおり毎回注入で退避

### task.create は存在しない依存先を弾く

依存先 (deps の target) が実在タスクか照合し、未知なら理由つきで弾く (壊れた依存グラフを防ぐ)。
前方参照は task.create では受けず、後から task.link で張る。

### decision は耐久性ある判断だけ。一時メモは task.note へ

一時的・作業状態の判断まで来歴に乱立し検索性とコンテキストを圧迫していた。

- Stop の継続文・decision.record の説明・get_info を「耐久性ある設計・方針判断だけ。一時は task.note」へ揃える
- task.note を追加 (`## Notes` に日付つき追記)。来歴を作らない軽量記録
- task.update を title 変更へ拡張。title (タスクが何であるか) の変更だけ reason 必須で来歴へ残す。
  status / links / deps の変更は来歴なし (workflow と metadata まで来歴にすると過剰記録へ逆戻り)

### tool の発見性は get_info で上げる

AI が phase 変更 (state.set) と task の rename (task.update) で tool に気づけず迷った。
get_info の instructions に state.set と「rename は task.update で (新タスクを作らない)」を明記し、
task.create の説明にも rename は task.update と添えた。実機再検証で両方とも迷わず到達するようになった。

## 捨てた案

- 来歴を中央台帳 (TOML/JSON に全件) で持つ → 並行ブランチで衝突
- 封筒を mcp 側の型にする → 「フィールドの正本は core」(`docs/decisions/20260611-MCP設計.md`) に反する
- core に schemars を足し封筒へ JsonSchema → core を MCP へ結合させる
- verify が検査コマンドをエコシステム自動判定 → 推測が外れ tool 非依存方針と緊張
- structured_content で封筒を返す → Codex の対応が不確実

## 見直し条件

- 来歴件数が増え 1 ファイル運用がディレクトリ肥大で扱いにくくなった時
- 封筒 data を型付き schema で開示する必要が出た時 (Codex が structured output を要求する等)
- 検査コマンド列挙では足りず quality.toml の機械検証 (Phase 6) が要る時
- gate=open 来歴の統一が、来歴に出したくない一時的判断点で窮屈になった時
