# Phase9 性質軸適応機構

## 状態

採用 (設計確定・未実装)。Phase 9 (`docs/roadmap/20260612-完成ロードマップ.md` の Phase 9 節) の前提。
要件への落とし込みは別途 (`docs/requirements/`)。本書は採用判断の正本。
設計判断はマネージャ問いで人間が1件ずつ確定済み (2026-06-18)。
前提整理は `docs/handoff/20260617-性質軸適応機構とブランチ作業記憶.md`。

## 背景

owox-harness を使う側の AI 駆動開発の進め方の話であり、owox-harness 自体の作り方ではない。
AI 駆動開発の方法論 (PRFAQ・理想先行・段階化・クリーンアーキテクチャ) は、人間判断と AI 自律の境界を自然に定義する。
owox の存在理由 (人間が重要判断を握り AI 自律を最大化) と構造的に一致する。
PRFAQ 承認・理想の優先順位・段階分割・層境界 = 人間ゲート、その間の実行 = AI 自律。

これらは現状 owox に載っていない。canon の rules/practices へ人間が書けば乗るだけ。
本機構は、この方法論を owox が既定で持ち、プロジェクトの性質に応じて出し入れする。

## 採用設計

### 性質軸を追加 (時間軸と直交)

owox は既に適応軸を1本持つ = state.toml の phase (initial/stable/maintenance) = 時間軸。
enforcement の強さと session 開始の振る舞い案内を変える。

今回追加は性質軸 = プロジェクトの種類で振る舞いを変えるもう1本。立ち上げで決まりほぼ固定。phase と直交。
ただし永久ロックでなく、逆生成で再検出・人間が後から変更できる (phase が状態で動くのと同じく滅多に動かないが動かせる)。

### 軸が基本単位・プリセットは砂糖衣

固定の named プリセット列挙は採らない (混成で破綻・追加でコード変更が要る)。
直交する性質軸を基本単位にし、各軸が1モジュールを on/off/差し替え。Phase 9 の軸は4本:

- requirements-shape: prfaq / lightweight
- prioritization: ideal-first / incremental
- delivery: phased / continuous
- architecture: layered / flat

4軸は今回扱う4方法論と1対1。プロファイル (束) = 軸選択の名前付き束にすぎない。
混成は軸を混ぜ、新性質は束を足すだけ (コード変更不要)。
owox 既存流儀 (quality.toml が境界宣言・review.toml がトリガ宣言する合成型 config) と一致。

軸セットは Phase 9 では4本のまま。投機的に増やさない。軸追加 = 1モジュール追加 = コード変更なので実需要が出た時だけ足す (束追加はタダだが軸追加はタダでない)。

### 既定束 = フル方法論・軸で上書き可

姿勢: フル方法論を既定とし軸で上書き可。素の新規は prfaq + ideal-first + phased + layered を既定で得る。
軸で全 off も可で、その場合 owox は中立な土台に戻る (方法論の押し付けでない)。
フル方法論は「性質未確定の素の新規」の既定であり、kickoff が性質を確定したらそこから軸を導出して上書きする (押し付け感を消す)。

### 宣言場所 = 新規 profile.toml

性質 (固定) を新ファイル profile.toml に書く。時間で動く state.toml と別ファイルにして関心を分ける。
owox の1概念1ファイル流儀と一致。

### 4方法論を既存面へ実装 (新サブシステムを足さない)

- PRFAQ → 要件起草スキル req (単一)。requirements-shape の値で中の起草方法が prfaq / lightweight に切り替わる。
  profile が変わっても skill 名 req は不変・中の振る舞いだけ変わる (入口の名前は安定・中身が軸で切り替わる owox 流儀)。
  PRFAQ は要件の高度のみ決める。性質はモジュール on/off だけ決め高度は決めない。
- 理想先行 → 要件正本層 (Phase 6) に今ない優先度属性を足す + 優先順位付けの人間ゲート。
  優先度 = 人間が並べたランク・AI は提案まで (理想の優先順位は人間判断を機械で守る)。
- 段階化 → タスク/要件に軽量な stage グルーピング。delivery=phased の時だけ active。
- クリーンアーキテクチャ → quality.toml の層境界を唯一の層真実として再利用し3分解:
  - 依存性ルール = quality.toml の層境界/依存方向 (既存そのまま)
  - 進行度/達成度 = 要件/タスクに layer タグ → 層別に充足報告。quality.toml [[layers]] に任意の name を足し、layer タグはこの名前集合へ照合する (未知は弾き有効名を返す・name 未宣言なら照合しない後方互換)。層別報告とゲート層を同じ層名で揃える
  - 自律度勾配 = ゲートで層別に強弱 (下記)
  - architecture=layered の時だけ層機構 active。script は全 off

### 層別自律度勾配

AI が人間承認なしで変えてよい度合いを構造の層ごとに変える。核ほど慎重:

- guarded (核/内側) = 操作前に人間・phase 問わず固定
- supervised (中間) = 助言 + 記録・phase が commit-block を変調
- free (外側) = 安全ゲートのみ

### 軸×phase の合成規則 (最重要)

二重ゲートの混乱を避けるため、phase 軸と層 軸を別種の直交ゲートにする:

- phase ゲート = 遵法の厳しさ (非準拠に初期=警告 / 安定=中間 / 保守=block)
- 層 ゲート = 変更権限の所在 (その層を AI が人間承認なしで変えてよいか)

別の関心なので両ゲートが独立に発火し AI は両方を通す (AND)。
1判定へ畳む時は厳しい方が勝つ (max)。重症度のはしご1本:
free < warn < commit-block < pre-action-human-gate。

層の自律度をはしごへ写像:

- guarded → pre-action-human-gate (phase 不問)
- supervised → advise + 記録 (phase が commit-block を変調)
- free → 安全ゲートのみ

最終 enforcement = max(層写像, phase写像)。検算 (矛盾しないことの確認):

- 初期 + free外側 → free (AI 全自律)
- 初期 + guarded核 → human-gate (早期でも核は人間)
- 保守 + free外側 → commit-block (phase が勝つ)
- 保守 + guarded核 → human-gate (層が勝つ)

実装は既存ゲートの拡張。guarded の人間ゲートは canon.propose (固定正本の変更を提案→人間→承認時に自動適用) の仕組みをコード層へ広げる。層の真実は quality.toml 境界を再利用。

### guarded 発火範囲 (v1)

guarded 層の全編集で操作前ゲートを鳴らすと AI 自律を殺す。v1 は4段で線引き。全て既存 quality.toml 部品 (パターン/パス規則) で実現でき構文解析器が要らない:

1. guarded 層のファイル削除 (不可逆) → 操作前に人間ゲート (パスだけで判定でき書込前に分かる)
2. (任意・プロジェクトが選ぶ) guarded 層の「契約面」を quality.toml にパス/パターンで宣言 (例 `*/ports/*`・`*_interface.*`) → そのファイル編集は人間ゲート。構文解析せずパス規則の流用で契約っぽい所だけ守るを近似 (パスだけで判定でき書込前に分かる)
3. guarded 層で境界違反/依存方向違反 → commit-block。これは編集内容に依存し書込後にしか判定できないため操作前ゲートにできない。commit で guarded を phase 不問で block へ落とす
4. それ以外の guarded 層編集 → 助言 + 記録で流す

実装の線引き = 操作前に分かるもの (削除・契約面パス) は操作前人間ゲート・書込後にしか分からないもの (境界違反) は commit-block。操作前ゲートは編集前 hook で発火し commit-block は commit hook で発火する (`crates/core/src/gate.rs`)。

正確な契約/型変更の検出 (構文解析が要るもの) は後段 (別 Phase) へ。

### 逆生成での性質検出 (厚い検出 + draft 人間確定)

既存プロジェクトに後入れする時、安い・言語非依存・ネットワーク不要のシグナル (ファイル構造・名前・見出し・git ログ/タグ/ブランチ) で各軸の寄りを当てる:

- architecture (layered/flat): 層ディレクトリの有無 (domain/usecase/infra・ports/adapters・core/ui 等)・quality.toml 層境界の有無・import/依存方向の一貫性
- requirements-shape (prfaq/lightweight): docs/requirements/ や構造化要件ファイルの有無・PRFAQ 風文書 (ファイル名・プレスリリース風見出し・FAQ 節)・RFC/設計文書ディレクトリ・README の充実度 (Goals/Non-goals)
- prioritization (ideal-first/incremental): ロードマップ/Vision 文書の優先順位記述 → ideal-first 寄り・issue 駆動の小刻みコミット → incremental 寄り
- delivery (phased/continuous): git タグ/リリースの規則性・CHANGELOG のバージョン区切り・release ブランチ → phased・trunk ベース/連続マージ/タグ無し → continuous

守り:

- 出力は profile.toml の draft (提案) + 根拠。いきなり確定しない。kickoff で人間が確認・確定 (owox の提案→人間ゲート流儀)
- 必ず根拠を見せる (例「domain/usecase/infra があるので layered と推定」)。厚いヒューリスティックは外すので証拠を添え自動確定しない
- v1 は owox 組込みヒューリスティック。per-project の検出規則拡張は将来

### kickoff の導線 (生軸を人間に触らせない)

人間に生の軸を触らせず、人間の言葉で性質だけ聞き owox が軸を導出する:

- 主導線 = 名前付き束を人語で選ぶ (クリーンアーキ型アプリ / 使い捨てスクリプト / ライブラリ / データ基盤 / 調査 等)。各束が裏で4軸へ展開
- 新規 = 束を選ぶ・既定はフル方法論束 (クリーンアーキ型アプリ)
- 既存 = 逆生成が束を先に当てて「推定 ○○型 (根拠つき)」を提示 → 人間は合ってる/直すだけ
- 上級者の逃げ道 = profile.toml に生軸が書かれるので直接編集可。通常の導線では触らせない
- 束→軸の対応表は owox 組込み + 束追加可 (config 追加・コード不要)。具体の束→軸対応は要件で詰める

### profile.toml の型

owox 合成型 config 流儀 (quality.toml/review.toml) に揃える。

per-project (profile.toml に書く):

- `[profile] preset = "..."` 名前付き束を選ぶ (主導線)
- `[profile.axes]` 束からの差分・上級者のみ。省略時は束の展開値
- `[bundles.<name>]` このプロジェクト独自の束 (任意・コード不要)

owox 組込み (profile.toml に書かない):

- 軸レジストリ = どの軸が在るか + 各軸の許容値 (各値が1モジュールに対応するため組込み)
- 標準束

軸の値域:

- 各軸 = 二値固定でなく列挙集合として設計 (将来 architecture に hexagonal 等を足せる余地)。v1 は各軸2値
- 軸の追加・軸への新値追加はモジュール追加 = コード変更が要る (枠の制限)

束の解決順: 実効軸 = 束の既定値に `[profile.axes]` の上書きを重ねる。

型検証 (読込後 Rust):

- preset は既知の束 (組込み or プロジェクト定義) であること
- axes のキーは既知の軸・値は許容値であること。未知は弾いてフィードバック (canon.add 未知 section と同型)

保存しないもの: 逆生成の根拠 (evidence) は profile.toml に残さない。kickoff 提示時だけ見せる (保存ビューは腐る哲学)。

## 変更境界

触る (Phase 9):

- profile.toml 新規 (性質軸の宣言)・読込/型検証・軸レジストリと標準束 (組込み)
- 解決点1つ: 軸 → active モジュール (floor_context 合成と同型)
- 要件正本に優先度属性、要件/タスクに任意 layer・stage タグ
- ゲート合成に層×phase の max 規則 + guarded の人間ゲート (canon.propose をコード層へ拡張)
- 逆生成の性質検出 (厚いシグナル) + profile.toml draft 生成
- kickoff の束選択導線
- 要件起草スキル req (requirements-shape で起草方法が切り替わる)

触らない (守る振る舞い):

- phase 軸は不変。性質軸を別物として追加
- 高度固定: PRFAQ は要件高度のみ。性質はモジュール on/off だけ決め高度は決めない
- 完了判定は受け入れ基準/振る舞いベース不変。layer タグは進行度報告のみ
- canon 直読み禁止・core は時計を読まない (today は mcp 供給)

## 危険

- guarded 人間ゲートが重すぎて AI 自律を殺す → v1 の4段で発火を絞る。実機で範囲を調整
- 厚い逆生成検出が外す → draft + 根拠提示で人間が判断。自動確定しない
- 二重ゲートで AI も人間も混乱 → 別種・AND・max・はしご1本で解消。検算表で矛盾なしを確認済み
- 性質を永久ロックして実態とずれる → 逆生成再検出 + 人間変更を最初から許す
- 2つの新機構 (性質軸とブランチ作業記憶) を絡めすぎる → stage/delivery とブランチ運用は v1 疎結合 (`docs/decisions/20260618-Phase9-ブランチ作業記憶層.md`)

## 検証方針

- 単体: profile.toml の読込/型検証 (未知軸/未知値を弾く・preset 解決・axes 上書き)、軸→active モジュールの解決、合成規則の max (検算4ケース)、guarded 4段の発火判定、逆生成シグナルの寄り推定
- 結合: serve 経由で profile に応じ active モジュールが変わる、kickoff の束選択→profile.toml 生成、逆生成 draft の根拠提示
- 手動 (実機): 束を人語で選び生軸を触らずに方法論が出し入れされる、既存 repo から性質 draft が根拠つきで出る、核の境界違反/削除で人間ゲートが鳴る

## 捨てた案

- 固定 named プリセット列挙を基本単位にする → 混成で破綻・追加でコード変更が要る (人間判断で却下)
- 軸を投機的に増やす (テスト厳格度/レビュー強度/文書化レベル) → owox が既に phase 適応や review.lenses で扱い二重になる。実需要時に足す
- 性質を永久固定 → 実態とずれる。再検出/変更可にする
- 逆生成で根拠を profile.toml に保存 → 保存ビューが腐る。kickoff 提示時のみ
- 生軸を kickoff で直接聞く → 人間に軸の専門用語を触らせる。束を人語で選ぶに寄せる
- 契約/型変更を v1 から構文解析で検出 → 言語ごとに重い。v1 はパス宣言で近似し正確検出は後段

## 見直し条件

- guarded の4段が実機で重い/緩いと分かった時 (範囲の再調整)
- 4軸で性質を表しきれない実需要が出た時 (軸追加)
- 軸が3値以上必要になった時 (列挙値の追加・モジュール追加)
- 逆生成シグナルの寄り推定が外しすぎる時 (シグナル/重みの見直し)
- 束→軸の標準対応が実利用と合わない時
