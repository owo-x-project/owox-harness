# MCP コンテキスト削減

採用判断。2026-06-20。
元議論 `docs/handoff/20260620-自律度方針とコンテキスト効率の深掘り.md` の論点4。

MCP の tools/list は接続時に1回ロードされ session 中コンテキスト窓を固定占有する。
これを実測で測り直し、安全に削れる手段を確定した。

## 背景と実測の訂正

handoff は「inputSchema が最大費目 (約24,560字)・doc comment 転写9,100字が本命・schema スリム化が効く」と概算していた。
実機 (owox serve を stdio で起こし tools/list を取得・推定は tokens.rs の estimate_tokens) で測り直したところ前提が覆った。

- ツール数: 46 (handoff の概算47は誤り)
- inputSchema 総量: 16,073字 / 5,114トークン (概算24,560字を大きく下回る)
- うち description 転写分 (doc comment 由来): 3,688字 / 1,997トークン (概算9,100字の4割)
- ツール本体 description (inputSchema 外の機能説明): 13,539字 / 3,404トークン
- 起動時の MCP 実載は inputSchema 5,114 + 本体 description 3,404 = 約8,500トークン

最大の費目は doc comment でなく構造定型 (意味に影響しない反復) だった。

- `"$schema":"https://json-schema.org/draft/2020-12/schema"` が全ツール反復 (34回・1,904字)
- `"title":"XxxParams"` struct 名 (34回・946字)
- `"default":null` / `"default":[]` (serde default 由来・約47件・574字超)
- nullable 型の `,"null"` (50回・350字)
- 合計 約3,400字 / 約850トークン

handoff の「inputSchema doc comment が本命」は誤りで、本命は構造定型の除去だった。

## 採用した案

合計で約2,130トークン削減 (起動時 MCP 実載 約8,500の約25%)。すべて意味を保つ。

### 構造定型の除去 (最大・約850トークン)

生成後除去 (B 方式) を owox 側だけで行う。schemars/rmcp の改造は不要。

- schemars 1.2.1 / rmcp 1.7.0。rmcp の schema_for_type が SchemaSettings を内部固定し生成時の抑制 (A 方式) は差し込めない。よって生成後に JSON を後処理する
- `ToolRouter.map` / `ToolRoute.attr` / `Tool.input_schema` はいずれも pub。サーバ構築時 (`crates/mcp/src/serve.rs` の tool_router 受け取り箇所) に各 inputSchema を後処理し、`$schema` / `title` / 値が null・空配列の `default` を取り除く。`Arc::make_mut` で copy-on-write しキャッシュを壊さない。tool 追加時も自動適用され追従不要
- title・default の除去は完全に安全。クライアントの引数解釈に無関係で、デシリアライズは Rust の `#[serde(default)]` が担い schema と独立 (schema から消してもパース結果は変わらない)
- `$schema` の除去だけは実機 Codex 検証をゲートにする。MCP 仕様は `$schema` を必須化していないが、欠落で JSON Schema draft の推定に失敗する厳格クライアントの理論的懸念がある。target 検証で `$schema` 抜き schema を Codex に食わせ tools/list が正常に解釈されるか確認してから外す。拒否されたら `$schema` だけ残す安全弁 (title・default は依然削れる)

### inputSchema の冗長除去 (約280トークン)

description 中の括弧補足を削る。機械的・低リスク。

- `(任意)` (13件) は serde default 付きで required 配列から既に外れており重複。完全に削れる
- `(人間が読む 1 行)` `(repo 相対)` `(上級者)` 等の括弧補足も本文へ畳むか削る
- memory の「生成・注入文に括弧の補足説明を使わない」方針と整合する

### ツール本体 description の短縮 (約1,000トークン)

ツール本体 description (3,404トークン) を意味維持で約30%圧縮する。

- 冗長な言い回しを削る。ツール選択の判断材料は保つ (発見性を落とさない範囲)
- 46ツール本文を1つずつ書き換える手作業。圧縮しすぎると AI のツール選択精度が落ちるので、削るのは重複・冗長に限る

## 採用理由

- 構造定型除去は意味ゼロの反復を消すだけで最大の削減。owox 側完結・tool 追加に自動追従し清潔
- title・default は安全が確定しているので無条件。`$schema` だけ実機懸念があるので検証ゲートで分ける
- 冗長除去は機械的で安全・括弧禁止方針と一致
- description 短縮は効果が大きいが手作業で発見性に触れるので、重複・冗長に限定する
- 安全策の合算で約2,130トークン取れるため、リスクのある手段 (遅延公開・enum 化) に手を出す必要がない

## 捨てた案

- enum 化 (status 等を Rust enum + schemars の enum 制約へ) → JSON Schema の enum 制約が増え doc comment の列挙行が減る差し引きが実質±0トークン。トークン目的では旨味なし。入力検証・誤値弾きの品質目的なら別途採用可
- 遅延公開 (experience/profile/state を起動時 tools/list から外す) → 起動時に外すと AI がツールの存在を知れず床 routing からも外れる (memory の発見性原則)。動的公開機構は無く、実装は生成時の静的フィルタで「そのターゲットで恒久無効化」になる。削減約820トークンに対しリスク・複雑性が見合わない。安全策で十分なので不要
- 生成時抑制 (A 方式・schemars settings で $schema/title を出さない) → rmcp が SchemaSettings を内部固定し差し込めない。rmcp フォークか各 tool で input_schema を手書きする高コストで将来追従が破綻

## 見直し条件

- 実機 Codex が `$schema` 欠落で tools/list を拒否 → `$schema` は残し title・default のみ除去
- description 短縮後に AI のツール選択精度が落ちた → 削りすぎた本文を戻す
- 起動時 MCP 実載がさらに問題になった → enum 化・遅延公開を動的公開機構とセットで再検討
- rmcp / schemars のバージョン更新で inputSchema 生成経路が変わった → 後処理箇所を再点検
