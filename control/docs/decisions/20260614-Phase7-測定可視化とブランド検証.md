# Phase7 スライス3: 測定の可視化 + ブランド検証の設計

## 状態

採用・設計確定。実装はこれから。Phase7 スライス3 (`docs/requirements/20260614-Phase7-腐らず成長する.md`)。
スライス1・2 (腐敗検知の中核・コード/repo 拡大) は実装・機械確証・コミット済み。
スライス3 は独立で軽い (`docs/requirements/20260614-Phase7-腐らず成長する.md` の依存節)。

## 背景

旗「最小コンテキスト」を主張でなく測定可能な事実にし、ブランド (用語・禁止語・表記) のぶれを機械で防ぐ。
3 部品を立てる: 文脈出力のトークン量可視化・文脈出力時の秘密情報走査・ブランド機械検証。
いずれも既存の手本へ載せ、新サブシステムを作らない。

人間判断 (plan の論点) を確認し確定した:

- トークン量は日本語対応の軽い推定で出す。正確なトークナイザは入れない (重い・新規依存・言語非依存方針に反する)
- 禁止語は glossary.md に構造化する。用語集を「用語+定義+禁止語」の正本にする (`docs/decisions/20260611-設計原則.md` のブランド固定)
- 表記・文体検証は正規表現で表せる規則のみ機械化する (用語/禁止語は確実・助詞省略など構文解析が要るものは対象外と明示)

## 採用設計

### トークン量の推定は core の純関数

core に `tokens` モジュールを新設し `estimate_tokens(text) -> usize` を置く。
決定論・新規依存なしの軽い推定。core はファイル読取と純計算を持てる (時計/git は持たない) 制約に合う。

- 日本語混在で `文字数/4` は実数より大幅に小さく出るため CJK を別勘定にする
- 漢字・かな・全角など CJK 文字は 1 文字 ≒ 1 トークン、それ以外は 4 文字 ≒ 1 トークンで合算する
- 厳密値でなく桁感を示す推定。verify.checks のような pass/fail ゲートにはしない (主張でなく事実を見せるのが目的)

### 文脈出力にトークン数と秘密走査を載せる

context tool は封筒でなく描画本文を直接返す例外 (`crates/mcp/src/serve.rs` の `render_context`)。
描画の末尾へ推定トークン数の 1 行を足し、同じ本文を秘密走査して検出時は警告行を足す。

- トークン行: owox が文脈ナビとして注入する情報量を毎回数値で見せる (最小コンテキストの測定可能化)
- 秘密走査: 描画本文を `secret::scan` にかけ、検出した種別と理由を警告行に出す。秘密の値そのものは出さない
- context は読取専用ナビなので block しない。警告に留める (秘密の混入は target の設定ミスで、止めるより気づかせる)

### 秘密走査は irreversible 方式の再利用モジュール

core に `secret` モジュールを新設する。`irreversible.rs` と同じ「組込み正規表現 + 将来の target 拡張」の型に載せる。
スライス4 の経験 export/import が本命の再利用先で、ここで土台を据える。

- `SecretFinding { id, detail }` を返す。検出元の識別子と人間向け理由を持ち、秘密の値は載せない (走査結果が新たな漏れ口にならない)
- 組込みパターンは高価値で誤検出が少ないものだけ持つ: 秘密鍵ヘッダ (`-----BEGIN ... PRIVATE KEY-----`)・AWS アクセスキー (`AKIA` 形式)・汎用の token/secret/password 代入。保守的に置き正常テキストを巻き込まない
- `scan(text) -> Vec<SecretFinding>` を公開。target 固有パターンの受け口は将来 (スライス4 で export/import の需要が立つ時) に rules.md の節として足す。今は組込みのみ
- irreversible と同じく組込みパターンは読込なしの const、妥当性は compile テストで担保する

### ブランド検証は glossary.md の禁止語 + 既存 boundaries

「ブランド機械検証」は新サブシステムでなく既存 quality の禁止パターン機構の拡張として実装する。

#### 禁止語は glossary.md の予約見出し Forbidden に構造化

glossary.md に予約見出し `Forbidden` を新設し、配下の `- パターン: 理由` を禁止語エントリにする。
他の見出しは従来どおり用語定義 (`- 用語: 説明`)。これで用語集が「用語+定義+禁止語」の単一正本になる。

- 禁止語パターンは正規表現で持つ (owox 全体が正規表現で揃う。irreversible・boundaries と同じ)。読込時に妥当性を検証し誤記を早期に弾く
- 単純な語はそのまま正規表現の literal として書ける。表記の確実な規則 (例: 日本語文中の英語の特定語) も同じ枠で書ける
- 禁止語は人間が編集する (ブランドは人間が握る。`crates/core/src/glossary.rs` の変更・削除が人間ゲートなのと同じ思想)。glossary.add は通常用語の追記のみで Forbidden を触らない

#### run_brand が QualityViolation として報告

core に `run_brand(glossary, work_dir, files) -> Vec<QualityViolation>` を置く。
追跡テキストファイルを読み、禁止語パターンに当たれば `QualityViolation { kind: "brand", path, detail }` を作る。

- 報告チャネルは quality 違反と同じ。新チャネル・新ゲートを作らない (要件「quality で機械検証」に従う)
- detail に当たったパターンと理由を載せる。秘密走査と違い禁止語は秘密でないのでパターンを出してよい
- 検査の流儀は `run_quality` の boundaries と同じ (glob で対象ファイルを絞り、内容へ正規表現マッチ)。実装も内部で同じ補助を使い二重化しない

#### 表記の正規表現規則は既存 boundaries を流用

正規表現で表せる表記規則 (MDテーブル禁止・特定の英語混入など) は quality.toml の `[[boundaries]]` で既に書ける。
ブランド用に新機構を足さず、target は boundaries に表記規則を書く。glossary の禁止語は語彙の正本、boundaries は正規表現の power-user 枠と役割を分ける。

- 助詞省略・記号化・相槌の有無など構文解析が要る文体は機械検証の対象外と明示する (owox は言語非依存・正規表現の範囲に留める)
- Rust の正規表現は後読み (lookbehind) を持たないため「control/target が前に付かない harness 単独」のような文脈依存は厳密には書けない。確実に書ける規則に留め、書けないものは AGENTS.md の人間向け記述に委ねる

### 報告経路と効かせ方 (既存チャネルに合流)

- context tool: トークン行と秘密警告を描画本文へ。block しない
- verify.run: mcp が run_quality と run_brand の違反を結合し既存 `quality_violations` 引数へ渡す。`run_verify` の署名は変えない。data.quality に kind="brand" が混ざって出る
- commit ゲート: 変更なし。quality 違反 (boundary・budget・brand) は保守 phase で commit を止め、初期・安定は警告のみ (`crates/core/src/hook.rs` の commit_gate が kind を問わず QualityViolation を扱うため自動で適応)
- next: 変更なし。ブランド・トークン・秘密は next に載せない (next は素早い航法に保つ)

### 責務分担

- core: estimate_tokens (純計算)・secret::scan (組込み正規表現)・run_brand (glossary 禁止語の走査)・Glossary への Forbidden 解釈
- mcp: context tool の描画でトークン行と秘密警告を合成、verify.run で run_quality と run_brand を結合

## 変更境界

触るもの:

- crates/core/src/tokens.rs: estimate_tokens の新設・テスト
- crates/core/src/secret.rs: scan と組込みパターンの新設・テスト
- crates/core/src/model.rs: Glossary へ forbidden フィールド、from_markdown の Forbidden 見出し解釈
- crates/core/src/quality.rs: run_brand の新設 (QualityViolation kind="brand")・補助の共有
- crates/core/src/lib.rs: 新 API の公開
- crates/mcp/src/serve.rs: render_context へトークン行と秘密警告、verify.run で run_brand を結合
- tests/fixtures・target サンドボックス・docs

触らないもの:

- run_verify の署名・data.quality の形・完了3区別
- commit_gate (QualityViolation を kind 不問で扱うため brand も自動で乗る)
- next の描画・decay チャネル (スライス1・2)
- glossary の通常用語の CRUD (add/update/remove)・boundaries の既存挙動

守る振る舞い:

- core は git/時計を持たず決定論 (ファイル列挙は mcp が与える・トークンは推定)
- 文脈出力は read 専用ナビで block しない (警告のみ)
- glossary.md に Forbidden 見出しが無くても従来どおり動く (禁止語は空)
- 禁止語・秘密の走査が誤検出で作業を妨げない (保守的な組込み・閾値調整は人間)

## 危険

- トークン推定の不正確さ。桁感を示す推定と明示し、pass/fail ゲートにしない。CJK 別勘定で日本語の過小評価を抑える
- 秘密走査の誤検出 (狼少年化)。組込みは高価値・低誤検出のものだけ。context では block せず警告に留める。export (スライス4) では人間ゲートへ送る
- 禁止語の正規表現が広すぎて正当な語を巻き込む。報告のみ (verify.run・初期 phase) で block は保守 phase のみ。人間が glossary.md で調整できる
- ブランド検証と code 用 boundaries の役割混同。glossary 禁止語 = 語彙正本、quality.toml boundaries = 正規表現の汎用枠と分ける
- Forbidden 見出しの予約が target の自由な見出しと衝突。見出し名 Forbidden を 1 つ予約するだけに留め、他見出しは従来どおり用語として読む

## 検証方針

- 単体: estimate_tokens が CJK と ASCII で妥当な桁を返す・空文字 0。secret::scan が組込みパターンを検出し正常テキストを巻き込まない・値を漏らさない。Glossary が Forbidden 見出しを禁止語へ・他見出しを用語へ振り分ける。run_brand が禁止語を検出し glob で対象を絞る・無設定で空
- 結合: render_context の末尾にトークン行が出る・秘密を仕込むと警告行が出る。verify.run の data.quality に kind="brand" が混ざる。commit ゲートが保守で brand 違反を止め初期で警告のみ
- 手動 (後の一括対話検証へ): context tool でトークン数が見える・秘密を仕込んだ context.md で警告が出る・glossary.md の Forbidden に語を入れ verify.run が brand 違反を出す・boundaries で表記規則を書き検出する
- cargo test / clippy / fmt
- 未確認として残す: context が指す参照先ファイル自体の秘密走査 (将来 opt-in)・target 固有の秘密パターン (スライス4)・トークン推定の係数調整

## 捨てた案

- 正確なトークナイザ導入 → 重い・新規依存・言語非依存方針に反する。桁感の推定で旗の測定可能化は足りる
- 単純な 文字数/4 → 日本語で大幅な過小評価。CJK 別勘定にする
- 禁止語を quality.toml の boundaries にだけ置く → 用語の正本が glossary と quality.toml に分散しブランド固定の意図が薄れる。語彙は glossary に集める
- 禁止語を literal 文字列マッチにする → owox 全体が正規表現で揃うのに 1 箇所だけ別流儀になる。正規表現で統一し literal はそのまま書ける
- ブランド検証に専用チャネル・専用ゲートを新設 → quality チャネルで足りる。新ゲートは commit の判定を複雑化する
- 文脈出力の秘密検出で block → context は読取専用ナビ。block すると正当な作業を止める。警告に留める
- 表記文体を全面的に機械検証 → 助詞省略など構文解析が要り owox の言語非依存に反する。正規表現の範囲に留める

## 見直し条件

- トークン推定が運用で桁を外し正確な計測が要ると分かった時
- 秘密走査の組込みが誤検出/見逃しで調整・target 拡張が要る時 (スライス4 で export/import の需要が立つ時)
- 禁止語の正規表現だけでは表記規則を表せず別機構が要ると分かった時
- ブランド違反を quality チャネルでなく独立に扱う必要が出た時
- context が指す参照先ファイルの秘密走査が要ると判断がついた時
