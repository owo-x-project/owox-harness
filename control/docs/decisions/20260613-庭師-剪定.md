# 庭師 (剪定)

## 状態

採用 (概念確定)。実装は Phase 7 (腐敗検知) + Phase 8 (subagent)。

## 背景

owox の成長観は「育てる」だけでは保てない。検証つきで育つ (農家) のと対で、不要を見つけて刈り込む (庭師) が要る。
最小コンテキストと堅牢さは、増やす力と減らす力の両方で保たれる。剪定が無いと、コード・タスク・来歴・ルールが溜まり腐る。

骨は既にある:

- 腐敗検知 (`docs/decisions/20260611-タスク管理.md`): 放置 (stale) / 孤立 (リンク無) / 重複 / ゾンビ (永久 block) / done 未検証 を機械検出し掃除を促す
- 記憶の鮮度 (`docs/decisions/20260611-設計原則.md`): 判断・経験に時間と鮮度を持たせ、古さを腐敗検知へ連動する
- 削除基準 (rules 正本): 何を消してよいか・消す前にすること

足りないもの:

- 腐敗検知の対象がタスク・判断・経験中心で、コードや repo 全体の不要 (死んだコード・未使用・重複) が明記されていない
- 検証つきで剪定する専用の役割 (庭師 subagent / 掃除 skill) が target 向け生成にまだ無い (cleanup skill は control repo 用であって生成物ではない)

## 決めたこと

### 庭師を target-harness の一級概念にする

農家 (育てる) / 庭師 (刈り込む) を対の役割として持つ。庭師は検証つきで不要を剪定する。

- 機械検出 (腐敗検知) の対象を広げる: タスク・判断・経験に加え、コード・repo 全体の不要 (死んだコード・未使用・重複・孤立ファイル) を含める
- 庭師 subagent / 掃除 skill を target harness が生成する。役割・権限はオーケストレーション (`docs/decisions/20260611-オーケストレーション.md`) の役割別 subagent に乗せる
- 来歴の溜まりは腐敗検知が鮮度で警告し、context は open だけ見せる構造で汚染を防ぐ (`docs/decisions/20260613-Phase4-tool記録層.md`)。サイズ問題化時は superseded を読む経路の外へ退避 (削除でなく退避)

### 剪定は必ず verify と削除基準を通す (盲目削除しない)

- 消す前に確認する (削除基準は rules 正本)。不可逆な削除は人間ゲートへ
- 剪定の後に verify を通し、振る舞いが変わらないことを機械で確かめる
- 何を・なぜ剪定したかを来歴に残す (回帰捕捉と一貫)

## 理由

- 育てる力だけでは腐敗を防げず、最小コンテキストが主張倒れになる。刈り込む力を対で持たせる
- 盲目削除は事故を生む。owox の旗 (検証可能な正本) は剪定にも適用する。verify と削除基準を通せば、減らす操作も安全に機械化できる
- 既存の腐敗検知・削除基準・subagent に載せ、新サブシステムを作らない (品質保証と同じ方針)

## 捨てた案

- 剪定を人手だけに任せる
- 検出した不要を自動で即削除する
- 来歴・タスクの古いものを物理削除する

## 捨てた理由

- 人手任せは溜まりを防げず、結局腐る
- 自動即削除は盲目削除で事故る (削除基準・verify を飛ばす)
- 来歴の物理削除は履歴の価値を壊す。退避で足りる

## 見直し条件

- 腐敗検知の対象拡大が誤検出で作業を妨げた時
- 庭師 subagent の剪定提案が過剰・過小で運用に合わない時
- 退避方針 (superseded の読み経路外移動) が必要になる規模に達した時
