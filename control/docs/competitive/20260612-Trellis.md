# Trellis

## 調査日

2026-06-12

## 分類

直接ライバル。

target harness 相当を repo に置き、複数 AI CLI、仕様、タスク、記憶、作業流れを支える。

## 概要

Trellis は、AI 開発用の工程管理と記憶管理を repo 内に置く OSS。

中心は `.trellis/`。
ここに仕様、タスク、作業記録、個人作業領域を置く。
各 AI CLI 向けには `.claude/`、`.codex/`、`.cursor/` などへ生成物を置く。

## 何がすごいか

### 導入

`npm install` と `trellis init` で始められる。
初回導入時に AI CLI を選び、必要な設定を生成する。

### 作業流れ

計画、実装、確認、記録、仕様更新の流れが具体化済み。

主な流れ:

- 要望を task と `prd.md` にする
- 実装用の文脈を選ぶ
- 実装エージェントがコードを書く
- 確認エージェントが仕様、差分、lint、typecheck、test を見る
- 学びを `.trellis/spec/` へ戻す
- セッションを記録に残す

### 文脈管理

全部を読ませない。
task ごとに `implement.jsonl`、`check.jsonl`、`research.jsonl` で必要な仕様と調査だけを渡す。

これは owox-harness の「少ない文脈で正しく動く」と同じ方向。

### 役割分担

`trellis-research`、`trellis-implement`、`trellis-check` がある。
調査、実装、確認を分けるため、人間にも理解しやすい。

### チーム導入

共有仕様と個人作業領域を分ける。
新規参加者向けの初期 task もある。

### 資料

公式文書が厚い。
導入、作業流れ、構成、拡張、設定、雛形、予定表が揃っている。

## owox-harness が取り入れる点

- 初回導入後に立ち上げ task を作る
- 既存の `AGENTS.md`、`CLAUDE.md`、`.cursorrules`、`CONTRIBUTING.md` から規約を取り込む
- task ごとに実装用の文脈と確認用の文脈を分ける
- `continue` / `finish-work` 相当の少ない入口にする
- 難しい不具合解決後に再発防止を記録する skill を持つ
- 仕様雛形を配布できるようにする
- 新規参加者向けの導入 task を持つ
- 生成物と正本の差分、保護範囲、更新手順を明示する

## owox-harness が超える点

### 検証可能な正本

Trellis は良い作業流れが強い。
owox-harness は正本そのものを型と検証で守る。

### 完了3区別

Trellis は task 完了と確認を扱う。
owox-harness は作業完了、要件完了、検証完了を分ける。

### 来歴

Trellis は記録と task を持つ。
owox-harness は判断を ID、状態、根拠、参照つきで扱う。

### task 腐敗防止

Trellis の task は流れとして強い。
owox-harness は完了未検証、孤立、重複、放置、永久停止を機械検出する。

### 人間ゲート

Trellis は作業境界を運用で守る。
owox-harness は不可逆操作、正本昇格、外部送信、依存追加などを needs_human として構造化する。

### 経験分離

Trellis は学びを仕様へ昇格する。
owox-harness は汎用経験とドメイン固有情報を型で分け、export / import に人間ゲートを置く。

### MCP 主入口

Trellis 公開資料では MCP 主入口は確認できない。
owox-harness は MCP resource / tool / 共通返り値で、読み取り、操作、人間判断点を構造化する。

## owox-harness が負けている点

- 認知
- GitHub 星数
- 対応 AI CLI 数
- 導入の軽さ
- 公式文書量
- すぐ使える作業流れ
- 雛形配布
- チーム導入
- 実利用の見え方

## 勝つための判断

Trellis と同じ「便利な作業流れ」だけで競わない。

owox-harness の旗は、検証可能な正本。

最初に見せるべき実演は、Trellis でも便利に進むが、owox-harness だけが止められる失敗。

例:

- 依存追加理由なし
- 受け入れ基準と test の参照なし
- task の自己申告完了
- 対象外の変更混入
- 不可逆操作の未承認
- 正本と生成物の不一致
- ブランド用語違反
- 汎用経験へのドメイン固有情報混入

## 見直し条件

- Trellis が型付き正本を導入した時
- Trellis が完了3区別を導入した時
- Trellis が task 腐敗検知を導入した時
- Trellis が MCP 主入口で判断ゲートを持った時
- Trellis が経験 export / import とドメイン分離を持った時
- owox-harness が多 AI CLI 対応を前面に出したくなった時
