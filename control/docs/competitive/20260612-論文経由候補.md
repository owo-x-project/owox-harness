## 何がすごいか

- 仕様駆動や文脈管理の研究側比較に出てくる
- owox-harness の旗と近い可能性がある
- 実装前の仕様、既存 code からの仕様復元、作業流れ正本化に関係する
- GitHub Spec Kit、BMAD-METHOD、Trellis と同じ比較軸へ入る可能性がある

## 候補別メモ

### OpenSpec

仕様駆動候補。

見る点:

- 仕様をどう書くか
- 計画と実装へどうつなぐか
- 検証条件を持つか
- 正本が型や規則で守られるか

### Get Shit Done

文脈管理、作業流れ候補。

見る点:

- AI 開発の進行をどう管理するか
- 文脈をどう集めるか
- 完了条件をどう扱うか
- 人間確認を持つか

### Spec Kitty

仕様駆動候補。

見る点:

- 仕様の雛形
- 実装前の合意
- task 分解
- AI CLI 連携

### Reversa

文脈管理、仕様復元候補。

見る点:

- 既存 code から仕様を復元するか
- 復元仕様を検証できるか
- 仕様と実装のずれを検出するか
- owox-harness の既存 target repo 取り込みに使えるか

### Spec-Flow

仕様駆動、作業流れ候補。

見る点:

- 仕様から実装までの流れ
- 仕様変更の追跡
- 検証条件
- 複数 AI CLI への移植性

## owox-harness が取り入れる点

- 研究側の比較軸
- 仕様駆動の用語整理
- 既存 code から仕様を起こす観点
- 仕様変更と実装差分の追跡
- 検証条件を仕様に含める流れ

## owox-harness が超える点

### 一次情報確認後に判断

現時点では一次 URL 未確認。
owox-harness の設計判断へ直結させない。

### 正本生成

論文上の比較対象は、実装や導入体験が不明。
owox-harness は実際に target harness を生成し、検証する。

### 来歴

owox-harness は仕様、判断、task、検証結果を来歴へ結ぶ。
この軸で各候補を再確認する。

## owox-harness が負けている点

- 研究側で整理された比較軸
- 仕様復元の理論
- 既存 code からの仕様抽出
- 論文での評価

## 勝つための判断

論文経由候補は「勝つ相手」というより、評価軸の取り込み対象。

一次情報が見つかるまで、要件へ移さない。
ただし仕様復元、検証条件、仕様と実装のずれ検出は owox-harness の重要候補として残す。
