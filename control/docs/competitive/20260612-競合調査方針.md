# 競合調査方針

## 状態

採用済み。

## 目的

owox-harness の勝ち筋を保つため、競合と参考資料を継続調査する。

競合調査は、真似するためではなく、次を明確にするために置く。

- 何に負けているか
- 何を取り入れるか
- 何を超えるか
- owox-harness の旗が今も強いか

## 文書方針

### 配置

競合情報は `docs/competitive/` に置く。

採用判断に変わった時だけ `docs/decisions/` へ移す。
要件に変わった時だけ `docs/requirements/` へ移す。

### 調査票

競合ごとに 1 ファイル作る。

必須項目:

- 調査日
- 分類
- 何をしているものか
- すごい点
- owox-harness が取り入れる点
- owox-harness が超える点
- owox-harness が負けている点
- 見直し条件
- 参照

### 書き方

- 事実と推測を分ける
- 星数、対応数、版など変わる値は調査日つきで書く
- 長い引用を置かない
- 競合を貶さない
- owox-harness の旗へ必ず戻す

## 参照方針

### 優先順

1. 公式リポジトリ
2. 公式文書
3. 公式リリース
4. 公式課題、公式議論
5. 論文、第三者記事

第三者記事は発見と比較補助に使う。
断定の根拠にはしない。

### 記録

- 外部参照は裸 URL で書く
- 手元参照はパスで書く
- 取得日は調査日に含める
- 不確かなものは「要一次調査」と書く

### 再調査条件

- 競合が検証可能な正本へ寄った時
- 競合が MCP を主入口にした時
- 競合が型付き正本、来歴、検証ゲートを持った時
- 競合の利用者数、対応 AI CLI 数、配布形が大きく変わった時
- owox-harness の旗と競合の旗が重なった時

## 分類

### 直接ライバル

target harness 相当を repo に置き、複数 AI CLI や作業流れを支えるもの。

### 仕様駆動

要件、仕様、計画、実装をつなぐもの。

### 作業流れ

AI 開発の段階、役割、成果物を束ねるもの。

### 文脈管理

AI に必要な情報を小さく渡すもの。

### タスク管理

AI 作業のやること、状態、完了を扱うもの。

### 実行エージェント

コード変更、検証、PR 作成を行うもの。

owox-harness は実行エージェントそのものを作らない。
ただし、上位の制御対象として見る。

## 調査候補

候補一覧は `docs/competitive/20260612-競合候補一覧.md` を正本にする。

この文書では候補の置き方だけ決める。

### 最優先条件

- target harness 相当を repo に置く
- 仕様、計画、タスク、検証、文脈のいずれかを正本化する
- 複数 AI CLI または複数実行エージェントを束ねる
- MCP、skills、subagents、hooks、AGENTS.md 相当を扱う
- owox-harness の旗と競合する

### 低優先条件

- 実行エージェントそのもの
- 汎用 agent 基盤
- AI 向け文脈圧縮だけの道具
- すでに停止、終了、アーカイブ済みのもの

低優先でも、owox-harness の target harness 設計、task 設計、文脈設計、検証設計に影響する場合は候補に残す。

## owox-harness の比較軸

- 検証可能な正本があるか
- 正本が型で守られるか
- 人間判断が来歴に残るか
- 完了を3区別できるか
- task が検証なしに完了にならないか
- 腐敗した task を検出できるか
- 文脈が最小化されるか
- 生成物と正本が分離されるか
- 複数 AI CLI に移植できるか
- 経験が export / import できるか
- ドメイン固有情報が汎用経験へ混ざらないか
- 不可逆操作で人間ゲートが働くか

## 現時点の勝ち筋

Trellis、GitHub Spec Kit、BMAD-METHOD と同じ土俵で「便利な AI 開発作業流れ」を競うと負けやすい。

owox-harness は次に絞る。

- 検証可能な正本
- 型付き正本
- 来歴
- 完了3区別
- task 腐敗検知
- 人間ゲート
- 経験の分離と移植
