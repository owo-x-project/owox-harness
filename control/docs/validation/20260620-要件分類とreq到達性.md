## 根本原因

床ルーティング (crates/core/src/hook.rs render_routing) が要件登録の要望を req スキルでなく
requirement.create ツールへ直接飛ばしていた。req スキルは明示起動のみで本文 (方法論) が読み込まれず、
エージェントは方法論を通らず生ツールを叩いた。床は axes を注入済みだが、行動の地点 (ツール) に
振る舞いが縛られていないため無視された。生成物・要件は正しいが、エージェントが実際に通る経路から
到達不能だった (メモリ tool-discoverability-driven-by-floor-routing と同型)。

## 是正1 (3点)

- 床ルーティングの要件行を req スキルへ向け、非交渉事項 (kind 分類・制約は来歴へ・ideal-first は
  優先度を人間へ) を1行で行動の地点に置いた (hook.rs)
- requirement.create のツール説明へ同じ非交渉事項を移した (mcp serve.rs)
- 機械ガード: ideal-first で AI が create に priority を付けたら弾く (mcp serve.rs ai_priority_blocked)。
  限界は create のみ・update は人間判断の正規チャネルとして許す (完全な人間ゲートは将来)

- ✓ 技術前提 (既存ストレージ層を使う) を decision.record (adopted) へ分離・受け入れ基準にしない
- ✓ ideal-first で priority を create に付けず・最後に「優先順位を付ける」を人間向け次手として提示
- ✓ requirement.create でツール経由・別途文書なし

エージェントがスキルを実起動しないため発火しない (床へ直書きした非交渉事項だけが効く同じ構図)。

## 是正2 (PRFAQ を行動の地点へ直書き)

  lightweight/incremental では出さず床を最小に保つ

## 3回目の結果 (是正2後・語りは依然不発・ゲート本質は成立)

同じプロンプトで再検証:

- ✗ PRFAQ 起草はまだ不発。床へ直書きしてもプレスリリース風要約も着手前の人間確認もしなかった
- ✓ 中核は3回連続維持: 種類分割・技術前提を decision へ・priority を create で付けず・ツール経由
- ✓ 毎回 status=draft で作成し accepted へ上げず、実装にも入らず次手を人間へ提示

発見: 床へ直書きした「種類・制約・優先度」は効くのに「止まって人間へ確認」だけ効かない。前者は AI が
止まらず自分でできる指示、後者は止まって問い返事を待つ指示で、人間が十分な指示を出した状況では AI が
飛ばす。発見性 (到達) の問題でなく、散文では「止まって人間ゲートを通れ」を強制できないという別種の壁。

## 結論 (検証成功として閉じる)

Phase9 決定が定める通り PRFAQ の荷重部分は「人間が what/why を承認するまで固めない」人間ゲートであり、
プレスリリースの語りではない。そのゲートは実際に効いている — 3回とも AI は draft で作り accepted へ上げず
実装に入らず人間へ次手を渡した。固める判断は人間に残っている。不発なのは語りの体裁だけ。

判断 (人間決定): 語りの体裁を散文で追うのは低価値かつ drift を招くため追わない。床の PRFAQ 合図は
advisory として残し、強制しない。AI の自己 accepted は3回とも未観測のため機械ゲートは足さない
(投機的機構を避ける owox 方針)。種類分類・PRFAQ 正本一本化・優先度ガードは実機成立とみなし閉じる。

見直し: prfaq で AI が自分で accepted へ上げる事象を観測したら、accepted 昇格を人間ゲート化する。

## 追補: PRFAQ をツール契約へ昇格 (再開)

上の閉じ方だと prfaq 設定が lightweight と行動が変わらず設定の意味がない、と人間が再提起。PRFAQ の
逆算の蒸留 (誰が・なぜ便益を得るか) は「AI が自分でできる」obligation ゆえツールで強制できる、と再判断
(`docs/decisions/20260620-要件分類とPRFAQ正本.md` 追補)。requirement.create に benefit を足し、
requirements-shape=prfaq で欠落を弾く機械ゲート (prfaq_benefit_missing) を入れた。benefit は来歴へ
記録し要件へ link する。

機械確証: cargo 322 passed・clippy 0。prfaq_benefit_missing を単体確認 (prfaq+便益なし→弾く・
便益ありや lightweight や profile 未解決は素通り)。create の benefit→採用済み decision 記録と要件への
link を単体確認。

### 4回目の結果 (benefit ゲート後・逆算の蒸留が実機成立)

新ビルドを `/usr/local/bin/owox` へ入れ target を `owox setup` で再生成。同じ統制プロンプト
(機能=入力履歴一覧 + 非機能=0.5秒以内 + 技術前提=既存ストレージ層を使う・「要件として残して」) で検証。

- ◎ benefit ゲートが逆算の蒸留を引き出した。requirement.create 2件とも benefit を自分で埋めた
  (「利用者が過去の入力をすぐ確認して再利用でき再入力の手間を減らせる」等)。弾かれて再試行でなく
  初手で充足。床文言+ツール説明が誘導し、ゲートは背後の保険として働いた
- ◎ 各 benefit が採用済み decision として来歴へ残り、要件へ双方向 link (要件の `## Links - decision`・
  decision の `## Links - requirement` を実ファイルで確認)。3回目まで空だった便益・なぜの来歴が埋まった
- ✓ 中核は4回連続維持: 機能/非機能で分割タグ・技術前提 (既存ストレージ層) を adopted な decision へ分離・
  ideal-first で priority 自己設定せず・status=draft 据え置きで accepted へ上げず・verify.run まで走り
  needs_human を受けた
- ✗ 兆候なし: benefit の空文字素通り・でっち上げ・ゲート回避は起きず

結論: PRFAQ がツール契約への昇格で初めて lightweight と異なる強制された行動差を生んだ
(prfaq では便益・なぜを述べないと要件が作れず、それが来歴へ残る)。同型機構 (ai_priority_blocked) の
実機成立に続き、benefit ゲートも実機成立として閉じる。
