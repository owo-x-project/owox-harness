# Phase5 スキルと入口 検証

## 対象

Phase5 (スキルライフサイクル・入口=薄い skill・setup・glossary 動的化・canon 直読み禁止)。
owox serve の skill.* / glossary.*、4 hook (session-start / pre-tool-use / user-prompt-submit / stop)、生成物 (AGENTS.md・`.codex/{config.toml,hooks.json}`・`.agents/skills/`)。

## 方法

release 版 owox を `/usr/local/bin/owox` へ配置済み。target repo (`/workspace/product/target`、Codex 0.139.0 のサンドボックス) で 3 系統を確認した。

- hook 単体: 各 hook へ入力 JSON を流し stdout を確認
- MCP stdio: `owox serve` へ JSON-RPC を流し skill ライフサイクル・glossary tool を通し実行
- 実機 Codex: `codex exec`(非対話)を target で起動し、hook 発火・session 注入・MCP 連携・直読み拒否を実測

検証のため育成 skill 2 個を `.owox/skills/` へ配置 (tidy: テスト合格 / flaky: テスト失敗)。

## 確認できたこと (hook 単体)

- session-start: Vision・phase=initial の振る舞い案内・glossary 用語名の列挙・「canon を直読みするな / glossary.lookup を使え」・owox://context / owox://next の案内を注入
- pre-tool-use: `cat .owox/glossary.md` を deny。理由に「canon を直読みするな、session 文脈と owox:// と glossary.lookup から届く、skills は読み書き可」
- pre-tool-use: `cat .owox/skills/foo/SKILL.md` は素通り (exit 0)。著作の作業域は通る
- user-prompt-submit: 「canon を編集したい」で canon の定義だけを注入 (出た語のみ・段階的開示)

## 確認できたこと (MCP stdio・skill ライフサイクル)

- skill.list → flaky=draft/failing、tidy=registered/passing の 2 軸を返す
- skill.register flaky → failed (tests failed)。`.agents/skills/flaky/` は生成されない
- skill.register tidy → ok。`.agents/skills/tidy/{SKILL.md,agents/openai.yaml}` を生成
- skill.promote tidy → ok。来歴 `20260613-Promote-skill-tidy` を adopted で残し、生成し直し openai.yaml が `allow_implicit_invocation: true` へ。owox.toml に `promoted = true` を永続化
- skill.list (昇格後) → tidy=promoted
- skill.remember tidy → `.owox/skills/tidy/memory.md` へ日付付き追記。生成物へは出ない
- 標準入口 8 個 (kickoff / next / status / decide / verify / review / task / handoff) が `.agents/skills/` に生成。全て `allow_implicit_invocation: false`(明示起動のみ・勝手に発火しない)

## 確認できたこと (MCP stdio・glossary)

- glossary.lookup canon → 定義を返す (found=true)
- glossary.add "build pipeline" → ok。canon の glossary.md へ追記、来歴を adopted で残す (AI 可)
- glossary.lookup "build pipeline" → 追加した語が引ける
- glossary.update greeting → needs_human。open gate (kind=brand) を立て、canon は変えない
- glossary.remove "smoke check" → needs_human。open gate を立て、canon は変えない
- canon 確認: add のみ反映、update/remove は不変。来歴は add=adopted・update/remove=open

## glossary 動的化

- glossary.add 後に session-start を再実行すると、注入される用語名一覧に "build pipeline" が増える。AGENTS.md への静的列挙でなく、その時の canon を hook が読む

## 実機 Codex (codex exec, 2026-06-13)

target で `codex exec` を起動して実測。

繋ぎ込み (codex doctor / mcp list で確認):

- codex は **プロジェクト直下の `.codex/config.toml` の mcp_servers を読む**。`codex mcp list` で owox (command=owox, args=serve) が enabled
- codex は **プロジェクト直下の `.codex/hooks.json` を読む**。`~/.codex/config.toml` の `[hooks.state]` に target の session_start / pre_tool_use / stop / user_prompt_submit が trusted_hash 付きで登録 (信頼済み=発火する)
- グローバル `~/.codex/config.toml` に owox MCP の記載は無い。検出された 1 MCP は target のプロジェクト設定由来
- 0.139 の feature flags に hooks は有効。skills 専用の有効化フラグは無く、skill 系は既定で有効 (実験フラグ不要)

対話 1 ターンで実測できたこと:

- SessionStart hook 発火。モデルがファイルを読まずに Vision を正確に復唱
- glossary 用語名 5 個 (canon / target harness / greeting / smoke check / build pipeline) をモデルが列挙。動的追加の build pipeline も実機 session 注入に乗った
- UserPromptSubmit hook 発火
- canon 直読み: `cat .owox/glossary.md` の指示に対し、モデルが AGENTS.md の指針を理由に自分で拒否し実行しなかった
- MCP 連携: モデルが owox の tool (glossary.lookup / skill.list / verify.run) を発見し呼び出そうとした (MCP サーバ接続・tools/list は機能)
- 入口 skill: モデルが `.agents/skills/verify/SKILL.md` を読み、その手順どおり verify.run を呼んだ (生成 skill を codex が発見・内容に従う)
- Stop hook: 完了をブロックし verify へ誘導、その後の再 Stop で完了。継続が実機で効く

## 詰まり (codex exec の制約・owox 側の欠陥ではない)

- 実機での MCP tool 呼び出しが全て `user cancelled MCP tool call` で失敗
- 原因: `codex exec`(非対話)は approval を常に never に固定 (`-a` / `-c approval_policy` で上書き不可)。codex は MCP tool 呼び出しに承認を要し、承認できない非対話では即キャンセルする
- 切り分け: 同じ tool を MCP stdio へ直接流すと全て正答を返す (上記)。tool 実装は正常で、詰まりは codex 非対話の承認ゲートが原因
- 帰結: 「実機 Codex を通した MCP tool の完走」は対話 Codex (人間が 1 回承認する) が要る。headless では `--dangerously-bypass-approvals-and-sandbox` でしか通せない

## 未確認・残り (codex exec 段階。下の対話 Codex 検証で全て解消)

- 対話 Codex (人間操作) での MCP tool 完走 → 解消 (承認無しで完走)
- `/skills` の一覧表示と `$<skill>` 起動 → 解消 (表示・起動とも確認)
- 昇格後の暗黙起動 → 解消 (自動発火を確認)
- target の汚れ: 検証で生成した skill・glossary・来歴・生成物。要 restore

## 結論 (codex exec 段階)

hook・MCP tool・生成物・glossary 動的化・skill ライフサイクル・canon 直読み禁止は、単体と MCP stdio で全て期待どおり。
実機 Codex でも hook 発火・session 注入・MCP 接続・skill 発見・Stop 継続・直読み拒否を確認。
残るは対話 Codex でしか観られない 3 点 (MCP 完走・/skills 表示・$起動・暗黙起動)。これらは人間の対話で詰める。

## 対話 Codex 検証 (人間操作, 2026-06-13)

対話 Codex で残り 3 点を確認。最初に粗が出たため是正実装を入れ (`docs/decisions/20260613-Phase5-実機検証の是正.md`)、再検証した。

是正前に見つけた粗:

- `$next` が機能せず。入口 skill 本文「Read owox://next」に対しモデルが resource を取りに来ず (read_mcp_resource を使わず) canon 直読みへ退避し deny され迷走。未決も ready も返せない
- `$verify` が verify.run を呼ばず生 `test.sh` を直接実行 (owox の検証層を通らない)
- 生成 `agents/openai.yaml` が Codex 正式スキーマと不一致 (`allow_implicit_invocation` がトップレベル・`interface` 欠落)
- 生成物が英語固定でモデルが英語応答

是正後の再検証 (全て期待どおり):

- MCP tool 完走: 対話 Codex は承認プロンプト無しで `owox.glossary.lookup` / `skill.list` が完走 (codex exec の `user cancelled` は非対話固有と確証)

繋ぎ込みの確認 (codex doctor / mcp list):

- codex はプロジェクト直下の `.codex/{config.toml,hooks.json}` を読む (owox MCP=enabled・4 hook=trusted)

## 対話検証で見つけた残り粗 (改善候補)

- Stop block のノイズ: 作業ツリーに未コミット変更があると、AI がそのターンで何も触らなくても毎回 Stop が verify/記録を促す。Stop hook はターン差分でなく git のツリー状態を見るため (安全側だが複数ターン作業でうるさい)。実運用ではコミットして終えるので鳴り止む
- skill ライフサイクルの入口コマンド欠落: register / promote / list は tool だけで人間が叩く入口 (薄い skill) が無い。そのため「skill を登録して」が codex 内蔵 skill-creator の暗黙起動に化け、owox の skill.register に届かない。明示 tool 指定で回避できるが、入口を足すのが筋 ($verify と同じ「入口が無いと逸れる」構図)

## 結論

Phase5 の主機能 (読み tool・入口 skill・glossary 動的化・skill ライフサイクル・昇格と暗黙起動・canon 直読み禁止・応答言語) は是正後の対話 Codex で全て期待どおり。残り粗は 2 件 (Stop ノイズ・skill 入口欠落) で致命的でない。
