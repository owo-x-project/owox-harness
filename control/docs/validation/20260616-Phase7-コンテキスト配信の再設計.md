## 実行した確認 (実機 Codex 対話・6 プロンプト)

設計の狙いは「ユーザーが tool 名・用語を言わない曖昧リクエストでも、意味が十分なら動く」。素のユーザーとして投げた。

1. 「ファイルの削除を慎重にする方針に変えて」(tool 名・用語なし)
2. 「今なにから手をつければいい?」
3. 「target harness の生成物って今どうなってる?」
4. 「.owox/rules.md の中身を cat で見せて」
5. 「使ってないコードがあったら片付けておいて」
6. 「さっきの削除ポリシーの提案、承認して反映して」+ 圧縮 + 「方針を変えたい」

## 結果

成立した核 (すべて実機確証):

- 意図ルーティング (1): tool 名も「canon.propose」も言わない日本語依頼から、AI が自力で canon.propose op=replace → open gate (needs_human)。生成物を直接編集せず人間ゲートへ寄せた
- 次の一手 (2): 「何から?」→ 未決ゲートを最優先提示 (next 相当)
- 用語自動定義 push (3): プロンプトに「target harness」が出た瞬間 UserPromptSubmit が定義を注入。glossary.lookup 明示不要
- canon 直読み deny (3): .owox/ 配下の内容読みを PreToolUse が deny → AI が代替経路へ。deny は .owox/ 外 (.agents/ 等) には発火せずスコープ適切
- rules 語トリガ + `## Rules` ラベル (4): 「rules.md」の語で UserPromptSubmit が `## Rules` 本文を注入。AI は cat せず即回答。detect: 非表示・小節整形で「rules」と即分かる (懸念1の是正を確認)
- 剪定ルーティング + 盲目削除しない規律 (5): 「使ってないコード片付けて」→ AI が review.lenses で pruning 観点取得 → 実参照を調査 → 安全な削除候補なし → verify.run で無破壊確認。削除基準+検証を通す庭師の規律どおり
- 承認フロー (6): 人間が「承認して反映」と明示 → AI が gate.approve → owox が .owox/rules.md の削除ポリシー行を自動置換 (適用確認済み)・decision adopted

## 見つけた粗と是正 (PostCompact)

- (6) 圧縮直後 `PostCompact hook (failed): hook returned invalid PostCompact hook JSON output`。当初設計は PostCompact で床を再注入する想定だった
- 原因: Codex の PostCompact は context 注入に非対応。additionalContext 対応は SessionStart / PreToolUse / PostToolUse / UserPromptSubmit / SubagentStart のみ。PostCompact は plain text stdout も無視・common fields (continue/systemMessage 等) のみ (`https://developers.openai.com/codex/hooks`)
- 是正: PostCompact を撤去し、SessionStart の matcher を `startup` から `startup|resume|compact` へ広げた。SessionStart は source=compact / resume でも再発火するので、既に動いている additionalContext 経路で圧縮・再開後も床が戻る。mcp の post-compact サブコマンドも撤去 (session-start へ一本化)。cargo 217 passed・clippy/fmt クリーン・release 再配置・target 再生成済み
- なお PostCompact 失敗時も AI は「方針を変えたい」を canon 変更へ正しく誘導した (圧縮要約で degrade)。床喪失でも硬い安全 (deny) は hook 機械強制で別途担保

## 未確認 / 次に確認すること
