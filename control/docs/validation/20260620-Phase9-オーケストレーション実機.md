## 失敗と再現手順

### 生成した custom agent_type を Codex が認識しない

handoff の最優先確認項目「カスタム agent_type で生成 TOML を親が spawn できるか」。結論=Codex 0.141 stable では不可。

再現:

- multi_agent feature は stable かつ有効 (`codex features list`)
- サンドボックスに `.codex/agents/<役割>.toml` を生成し、別途 `~/.codex/agents/<名>.toml` も置いた
- `codex exec` (read-only) で「spawn tool の agent_type を列挙」させると、返るのは組込みの `default` / `explorer` / `worker` のみ。生成5役割も `.codex/agents/`・`~/.codex/agents/` に置いた custom 定義も一切現れない
- Codex の spawn は `spawn_agents_on_csv` (CSV 各行に worker サブエージェントを1体 spawn する固定機構)。worker は固定 agent_type で、ファイル定義の custom 型は stable 経路に載らない。custom 定義の経路は `multi_agent_v2` (under development・既定 false) の可能性

未完: approval bypass を伴う実 spawn 完走は安全上の理由で未実行 (分類器が拒否)。よって「組込み型に prompt/developer_instructions を注入して責務を与える」フォールバックの実機完走確認は次回の対話検証へ残す。

## 修正案

設計 (`docs/decisions/20260620-オーケストレーション具体化.md` 観点2) が用意した安全弁どおり、Codex 写像を「custom TOML 主体」から「フォールバック = 組込み agent_type (worker/explorer 等) + spawn 時 prompt/developer_instructions 注入」へ切り替える。具体:

- `.codex/agents/<役割>.toml` 生成は Codex 0.141 stable では消費されないため、生成停止か `multi_agent_v2` ゲートにする
- 役割の責務・文脈・権限は spawn 時に親が prompt へ注入する経路を床提示で能動化する (床の `## Orchestration` は既に役割定義を持つので、親が組込み型へ写して spawn する案内を足す)
- agents.toml 正本 (役割×変種・ティア・sandbox・許可層) は CLI 非依存の真実源として保持し、Codex 写像の右辺だけ差し替える

これは大きさが「設計判断 + codex.rs 写像の再実装」なので、人間ゲート (どちらに倒すか) を経てから impl する。

### 粗 (要是正): シェル書き込みが層の編集ゲートを素通りする

最重要の発見。owox の機械強制は subagent 内でも発火する (サブの `git tag -d probe-tag` が PreToolUse hook で blocked) が、**guarded 層・契約面への書き込みが subagent 経由で素通りした**。

再現と切り分け (codex ログ DB `~/.codex/logs_2.sqlite` のツール呼出で確証):

- codex の worker (subagent) は apply_patch/Edit/Write を持たず、読み書き全てを `exec_command` (PTY シェル) で行う。codex はそのシェル呼出を tool_name=Bash として PreToolUse hook へ渡す
- 親 vs サブの同条件比較 (契約面 `src/core/ports/engine_port.sh` への 1 行追加と `git tag -d probe-tag`):
  - 親が契約面を編集 (apply_patch 経路) → deny (層ゲートが「contract surface that needs approval to edit」)
  - サブが契約面を編集 (`sh -c 'printf "..." >> src/core/ports/engine_port.sh'`) → **素通り** (deny なし・ファイルに追記成立)
  - サブの `git tag -d probe-tag` → deny (不可逆検出が発火)
- 根本原因 (`crates/core/src/hook.rs`): Bash 経路の判定が書き込みを見ていない。`pre_tool_use_decision` は不可逆正規表現と canon 直読みのみ照合。`layer_pre_action_gate` の Bash 分岐は `rm_targets` (削除) しか集めず、`>` `>>` `tee` `sed -i` `cp`/`mv` 等のシェル書き込みを編集として検出しない。apply_patch/Edit/Write のツール入力だけを編集として層照合するため、シェルで書く subagent の編集は guarded/契約面ゲートを丸ごと回避する
- 影響範囲: 不可逆 (コマンド正規表現) と削除 (rm) の保護は subagent でも効く。**抜けるのは「書き込み (新規・追記・上書き) による guarded 層削除以外・契約面編集」**。subagent は編集を必ずシェルで行うため、subagent の guarded 書き込みは常に素通りする。親も `printf >> guarded` のようにシェルで書けば同様に抜ける (ただし親は通常 apply_patch を使うので顕在化しにくい)

### 是正 (実装済)

人間ゲートを経て impl 済 (`docs/decisions/20260620-Phase9-シェル書き込みの層ゲート被覆.md`)。`crates/core/src/hook.rs` に `write_targets` を足し、Bash 経路でリダイレクト (`>`/`>>`・fd 付き・`&>`)・`tee`・`sed -i`・`cp`/`mv` の宛先・`dd of=` を編集として抽出し、apply_patch 編集と同じ `is_contract_surface` 照合へ載せた。契約面書き込みは deny・guarded 非契約面と free は素通り (apply_patch の rule4 と対称)。過剰検出は層 glob に当たらず無害なので取りこぼしより過剰側へ倒す近似。cargo 342 passed・clippy 0。実機 hook バイナリで subagent の実コマンド (`sh -c 'printf ... >> <契約面>'`) が deny へ変わり、guarded 非契約面・free 書き込みは素通り維持を確認。

対話 Codex TUI で end-to-end も確認済 (同セッション)。サブエージェント1体に契約面へのシェル追記 (`printf '\n# probe\n' >> src/core/ports/engine_port.sh`) を試させると、是正前は素通りだったのが **subagent 内で PreToolUse hook が deny** (「contract surface that needs approval to edit」) を返した。同サブの `git tag -d probe-tag` も引き続き deny で回帰なし。これで「subagent の主編集経路 (シェル) でも層ゲートが効く」が実機成立。
