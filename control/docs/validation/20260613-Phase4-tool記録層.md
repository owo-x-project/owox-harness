# Phase 4 tool 層・記録層 検証

## 対象

owox serve の tool 層 (decision.record / gate.list / gate.approve / verify.run) ・記録層 (`.owox/decisions/`) ・計算 resource (owox://next) ・共通返り値 (封筒)。

## 方法

release 版 owox を `/usr/local/bin/owox` へ配置。target repo (`/workspace/product/target`、Codex 0.139.0 の検証サンドボックス) で `owox serve .` を起こし、stdio へ JSON-RPC を流して確認した (initialize → notifications/initialized → 各 method)。

## 確認できたこと (実機 stdio)

### 能力告知・一覧

- initialize の capabilities に resources と tools の両方が出る
- get_info の instructions に owox://context / owox://next / decision.record / verify.run の使い方が出る (Codex は初期化時にこれを読む)
- tools/list が 4 tool を返す。tool 名のドット (decision.record 等) は MCP 仕様で許容され、そのまま通る
- 引数 schema が JSON Schema で出る (title/status 必須、links は入れ子オブジェクト)
- resources/list が owox://context と owox://next を返す

### tool 呼び出し (封筒で統一)

- decision.record status=adopted → ok。`.owox/decisions/20260613-Use-rmcp-for-MCP-server.md` を生成。data.id と decision_ids を返す
- decision.record status=open → ok。reason に「未決の人間ゲート」と明示
- gate.list → ok。open の来歴だけを pending に列挙 (adopted は出ない)
- gate.approve { id, note } → ok。来歴を adopted へ遷移し、Approval に「note — 日付」を追記。ファイル再読込で承認が残ることを確認
- verify.run:
  - 検査未設定 → needs_human + gate (completion-judgment)。completion は work/requirement/verification とも needs_human
  - 検査に失敗を含む → failed。verification=failed、results に各検査の passed/detail
  - 検査が全通過 → needs_human + gate。verification=passed、work/requirement は needs_human。next_actions で人間確認を促す
- 「verify が要件完了を人間判断 (needs_human) として返す」を実機で確認 (完了3区別)

### 計算 resource

- resources/read owox://next → status=open の来歴を Markdown で列挙。open が無ければ「Nothing is waiting」を返す
- ready タスクは task 層導入後 (本スライス未実装)

### 記録ファイル

- decision は `# title` + `## Status/Rationale/Links/Supersedes/Approval` の Markdown
- 描画 → 再解釈の往復が一致 (gate.approve が読み戻して書き直せる)
- 同日同 slug は末尾連番 (`-2`) で一意化 (単体テストで確認)

## 単体テスト

core 48 + mcp 2 = 50 通過。記録層 (往復・slug・gate 遷移)、verify (3 状態)、封筒、日付変換 (civil_from_days の既知日付・閏日) を含む。

## 設計判断

- 封筒は CallToolResult のテキスト (pretty JSON) で返す。structured_content は Codex の対応が不確実なため使わない (`docs/decisions/20260613-Phase4-tool記録層.md`)
- 日付は mcp が UTC で求め core へ渡す (core は時計を読まず決定論)。新規依存を足さず std から civil-date 変換
- 検査コマンドは config.toml の `[[verify.checks]]` に人間が明示列挙。target の既存 config に足してもマージ・読込とも壊れないことを確認

## commit/stop の機械ゲート化 (追加検証 2026-06-13)

`owox hook pre-tool-use` / `stop` へ PreToolUse / Stop の入力 JSON を流して確認 (target repo)。

- commit + 検査未設定 → 警告 (deny でない。機械検証できないと案内)
- commit + 失敗検査あり → deny。失敗した検査名を理由に含む (検査をその場で再実行している)
- commit + 全通過 + 未承認 open gate → 警告 (件数を案内)
- commit + 全通過 + open gate 無し → 素通り (出力無し・exit 0)
- commit でない不可逆 (git push --force) → 従来どおり deny (最優先)
- stop + open gate 無し → 継続 (checklist)
- stop + open gate あり → 継続 + 未承認件数を表面化
- stop + 既継続 (stop_hook_active) → 受理 (ループ防止)

「AI が verify をサボっても commit 時に検査が走り、落ちれば止まる」歯を実機で確認。検査再実行方式 (状態を持たず古い結果で通さない) を採用 (`docs/decisions/20260613-Phase4-tool記録層.md`)。

## state 適応 (追加検証 2026-06-13)

- state.set { phase: maintenance } → ok。`.owox/state.toml` に `phase = "maintenance"` を書き、変更を adopted 来歴へ残す
- hook: 保守 + 未承認 open gate + (検査未設定) で git commit → deny (initial/stable では警告だったものが maintenance で block へ格上げ)。verify 失敗は phase 不問で deny を維持

## phase が振る舞い案内を変える (追加検証 2026-06-13)

- session 開始 hook が phase 向けの振る舞い案内を注入: initial → 大規模・構造的改修を歓迎/推奨、maintenance → 変更を小さく可逆に・回帰テスト。実機で initial / maintenance の出し分けを確認
- owox://state を撤去。resources/list は owox://context と owox://next のみ。phase は session push とゲートで効き、人間は state.toml を直読みする

## 検証可能タスク層 (追加検証 2026-06-13)

MCP stdio で task tool を通しで確認 (target repo、検査 `true` を 1 件設定)。

- task.create A・B (B は blocks A) → ok
- task.list ready=true → A だけ (B は前提 A 未 done で ready でない)
- task.close A → 検査通過で done。task.list ready=true → B が ready に変わる (前提解決)
- resources/read owox://next → ready タスクに B を列挙 (未決の来歴 + ready タスクの統合)
- task.drop B { reason } → dropped。破棄理由を adopted 来歴へ残し、task に decision link。来歴ファイルに理由が残ることを確認
- task.close 検査失敗 → failed (閉じれない)、検査未設定 → needs_human (単体テストで確認)

「done に検証必須・drop は理由を来歴へ・ready は前提 done」を実機で確認。

## 対話検証の手順と記録欄 (人間用)

stdio・hook 単体は確認済み。残るは「実機 Codex の対話で AI が tool を自発的に使い、記録・検証・gate・タスクが自然に回るか」。これは人間が対話して体感する。

準備 (済):

- owox を `/usr/local/bin/owox` へ配置
- target `/workspace/product/target` を再生成 (AGENTS.md / .codex/hooks.json / .codex/config.toml)
- `.owox/config.toml` に検査 1 件 (`canon exists`)。block を見たい時は command を `false` に変える

確認シナリオ (Codex を target で起動して対話):

- session 開始: Vision・現在 phase の振る舞い案内・owox://context / owox://next の案内が AI に届くか
- ナビ: 「何を読めばいい?」で AI が owox://context を取りに来るか
- 用語 push: glossary の用語をプロンプトに出すと定義が注入されるか
- 判断記録: 設計判断をさせ、AI が decision.record を呼ぶか。open にすると owox://next に出るか
- verify: 「検証して」で verify.run を呼び、完了3区別が返るか
- commit ゲート: 検査を `false` にして commit させ、deny で止まり AI が理由を見て直すか
- 人間ゲート: verify.run / task.close の needs_human を AI が人間へ提示して止まるか
- タスク: 作業計画をさせ task.create するか。task.close が検査を通すか
- 状態適応: state.set maintenance 後、振る舞い案内が保守寄りに変わり、未承認 open gate が commit を block するか
- 不可逆: `git push --force` 等を deny するか

## 対話検証結果 (実機 Codex, 2026-06-13)

target で Codex を起動し S0〜S10 を対話実行。`docs/handoff/20260613-Phase4対話検証ガイド.md` の手順。

発火・自発利用 (各 hook / tool):

- SessionStart hook: Vision・phase=initial の振る舞い案内・読む先 (owox://context / owox://next) を注入。確認
- UserPromptSubmit hook: プロンプトに出た用語 (greeting / smoke check) の定義を自動注入。確認
- PreToolUse hook (用語 push): 編集ごとに発火し、編集内容に応じ違う用語を注入 (greet.sh 編集前に greeting、test.sh 編集前に smoke check)。確認
- PreToolUse hook (機械ゲート): commit を検査失敗で deny、検査通過で素通り。maintenance + 未承認 open で deny へ格上げ。git push --force / git tag -d を deny。確認
- Stop hook: 「変更したら verify・記録」案内。未承認 open 件数を表面化し、承認後は消える。確認
- tool 自発利用: verify.run・decision.record (open/adopted)・gate.list・gate.approve・task.create/list/close/drop・state.set を AI が自発で呼んだ。確認

期待どおり動いた点:

- S0 文脈注入・S1 ナビ (owox://context のスコープ再現)・S2 段階的開示 (両用語注入し探さず使用)
- S3 検証の完了3区別 (verification=passed / work・requirement=needs_human) を人間へ提示
- S4 編集時の用語注入 + 編集後 AI が自発で verify.run
- S5 commit ゲート両面 (失敗で deny→AI が理由を見て修正 / 通過で素通り)
- S6 人間ゲート三段 (open 記録→owox://next・gate.list に出る→approve で adopted・承認注記)
- S8 タスク破棄 (理由必須・破棄理由を来歴へ)
- S9 maintenance 格上げ (未承認 open で commit deny。initial では警告だった)
- S10 不可逆 deny (AI が先回り拒否。口頭で実行を迫っても hook が上書きし deny)

動かなかった・直す点 (次スライス候補):

- dep target 検証なし: task.create が存在しない target を弾かず literal 受理 ("TASK1" プレースホルダがそのまま入った)。AI は drop→再作成で自己修復。不明 target を reject すべき
- state.set の発見性: phase 変更時 AI が canon 編集と誤解し context.md / config.toml / rules.md / brand.md / AGENTS.md を延々探索してから state.set に到達。phase 変更は state.set という誘導が不足
- maintenance の振る舞い案内変化は同一セッションで未実見 (SessionStart 注入のため再起動要)。別途 target 再起動で確認したい

人間が見つけた問題点 (対話検証中):

- Stop hook が変更なしでも毎回 block する。変更ゼロを検知して素通り (出力なし) にすべき。AI 側の「変更なしなので verify 不要」補足もそれで不要になる (機構)
- 用語注入が同セッションで同語を反復 (greeting を S1 / S2 / S4 で毎回)。session 単位で既出語を覚え再注入しない仕組みが要る。hook は毎回ステートレスなので注入済み語の session マーカーが必要 (機構)
- verify 失敗時など一時的・作業状態の判断まで decision に残ると記録が乱立し検索性低下・コンテキスト消費増。decision は耐久性ある設計・方針判断に限るべき。Stop hook の「significant な判断を記録」の significant が曖昧で過剰記録を招いた。significant の定義明確化 or 一時メモは task note へ逃がす誘導が要る (誘導)
- state.set への到達でエージェントが迷う (上記 state.set の発見性と同件)
