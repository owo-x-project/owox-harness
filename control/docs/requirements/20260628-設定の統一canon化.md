# 設定の統一 canon 化

## 状態

実装中。設計の真実源は `docs/decisions/20260628-設定の統一canon化.md`。

## 背景

prose 正本は canon 経由で AI が変更できるが、TOML 設定（config / quality / release / agents）と context.md は AI が書けず手動編集しかない。穴を埋め、全設定を canon 窓口から扱えるようにする。

## 目的

- 全設定項目を AI が canon 窓口から読み書きできる
- 設定値の変更は全て人間ゲートを通す
- 窓口 tool を増やさない（canon.add / canon.propose / canon.revert に寄せる）

## 対象

- `crates/core/src/canon.rs`: backend 抽象、structured backend、context backend
- `crates/core/src/model.rs` ほか設定型: 構造化エントリの照合と TOML 書き戻し
- `crates/mcp/src/serve.rs`: canon.add / canon.propose の target 拡張と引数

## 対象外

- 新規の設定書込 tool（config.set 等）
- prose backend の既存挙動の変更
- tool の mode 統合・description 変更
- gate.approve 連携の骨格変更

## 仕様

### backend 分岐

canon の窓口 tool は据え置き、target で backend を選ぶ。

- prose backend: brand / rules / practices / glossary / context.md。追加即時・変更削除は人間ゲート（既存維持）
- structured backend: config / quality / release / agents。全操作が人間ゲート（canon.propose のみ）

### structured backend の照合

変更削除はエントリの識別子で既存 TOML の対象を特定する。

- verify.checks は name
- quality.layers は name
- quality.boundaries は paths
- targets は cli 名
- release.artifacts は path
- 単一値（language / release.version など）はキー直指定

### 書き戻し

既存 TOML を読み、構造体を操作し、書き戻す。スキーマ検証を通してから書く。コメント・キー順序の保持は見直し条件（決定文書）に従う。

## 実現方針

- prose backend は既存の add_to_prose / append_under_heading をそのまま使う
- structured backend は target ごとに「追加エントリのスキーマ」「識別子フィールド」「TOML 読み書き」を持つ
- canon.add は structured target を弾き canon.propose へ案内する
- canon.propose は structured 変更を open gate にし、gate.approve が apply_pending_canon_change で書き戻す（prose と同じ承認経路）

## 完了条件

- config / quality / release / agents / context.md の各設定が canon 窓口で読み書きできる
- structured backend の追加・変更・削除が全て人間ゲートを通る
- 識別子による変更削除の照合が target ごとに動く
- 既存 prose backend の挙動を壊さない
- cargo / clippy / 既存テスト通過
- target repo の対話 Codex で設定変更が人間ゲートで止まることを確認（実機）

## 未決事項

- TOML 書き戻しでコメント・キー順序を保つか（toml_edit 採用の是非）
- agents.toml の構造化エントリを1引数に収める形
- structured 変更の差分を gate 提示でどう見せるか
