# Phase7 腐らず成長する (要件)

## 背景

Phase6 まででコアが完成し、旗「検証可能な正本」が現物で立つ。M2 達成。
だが正本は放っておけば腐る。タスクは放置・重複・嘘 done が溜まり、来歴・経験は古い前提のまま残る。コードは死コード・孤立ファイルが増える。
Phase7 は、この腐敗を機械で「検知・測定・警告」する側を足す。検出結果へ対応する実行側 (庭師 subagent・掃除 skill・自動修復) は Phase8 へ送る。達成で M3。

参照する正本:

- `docs/requirements/20260611-core-要件.md` の B群と完了条件
- `docs/decisions/20260611-タスク管理.md` (検証可能タスク層・腐敗検知の中核必須)
- `docs/decisions/20260611-設計原則.md` (記憶の鮮度・時間軸・適応度関数・ブランド固定)
- `docs/decisions/20260613-庭師-剪定.md` (検出側 = Phase7・実行側 = Phase8)
- `docs/decisions/20260614-Phase6-quality適応度関数.md` (quality の型と phase 適応)
- `docs/roadmap/20260612-完成ロードマップ.md` の Phase 7・M3

## 目的

正本が古びず検証つきで育つ状態にする。

タスク・記憶 (来歴/経験)・コードに共通の「放置・孤立・重複・古さ」を機械で測り、`next` と verify.run へ報告する。
文脈出力時に token 量を数値で示し秘密情報を走査する。ブランド (用語・禁止語・表記) を機械検証する。
経験を汎用型だけで export / import できるようにする。

## 対象

Phase7 全体を4スライスへ分ける。各スライスで plan → design → impl → 実機検証を回す。

1. 腐敗検知の中核: タスクの 放置 / 孤立 / 重複 / ゾンビ / done未検証 と、記憶 (来歴) の鮮度を共通の適応度関数で機械検出し報告する
2. 腐敗検知をコード/repo へ拡大: 庭師の検出側 (死コード・孤立ファイル・重複)。owox はファイルシステムレベルに絞り、構文解析は検査コマンドへ委譲する
3. 測定の可視化 + ブランド検証: 文脈出力に token 量を数値表示し秘密情報を走査する。ブランドの用語・禁止語・表記を quality で機械検証する
4. 経験 export/import + 状態適応の実機確認: 汎用型だけ export し import 時に秘密走査する。maintenance 振る舞いの再起動実見

依存: スライス1 が スライス2 と記憶警告の前提。スライス3・4 は独立で軽い。

## 対象外

- 検出結果への対応 = 庭師 subagent・掃除 skill・提案→承認→実行 (Phase8。本 Phase は検知・測定・警告・提示まで)
- multiCLI 生成・オーケストレーション・既存コードからの正本逆生成 (Phase8)
- 全言語の構文解析による死コード検出 (owox は言語非依存。深い解析はプロジェクトの検査コマンドへ委譲)
- 時間軸を根拠にした記憶の自動削除 (古さは見直しを促す合図であり、捨てる根拠にしない。`docs/decisions/20260611-設計原則.md`)
- 正確なトークナイザによる厳密な token 計測 (重い。推定で足りるかを design で確定)
- eval・チーム権限・release (D群。Phase9)

## 成功条件

- 完了タスクに検証があるか機械で確かめられる (Phase6 で close に検証必須を実装済。本 Phase は既存 done の検証欠落も検出する)
- 放置・孤立・重複・ゾンビ・done未検証 のタスクを機械検出し報告できる
- 古い来歴を機械検出し警告できる
- 腐敗検知の厳格度が phase で変わる (初期 = 緩く報告のみ、保守 = 厳しく commit を止める)
- 腐敗検知が誤検出で作業を妨げない (初期は緩く・閾値は人間が調整できる)
- 文脈出力時に token 量が数値で見える
- 文脈出力時に秘密情報を走査できる
- ブランドの用語違反・禁止語・表記違反を機械検出できる
- 経験を汎用型だけで export でき、ドメイン固有情報が混ざらない
- 経験 import 時に秘密情報を走査する
- maintenance の振る舞い案内が再起動後も効く
- 最小コンテキストが主張でなく測定可能な事実になる

## 仕様

### スライス1: 腐敗検知の中核

タスクと来歴は既に日付つき ID と 1件1ファイルで揃う (`crates/core/src/task.rs`・`record.rs`)。
ここへ、現物を読んで腐敗の信号を機械検出する適応度関数を足す。新サブシステムを作らず、quality の禁止パターン機構と同じ「機械検出して報告、効かせ方は phase 適応」の型に載せる。

#### 閾値・規則の置き場 (確定)

腐敗検知の閾値・規則は quality.toml の `[decay]` 節に置く。
quality の行数予算・禁止パターンと同じ置き場で、適応度関数を一箇所に集める。phase 適応の効かせ方も quality と同じ commit ゲート経由で再利用する。
無ければ owox 標準の既定値で動く (opt-in でなく既定で検知し、数値だけ人間が上書きできる)。

#### 古さの測り方 (確定)

タスクの「最後の活動」= ID 日付と note 日付の最新。これと today (mcp が与える) の差を日数とする。
core は時計も git も持たず today を引数で受ける既存制約に従う (`docs/decisions/20260613-Phase4-tool記録層.md`)。git の mtime は使わない (非決定論で core の方針に反する)。
来歴の古さも同様に ID 日付と today の差で測る。

#### タスクの腐敗検知 (5種)

- 放置 (stale): status が todo / doing / blocked で、最後の活動からの日数が閾値超え。done / dropped は対象外
- 孤立 (孤立): links (requirement / decision / verification) が空で、かつ deps も無い。どこにも繋がらないタスク
- 重複 (重複): ID から日付前置を除いた slug が一致する別タスクがある (`strip_date_prefix` を再利用。決定論で軽い。曖昧一致は持たない)
- ゾンビ (ゾンビ): 前提 (blocks) の対象が dropped で永久に done になれない、または blocks 依存に循環があり ready になれない
- done未検証: status が done だが verification link が無い (Phase6 の close は必ず付けるが、過去・import タスクの取りこぼしを拾う)

#### 来歴の鮮度

- 放置 open: status が open の来歴で、ID 日付から閾値超え。未決のまま放置された判断点を表面化する
- 見直し合図: status が adopted の来歴で、採用から見直し閾値超え。古い前提を警告する (削除はしない。見直しを促すだけ)

#### 報告経路と context 汚染対策

- `next`: 腐敗警告の節を足す。常時報告 (ready タスク・open 来歴と同じ扱い)。ただし件数を要約し、open / 放置中のものだけ見せ、全件は出さない (来歴溜まりで context を汚さない)
- verify.run: quality 違反と同じく decay 違反を報告のみで返す。完了3区別は変えない。phase 適応の判定は commit ゲートに集約する
- 効かせ方: 初期 = 報告のみ、保守 = commit を止める (quality と同じ phase 適応)。verify.run と next は phase によらず常時報告する

#### 責務分担

- core: タスク・来歴を読み、quality.toml の `[decay]` 閾値と today を受けて腐敗を判定する純関数 (decay モジュール想定)。決定論
- mcp: today を供給し、判定結果を `next` と verify.run の封筒へ載せる

### スライス2: 腐敗検知をコード/repo へ拡大

庭師の検出側。owox はファイルシステムレベルに絞り、構文解析は検査コマンドへ委譲する。
スライス1 の decay 機構と quality の上に積み、報告は同じ decay チャネル (verify.run の data.decay) を使う。

#### owox 直接検出: 重複ファイル

- 内容が完全一致する追跡ファイルを検出する。列挙は既存の `crates/mcp/src/files.rs` を再利用し、内容読取は quality と同じく core が行う
- 言語非依存・決定論で誤検出がほぼ無い。空・極小ファイルの雑音を避けるため最小サイズ閾値を quality.toml の `[decay]` に置く
- 死コード・未使用シンボルは owox が直接検出しない (構文解析を持たない)

#### 委譲検出: [decay.checks]

- 死コード・未使用依存など構文解析が要るものは、プロジェクトの検査コマンドへ委譲する
- quality.toml に `[[decay.checks]]` (name + command) を新設する。owox が走らせ、出力を decay として報告する advisory な枠
- 完了を判定する `[[verify.checks]]` (pass/fail ゲート) と区別する。死コードは「完了失敗」でなく助言なので、ゲートを汚さず decay として扱う
- 規約: 検査コマンドが decay を見つけたら非ゼロ終了する (run_check を再利用)。出力の要約を finding の detail に載せる

#### 報告経路と効かせ方

- next: スライス1 の タスク・来歴 decay のみ (高速)。repo 走査と外部コマンドを伴うコード decay は next に載せない (next は素早い航法に保つ)
- verify.run: タスク・来歴 decay に加え、重複ファイルと decay.checks の結果を data.decay へ報告する (全体像)
- commit ゲート: コード decay (重複ファイル・decay.checks) は advisory で commit を止めない。commit を止めるのは引き続き構造的タスク腐敗 (done未検証・ゾンビ) のみ
- 検出側のみ。剪定の実行 (庭師 subagent・掃除 skill) は Phase8

### スライス3: 測定の可視化 + ブランド検証 (確定)

設計確定済み。詳細は `docs/decisions/20260614-Phase7-測定可視化とブランド検証.md`。

- token 量の数値表示: 文脈出力 (context tool) が返す情報量を推定トークン数で示す。正確なトークナイザは入れず日本語対応の軽い推定 (CJK 別勘定) を core の純関数 estimate_tokens に置く
- 秘密情報走査: 文脈出力時に秘密情報を走査する。`irreversible.rs` と同じ「組込み正規表現 + 将来 target 拡張」方式を core の secret モジュールへ再利用する。検出時は警告のみ (read 専用ナビは block しない)。スライス4 の export/import が本命の再利用先
- ブランド機械検証: 禁止語を glossary.md の予約見出し Forbidden へ正規表現で構造化し (用語集を用語+定義+禁止語の正本に)、run_brand が QualityViolation kind="brand" として既存 quality チャネルへ報告する。正規表現で表せる表記規則は既存 quality.toml boundaries を流用 (新機構なし)。助詞省略など構文解析が要る文体は対象外

### スライス4: 経験 export/import + 二層ルール + 状態適応 (確定)

設計確定済み。詳細は `docs/decisions/20260614-Phase7-経験IOと二層ルール.md`。
人間判断で二層ルール (固定 / 成長) と canon.propose を本スライスへ統合した。

- 経験 export: 経験束 = skill (SKILL.md + scripts) + 成長層 practices.md。memory/tests/owox.toml/固定層は型で除外。本文は秘密走査し、クリーンなら書出 + needs_human・秘密検出時は failed で書かない
- 経験 import: 秘密走査し、検出時は needs_human で取り込まない。クリーンなら skill を draft (owox.toml 無し) で書き practices をマージ
- 二層ルール: 固定層 (brand/rules/glossary・人間が握る) と成長層 (practices.md・AI が育てる・鮮度 decay の見直し合図・session_start ライブ注入) を別管理。二層は権限差でなく内容差
- 統一 canon 編集: brand/rules/practices/glossary を 1 モデルへ。canon.add は AI 直接 + 来歴、canon.propose は変更・削除を人間ゲート。MCP は canon.add/canon.propose/glossary.lookup の 3 つへ集約 (旧 glossary CRUD・practice.add を撤去)
- 状態適応の回帰防止: maintenance 振る舞い案内が state.toml 永続で再起動後も効くことを確認 (回帰テスト + 実機)


## 実現方針

### 段階分け

スライス1から順に縦貫する。各スライスで plan → design → impl → 実機検証。

1. 腐敗検知の中核 (タスク + 来歴の適応度関数。既存 task 層・日付 ID の上に最短で立てる。後続の前提)
2. コード/repo へ拡大 (ファイルシステムレベルに絞り構文解析は委譲)
3. 測定の可視化 + ブランド検証 (独立・軽い)
4. 経験 export/import + 状態適応の実機確認 (独立・軽い)

### 変更対象 (スライス1)

- crates/core: decay モジュールの新設 (タスク・来歴を読み quality.toml の `[decay]` 閾値と today で判定する純関数)、quality.toml の型へ `[decay]` 節を足す、verify.rs の封筒へ decay 違反を載せる
- mcp 側: today の供給、`next` の描画 (`render_next`) へ腐敗警告の節、verify.run 封筒へ decay 違反、commit ゲートの phase 適応へ decay を組み込む
- 生成側 (targets/codex): 必要なら tool 説明の更新 (新しい入口は足さない。報告は既存の next / verify.run に乗せる)
- tests/fixtures と target サンドボックス
- docs: 本要件・design 判断・validation

### 危険

- 腐敗検知の誤検出・狼少年化。phase 適応で初期は緩く、閾値は人間が quality.toml で調整できるようにする
- 重複検出の slug 一致は、正当に分割した別タスクを重複扱いしうる。報告のみに留め block しない側へ倒す
- 来歴溜まりが context を汚す。next は open / 放置中だけ要約して見せ、全件は出さない
- 閾値の既定値が phase に合わない。初期 = 緩・保守 = 厳の phase 適応で吸収し、既定は緩めに置く
- コード検出 (スライス2) が既存検査コマンドと重複しうる。owox はファイルシステムレベルに絞り構文解析は委譲する
- 秘密走査 (スライス3・4) の誤検出。export は needs_human で人間ゲートへ送る

### 人間判断が必要な点

- 腐敗検知の閾値の既定値 (放置何日・見直し何日・phase ごとの厳格度)
- token 量を推定でよいか、正確なトークナイザを使うか (スライス3)
- 経験 export の汎用型とドメイン固有型の境界 (スライス4)

## 検証

target サンドボックスで実機 Codex により確認する。

- タスクを溜めて放置・孤立・重複・ゾンビ・done未検証 を作り、`next` と verify.run が報告するか
- 古い来歴 (open 放置・採用後長期) が警告されるか
- phase を初期 → 保守へ変え、保守で commit が止まり初期で報告のみになるか
- 閾値を quality.toml で調整でき、誤検出を抑えられるか
- next の腐敗警告が要約され context を汚さないか
- 再生成・並行ブランチで壊れないか (記録層と同じ確認)
- cargo test / clippy / fmt
