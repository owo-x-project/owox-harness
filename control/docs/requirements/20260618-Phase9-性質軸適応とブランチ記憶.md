## 背景

owox-harness を使う側の AI 駆動開発の進め方を支える。
AI 駆動開発の方法論 (PRFAQ・理想先行・段階化・クリーンアーキテクチャ) は人間判断と AI 自律の境界を自然に定義し、owox の存在理由と一致する。
これを owox が既定で持ち、プロジェクトの性質に応じて出し入れする機構を足す。

あわせて、並行作業 (worktree・マルチエージェント) でブランチ間の作業文脈が衝突する空白を、ブランチ単位の作業記憶層で埋める。

## 目的

- プロジェクトの性質に応じ、4方法論を軸で出し入れする (人間は性質を人語で言うだけ・生軸を触らない)
- 人間判断と AI 自律の境界を、時間軸 (phase) と層軸の直交ゲートで自然に効かせる
- 既存プロジェクトへ後入れする時、コードから性質の寄りを当て draft で人間確定する
- ブランチ単位の作業記憶でブランチ間の文脈衝突を解消する

最小コンテキスト・canon 直読み禁止・core は時計を読まない、を崩さない。

## 対象

- owox-harness を使う側のプロジェクト全般 (新規・既存)
- 並行作業 (worktree・同一マシン複数 CLI・マルチエージェント)

## 対象外

- マルチ CLI 生成 (Claude Code)・オーケストレーション (Phase 9 の別スライス)
- 契約/型変更の構文解析による正確検出 (後段 Phase。v1 はパス宣言で近似)
- 軸の3値以上・軸の追加 (実需要時。v1 は4軸2値)
- ブランチ作業記憶のマシン跨ぎ共有 (v1 は同 repo 同ディスクのみ)
- 性質軸とブランチ運用の強い連動 (v1 疎結合)

## 成功条件

- 束を人語で選ぶ/逆生成 draft を確定するだけで、生軸を触らず方法論が出し入れされる
- 既存 repo から性質 draft が根拠つきで出て、人間が確認・確定できる
- 核の削除/契約面編集で操作前に人間ゲートが鳴り、核の境界違反は commit で block、その他の核編集は助言で流れる
- 軸×phase の合成が検算4ケースで矛盾なく効く
- ブランチをまたいでも作業文脈が混ざらず、放置ブランチ記憶が腐敗検知に出る
- cargo / clippy 通過、MCP stdio・hook で機械確証、実機対話で確認

## 仕様

### A. profile.toml と軸解決

- `.owox/profile.toml` 新規。性質 (固定) を宣言。state.toml (可変) と分離
- per-project: `[profile] preset`・`[profile.axes]` (束からの差分上書き)・`[bundles.<name>]` (独自束)
- owox 組込み: 軸レジストリ (軸と許容値・各値=1モジュール)・標準束
- 4軸2値: requirements-shape (prfaq/lightweight)・prioritization (ideal-first/incremental)・delivery (phased/continuous)・architecture (layered/flat)
- 軸は列挙集合として型設計 (将来値追加の余地)。v1 は各軸2値
- 解決: 実効軸 = 束の既定値に axes 上書きを重ねる → active モジュール集合 (floor_context 合成と同型)
- 既定束 = フル方法論 (prfaq+ideal-first+phased+layered)。全 off も可
- 型検証 (読込後 Rust): preset は既知の束・axes キーは既知の軸・値は許容値。未知は弾きフィードバック (canon.add 未知 section と同型)
- 失敗時: 未知 preset/軸/値は読込エラーで valid 一覧を返す。profile.toml 不在は既定束 (フル) として扱う

### B. kickoff 束導線 + 逆生成の厚い検出

- kickoff は名前付き束を人語で提示し選ばせる (生軸は見せない)。新規の既定はフル方法論束
- 既存は逆生成が束を先に当て「推定 ○○型 (根拠つき)」を提示 → 人間は合ってる/直すだけ
- 逆生成は安い言語非依存シグナルで各軸の寄りを推定:
  - architecture: 層ディレクトリ・quality.toml 層境界・依存方向の一貫性
  - requirements-shape: docs/requirements/ や要件ファイル・PRFAQ 風文書・RFC/設計文書・README 充実度
  - prioritization: ロードマップ/Vision の優先順位記述 vs issue 駆動小刻みコミット
  - delivery: git タグ/リリース規則性・CHANGELOG バージョン区切り・release ブランチ vs trunk ベース
- 出力は profile.toml draft + 根拠。自動確定しない。根拠を必ず提示
- 束→軸の標準対応は owox 組込み。独自束は config 追加 (コード不要)
- 上級者は profile.toml の生軸を直接編集可 (通常導線では触らせない)

### C. 4方法論の既存面実装

- PRFAQ: 要件起草スキル `req` (単一)。requirements-shape の値で起草方法が prfaq/lightweight に切替。skill 名は profile が変わっても不変・中身だけ変わる。PRFAQ は要件高度のみ決める。正本は要件正本層の一本で起草方法は保存先を変えない — PRFAQ も出力を要件へ蒸留しプレスリリース文書は別に残さず、便益となぜは来歴へ link (`20260620-要件分類とPRFAQ正本.md`)
- 種類分類: 要件へ任意の種類タグ (functional / non-functional の2値) を追加。性質軸と独立でどの要件にも付けられる (機能=テスト link で met 判定向き・非機能=quality.toml 連動側)。技術・設計の制約は要件でなく来歴へ。種類で対象者は分けない (`20260620-要件分類とPRFAQ正本.md`)
- 理想先行: 要件正本層 (Phase 6) に優先度属性を追加 + 優先順位付けの人間ゲート。優先度 = 人間が並べたランク・AI は提案まで
- 段階化: タスク/要件に軽量な stage グルーピング。delivery=phased の時だけ active
- クリーンアーキ: quality.toml の層境界を唯一の層真実として再利用し3分解:
  - 依存性ルール = quality.toml 層境界/依存方向 (既存そのまま)
  - 進行度/達成度 = 要件/タスクの layer タグ → 層別に充足報告。layer タグは quality.toml [[layers]] の任意 name へ照合 (未知は弾く・name 未宣言なら照合しない)
  - 自律度勾配 = ゲートで層別に強弱 (D)
  - architecture=layered の時だけ層機構 active。flat は全 off

### D. ゲート合成 (層×phase)

- phase ゲート (遵法の厳しさ) と層ゲート (変更権限の所在) を別種の直交ゲートにし AND で独立発火
- 1判定へ畳む時は max。はしご1本: free < warn < commit-block < pre-action-human-gate
- 層写像: guarded → pre-action-human-gate (phase 不問) / supervised → advise+記録 (phase が commit-block を変調) / free → 安全ゲートのみ
- 最終 enforcement = max(層写像, phase写像)。検算4ケース (初期+free=free・初期+guarded=human-gate・保守+free=commit-block・保守+guarded=human-gate)
- guarded 発火 v1 = 4段 (全て quality.toml 部品・構文解析不要)。操作前に分かるものは操作前ゲート・書込後にしか分からないものは commit-block:
  1. ファイル削除 → 操作前に人間ゲート (パスだけで判定可)
  2. (任意) quality.toml に宣言した契約面パス (`*/ports/*`・`*_interface.*` 等) の編集 → 操作前に人間ゲート (パスだけで判定可)
  3. 境界違反/依存方向違反 → commit-block (編集内容依存で書込後にしか判定できず操作前ゲート不可。guarded を phase 不問で block へ落とす)
  4. それ以外の guarded 編集 → 助言+記録
- 実装は既存ゲートの拡張。guarded の人間ゲートは canon.propose の仕組みをコード層へ広げる。層の真実は quality.toml 境界を再利用

### E. ブランチ作業記憶層

- work/ を拡張しブランチ別キー・git-ignored 継続 (新層足さず)
- 保管場所は1箇所に正規化し worktree 横断で同じバケツを引く
- キー = ブランチ名。rename/rebase 自動追従なし。detached HEAD は global 退避
- ライフサイクル = 自動削除しない。merge/削除で孤児化 → stale-branch-memory で剪定候補。恒久判断は decisions/・経験は experience/ へ明示昇格。破棄基準 = ブランチ消失+経過日数
- 並行分離 = ブランチ別キーで自然分離。同一ブランチのマルチエージェントは共有
- 読みはオンデマンド (tool)・床へ常時注入しない (任意で件数の気づきのみ)
- 書込時 secret 走査。stale-branch-memory を腐敗検知へ連動
- 共有 = 同 repo 同ディスクのみ。マシン跨ぎは対象外

## 実現方針

### 変更対象 (想定)

- crates/core: profile.rs 新規 (軸レジストリ・束解決・型検証)、quality.rs (層境界の層別自律度・契約面パス宣言)、要件/タスク型に優先度・layer・stage 属性、要件型に種類 (kind=functional/non-functional) 属性、decay.rs (stale-branch-memory)、work のブランチ別キー化と正規化
- crates/mcp: profile 読込と active モジュール解決、kickoff の束選択導線、逆生成の検出、ゲート合成 (層×phase の max・guarded 4段)、ブランチ記憶 tool
- 生成物側: 要件起草スキル req
- docs: 本要件・設計・validation

### 段階分け (案)

1. profile.toml と軸解決 (A) — 土台。これが無いと他が乗らない
2. ゲート合成 (D) — 層×phase の max と guarded 4段。既存ゲートの拡張
3. 4方法論の既存面実装 (C) — 優先度/layer/stage 属性・req スキル
4. kickoff 束導線 + 逆生成 (B) — A の上に導線と検出
5. ブランチ作業記憶層 (E) — 独立。並行で進めても良い

### 依存関係

- D・C は A (軸解決) に依存
- B は A・C に依存 (束→軸→active モジュール)
- E は独立 (性質軸と疎結合)

### 危険

- guarded 4段が重すぎ/緩すぎ → 実機で範囲調整
- 厚い逆生成が外す → draft+根拠で人間判断
- 2新機構を絡めすぎる → v1 疎結合を守る
- worktree で保管場所が散る → 正規化

## 検証

- 単体: profile 読込/型検証 (未知軸/値を弾く・preset 解決・axes 上書き)、軸→active モジュール解決、合成 max の検算4ケース、guarded 4段の発火判定、逆生成シグナルの寄り推定、ブランチ別キー round-trip・孤児 stale 判定・secret 検出で止める
- 結合: serve 経由で profile に応じ active モジュールが変わる、kickoff 束選択→profile.toml 生成、逆生成 draft の根拠提示、next/verify.run に stale-branch-memory、worktree/別ブランチで記憶が衝突しない
- 手動 (実機): 束を人語で選び生軸を触らず方法論が出し入れされる、既存 repo から性質 draft が根拠つきで出る、核の境界違反/削除/契約面で人間ゲート、ブランチをまたいでも文脈が混ざらない・merge 後に剪定候補に出る
- 検証記録は `docs/validation/` へ (Phase 8 と同様)

## 確定事項 (実装で決着)

- 束→軸の標準対応 = `crates/core/src/profile.rs` の builtin_bundle で確定。clean-arch-app=prfaq+ideal-first+phased+layered / script=全 off (lightweight+incremental+continuous+flat) / library=prfaq+ideal-first+continuous+layered / data-platform=lightweight+incremental+phased+layered / research=全 off
- 契約面パス宣言 = `crates/core/src/quality.rs` の Layer.contract_surface (任意の `Vec<String>`)。組込み既定パターンは置かずプロジェクトが quality.toml に宣言する
- stale-branch-memory 既定 = `crates/core/src/quality.rs` の DecayConfig で 30 日。来歴 review/調査 knowledge の 90 日とは別値 (ブランチは短命)
- 優先度属性 = `crates/core/src/requirement.rs` の Requirement.priority (`Option<u32>` 整数ランク・小さいほど高優先)。要件のみ・タスクには持たせない。AI は提案まで・人間が並べる
- stage タグ = 任意文字列 (`Option<String>`)。対応する真実が他に無いため値域制限なし・delivery=phased の時だけグルーピングに使う
- layer タグ = quality.toml [[layers]] の任意の name へ照合する (`crates/core/src/quality.rs` の check_known_layer)。未知は弾き有効名を返す・name 未宣言なら照合しない後方互換。層別充足報告とゲート層を同じ層名で揃える
