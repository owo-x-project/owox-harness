# owox-harness 要件

## 背景

`owox-harness` は、AIエージェントを使った開発で、人間が重要判断を握りながら、AIの自律性を活かすための target harness を作成・管理する道具。

企画書:

- `docs/concept/企画書.md`

決定:

- `docs/decisions/` を参照

## 目的

人間が重要判断を握ったまま、人間とAIが迷わず、少ない必読量で、要件を確実に実現できる target harness を作成・管理する。

さらに、プロジェクトの価値、判断、作業経験、検証可能な成長を target harness に組み込み、方向性の乱れを防ぐ。

## 解決したい問題

- 人間が毎回同じ説明をする
- 人間が次に何を決めるべきか分からない
- AIが目的や制約を読み落とす
- AIが要件・仕様・方針を勝手に推測する
- 必要情報が多すぎて、人間もAIも迷う
- 必読量とトークン消費が大きい
- 曖昧なルールが多く、機械検証できない
- 要件が実装や検証へ確実につながらない
- 実装だけ先に進み、判断や理由が残らない
- 作業経験が再利用できる形で残らない
- AIごとに指示や運用が分裂する
- レビューと修正が曖昧にループする
- target harness が自ら検証可能なルールで成長しない

## 対象

- 第一対象: AIエージェントを本格的に使う個人開発者
- 第二対象: 複数のAI CLIを使い分ける開発者
- 第三対象: チームへAI駆動開発を入れる開発者

## 対象外

- AIに全判断を任せる
- 人間判断を不要にする
- 何でも自動化する汎用道具にする
- 特定AI CLI専用に閉じる
- 会話ログをそのまま正本扱いする
- 検証不能な経験を自動で正本化する
- ドメイン固有情報を汎用経験へ混ぜる
- owlspec が将来扱う要件仕様管理そのものを深く作り込む
- owlview が将来扱う画面設計管理そのものを深く作り込む

## 成功条件

- 人間が次に決めることを迷わない
- AIが参照すべき情報を迷わない
- AIが作業してよい範囲を迷わない
- 必読量とトークン消費が減る
- 要件、仕様、実装、検証がつながる
- 複数AI CLI向け target harness を生成できる
- 判断・経緯が追跡可能、機械記録可能、検証可能
- 機械検証できるものは機械検証される
- 価値、方向性、言葉がぶれない
- 対象外、成功条件、変更方針、依存追加条件、削除基準が正本化される
- 先にやること、後回し、やらないことが分かる
- AIが自律作業でき、人間確認が必要な点で止まる
- 変更してよい範囲、確認必須範囲、破壊的変更の条件が分かる
- 作業完了、要件完了、検証完了を区別できる
- 要件、受け入れ基準、タスク、検証結果が link でつながる
- やること (タスク) が腐らず、検証つきで追跡できる
- AIが書いたコードの品質・堅牢性を機械で担保できる
- 品質バー、依存方向、層境界、複雑度予算を機械検証できる
- 作業経験を再利用可能に蓄積できる
- ドメイン固有情報を汎用経験へ混ぜない
- ルールを検証つきで追加、修正、削除できる
- 秘密情報、危険操作、外部送信、権限を制御できる
- 技術選定の採用理由、捨てた案、見直し条件、依存追加条件が残る
- 人間もAIも、次回作業を迷わず再開できる
- 個人利用からチーム利用へ広げられる
- owlspec / owlview と将来つながる余地がある

## 領域

### 生成

複数AI CLI向け target harness を生成する。

### 文脈

人間とAIへ必要最小限の情報だけ出す。

文脈出力時に token 予算を数値で示し、秘密情報を走査する。最小コンテキストを測定可能にする。

### 判断

判断・経緯を追跡可能、機械記録可能、検証可能にする。

### ブランド

対象プロジェクトの価値、方向性、言葉を固定し、方向性の乱れを防ぐ。

対象外と成功条件も固定し、作業の広がりすぎを防ぐ。

### 作業

AIが自律作業でき、人間確認が必要な点で止まれるようにする。

変更してよい範囲、確認必須範囲、破壊的変更の条件を明確にする。

変更方針、依存追加条件、削除基準を正本化する。

### 検証

機械検証できるものを決定論的な形にする。

作業完了、要件完了、検証完了を区別する。

要件、受け入れ基準、検証結果を link で追跡できるようにする。

### 品質

品質バー、依存方向、層境界、循環禁止、複雑度予算を宣言し、機械検証できる形にする。

### 経験

作業経験を再利用可能に蓄積する。

ドメイン固有情報は汎用経験へ混ぜない。

判断・経験に時間と鮮度を持たせ、古さを腐敗検知へ連動する。

次回作業を迷わず再開できる情報を残す。

### 成長

ルールの追加、修正、削除を検証つきで行う。

### 安全性

秘密情報、危険操作、外部送信、権限を制御する。

### 技術選定

採用理由、捨てた案、見直し条件、依存追加条件を残す。

### 運用

個人利用からチーム利用へ広げられるようにする。

## 実装順

具体内容は `docs/decisions/20260611-target-harness内容.md`。

### 第1段階 (Codex CLI、スキル・ライフサイクルまで)

- target repo に target harness を生成できる
- 生成対象はまず Codex CLI (`docs/decisions/20260611-第1対象CLI.md`)
- 正本ソース: brand (最小、対象外・成功条件を含む) / rules (変更方針・依存追加条件・削除基準・可逆性境界・人間ゲート・安全性) / context / targets
- 生成物: ルート指示ファイル + hook 5種 + Codex MCP 設定 + skills 配置 + slash command (`docs/decisions/20260611-ショートカット.md`)
- MCP: generate / inspect・diff / decide (来歴) / verify (完了3区別) / gate / skill ライフサイクル
- スキル・ライフサイクル: 作成 → テスト合格ゲート → 登録 → 改善、経験メモリ
- 決定事項・未決事項・検証結果を .owox/ に残す
- target repo で生成結果を検証する
- 生成物が再生成しても壊れないか確認する
- 複数AI CLI対応は構造だけ考慮し、実装は後回し

### 第2段階以降 (優先順位順)

競合調査 (`docs/competitive/`) 反映。旗「検証可能な正本」を裏打ちする度合いと、後続の前提になる順で並べる。順序は推奨であり、人間が見直せる。

#### A群 検証を閉じる (最優先・旗の核・後続の前提)

- requirements/ (要件・受け入れ基準・検証 link)
- quality.toml (品質バー・依存方向・層境界・複雑度予算)
- 品質保証 (plan-alignment・受け入れ基準trace・品質適応度関数・回帰捕捉・多視点レビュー。`docs/decisions/20260611-品質保証.md`)

狙い: 要件完了の機械判定を成立させ、完了3区別を完成する。タスクの link 先の前提でもある。

#### B群 腐らず成長 (旗の後半・最小コンテキストの裏打ち)

- 検証可能タスク層 (専用増分。腐敗防止を中核に。`docs/decisions/20260611-タスク管理.md`)
- 記憶の鮮度・時間軸 (来歴・経験に時間と鮮度を持たせ、古さを腐敗検知へ連動。`docs/decisions/20260611-設計原則.md`)
- 文脈の測定可視化 (文脈出力時に token 予算を数値表示し、秘密情報を走査。`docs/decisions/20260611-MCP設計.md`)
- ブランド固定の機械検証、適応度関数の作り込み (`docs/decisions/20260611-設計原則.md`)
- 状態適応・回帰防止
- 経験の export / import (汎用のみ、型でドメイン分離)

狙い: 正本が腐らず検証つきで育つ。最小コンテキストを主張でなく測定可能な事実にする。

#### C群 規模・移植・方向付け (差別化軸を広げる)

- 複数AI CLI向け生成 (`docs/decisions/20260611-開発アーキテクチャ.md` の Target トレイト)
- オーケストレーション (役割別subagentパターン生成・役割権限・モデルのティア振り分け。`docs/decisions/20260611-オーケストレーション.md`)
- subagents の作り込み
- 既存コードからの正本逆生成 (kickoff 拡張。既存 repo から rules / quality の初期案を出す。`docs/decisions/20260611-方向付け.md`)
- 方向付け (kickoff skill・Vision・次の一手・判断支援。`docs/decisions/20260611-方向付け.md`)

狙い: 第2・第3対象ユーザ (複数CLI・既存プロジェクト・チーム) へ広げる。

#### D群 条件付き・opt-in (前提が固まってから)

- eval (opt-in。改善検証を先、品質採点を後。`docs/decisions/20260611-eval方針.md`)
- 個人からチームへの運用拡張 (役割ベース権限・CODEOWNERS委譲・承認identity。`docs/decisions/20260611-チーム権限.md`)
- release.toml (配布運用がある対象プロジェクトのみ)

狙い: 価値はあるが旗の前提でない、または opt-in / 対象限定。

## 完了条件

完了を3区別する。

- 作業完了: 与えた範囲の作業が終わった
- 要件完了: 要件が満たされた
- 検証完了: 機械検証または人間判断で確かめられた

機械検証できるものは機械検証される。検証不能なものは人間判断が必要な項目として明示する。

第1段階では要件完了の機械判定を持たない。verify は要件完了を人間判断 (needs_human) として返す。要件とテストの trace (第2段階の品質保証、`docs/decisions/20260611-品質保証.md`) 導入後に機械判定へ移行する。

## 検証

- target repo を作る
- owox-harness で target harness を生成する
- Codex CLI が target harness を読めることを確認する
- hook の発火・決定的ブロック・MCP resource 読取を最初期に確認する (`docs/decisions/20260611-第1対象CLI.md` の注意)
- 作業開始時に必要情報だけ参照できるか確認する
- brand / rules の長文を人間が TOML で編集しやすいか早期に確認する
- 人間確認が必要な場面で止まれるか確認する
- 作業後に決定事項・未決事項・検証結果が残るか確認する
- 生成物が再生成しても壊れないか確認する
- .owox/ の型検証、参照検証、差分検証、承認検証を確認する
- 依存追加時に理由、代替案、見直し条件、検証結果が残るか確認する
- 公開挙動変更時に必要文書が更新されているか確認する
- 各領域に対応する target repo 検証を用意する
- 機械検証できる部分はスクリプト化する
- ルール改善は、改善前後の差分と検証結果を残す
