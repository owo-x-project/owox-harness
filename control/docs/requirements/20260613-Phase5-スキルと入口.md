# Phase5 スキルと入口

## 背景

Phase4 (MCP 中核) 完了。Phase5 は第1段階完成 (M1) の最後。Codex で読む・止まる・記録する・育つが一通り回る状態にする。
出発点は `docs/handoff/20260613-Phase5開始-スキルと入口.md`、`docs/roadmap/20260612-完成ロードマップ.md` Phase5。
既決方針は `docs/decisions/20260611-設計原則.md` スキルライフサイクル、`docs/decisions/20260611-ショートカット.md`。

計画前に一次情報 (OpenAI 公式) で Codex の現状を確認し、前提を更新した。

- Codex の custom prompts (slash の想定先) は廃止予定。置き場所は `~/.codex/prompts/` でホーム限定、repo 共有不可。公式は「再利用指示は skills を使え」と明記
- Codex に native Skills あり。repo 内 `.agents/skills/` (作業dir→repoルートへ遡って探索) で git 共有可。必須 frontmatter は name と description のみ。起動は暗黙 (description 一致で自動) と明示 ($名前・/skills) の2通り。テスト・検証機構は無い
- SKILL.md は横断標準 (Claude Code・Gemini CLI 等が同形式を採用)

含意。Codex ではスキルと slash 入口が同じ `.agents/skills/` に収束する。owox の「テスト合格ゲート」は Codex に無いため、そのまま差別化になる。

## 目的

AI が技を検証つきで育て、人間が入口を持つ。第1段階の一通り (読む・止まる・記録する・育つ) を実機 Codex で回す。

## 対象

4つ。着手順は setup → スキル → 入口 → glossary 動的化。

- setup: 別 target repo へ owox 配置 + Codex の MCP 設定 + 検査 を自動化する
- スキルライフサイクル: 作成 → 評価 (テスト合格ゲート) → 登録 → 昇格 (人間ゲート) → 改善。`.owox/skills/` を正本とし、合格分のみ `.agents/skills/` へ生成する。経験メモリ更新 (skill.remember)・昇格 (skill.promote) の口を持つ
- 入口 (slash 相当): commands.toml を正本とし、各コマンドを `.agents/skills/` の薄い skill として生成する
- glossary 動的化 + canon 直読み禁止: 用語を tool で CRUD・読取し、用語名は SessionStart で注入する。指針 canon の直読みを tool / resource / 注入へ一本化する

## 対象外

- 第2段階コマンド (/generate /gate /export /import)
- マルチ CLI 生成 (Claude Code / Gemini CLI。Phase8)
- 経験の export / import 汎用化 (Phase7)
- 腐敗検知の適応度関数 (Phase7)。経験メモリの鮮度連動は形だけ用意し、検知は Phase7
- バイナリの配布・install 自動化 (Phase9)

## 成功条件

- テスト合格した skill だけ Codex から見える (未合格は `.agents/skills/` へ出ない、または無効)
- setup で別 target repo へ導入でき、Codex が owox の MCP・hook・skill を認識する
- /kickoff /next /status /decide /verify /review /task /handoff が Codex から叩け、中身が MCP tool / resource へ繋がる
- 第1段階の一通り (読む・止まる・記録する・育つ) が実機 Codex で回る

## 仕様

### setup (先行)

- `owox setup [dir]`: 既存の生成器 (`crates/core` の registry / write_all) を土台にする
- やること: owox 実行ファイルの所在確認・案内、.codex/config.toml (MCP 登録) ・hooks.json ・AGENTS.md ・`.agents/skills/` の生成、設定の妥当性検査
- 反復導入でき、別 target repo の検証が楽になる

### スキルライフサイクル

- 正本 `.owox/skills/<skill>/`: SKILL.md (frontmatter の name / description + 手順) + tests/ + scripts/ (任意) + 経験メモリ (任意)。git 管理
- 型検証: frontmatter 必須項目、ドメイン分離 (export 型にドメイン固有項を入れない)
- 評価 = 登録ゲート: 同梱テスト合格で初めて `.agents/skills/<skill>/` へ生成する。未合格は生成しない、または無効化する
- 生成物: SKILL.md と scripts/、必要なら agents/openai.yaml (自動起動可否)。tests/ と経験メモリは Codex へ出さない (正本のみ)
- 改善: テスト失敗・実行誤りで再生成。経験メモリが改善を導く
- カタログ: frontmatter だけ読む進行的開示で出す (最小コンテキスト)
- 人間ゲート: 正本昇格、不可逆操作を含む skill は人間確認

### 入口 (slash 相当)

- commands.toml 正本: 名前・別名・呼ぶ MCP tool / skill
- 各コマンドを `.agents/skills/<command>/` の薄い skill として生成する。中身は対応する MCP tool / resource を呼ぶトリガ文
- 自動起動可否: 入口系は明示起動のみを既定にする (誤って自動起動させない)。経験を育てる技系 skill は暗黙起動も許す。per-skill で制御する
- namespace 付きを既定 (例 /owox:next) + よく使うものに別名。プロジェクトが commands.toml で追加できる
- 第1段階コマンド: /kickoff /next /status /decide /verify /review /task /handoff

## 実現方針

- core: skill モデル (SKILL.md frontmatter の型・検証)、commands.toml の型、skill 生成器
- mcp: skill カタログを返す tool、setup サブコマンド
- 既存の canon → generate / write_all を土台にする
- 段階: setup → skill 正本型 + テストゲート + 生成 → カタログ → commands から skill 生成
- 決定文書の更新を伴う: ショートカット (prompts → skills の転換)、設計原則のスキルライフサイクルへ Codex native との整合を追記。新規 decision として Phase5 の判断を残す

## 検証

- 単体: frontmatter 検証、テストゲート (合格/不合格で生成有無が変わる)、commands から skill 生成、config マージ
- 実機 Codex: setup 後に skill が /skills に出る、$next 等が MCP を引く、テスト未合格 skill が出ない、第1段階の一通りが回る

## 決定事項 (本計画で確定)

- slash 入口は skills 上に実現する。Codex の custom prompts は廃止予定・ホーム限定のため使わない (ショートカット決定文書を更新する)
- スキルは正本分離・生成。`.owox/skills/` を正本とし合格分のみ `.agents/skills/` へ生成する
- 着手順は setup → スキル → 入口 → glossary 動的化
- 入口系 skill は明示起動を既定、技系 skill は暗黙起動も許す。per-skill で制御する
- 設計判断の詳細は `docs/decisions/20260613-Phase5-スキルと入口.md`。要点:
  - skill 固有設定は owox.toml サイドカー、テストゲートは tests/ + run_check 再利用、生成は core モジュール + Target 拡張、カタログは tool
  - 昇格は 3 状態 (draft / registered / promoted)、自動起動 (implicit) は昇格後のみ解禁
  - 経験メモリ更新は skill.remember (memory.md 追記)、鮮度・腐敗検知は Phase7
  - glossary CRUD は add 可・update / remove は人間ゲート、用語名は SessionStart 注入、AGENTS.md の静的列挙は廃止
  - canon 直読み禁止は指針 canon が対象、`.owox/skills/` は著作の作業域として例外、強制は部分的
  - setup は設定生成 + 検査に絞る (install は Phase9)

## 未決事項 (design で潰す)

- skill のテスト実行機構: verify の検査再実行に乗せるか、skill 専用の走らせ方か
- カタログの出し方: MCP tool か resource か
- setup のバイナリ配置範囲: install まで持つか、所在前提で設定生成に絞るか
- 経験メモリの形と鮮度連動の境界 (Phase7 との線引き)
- skill 生成を CodexTarget の拡張にするか、専用 Target / 生成器にするか
- 自動起動可否の既定と per-skill 指定方法 (agents/openai.yaml の生成)
