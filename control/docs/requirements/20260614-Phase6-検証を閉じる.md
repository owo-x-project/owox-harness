# Phase6 検証を閉じる (要件)

## 背景

Phase5 まででコアの土台が回る (読む・止まる・記録する・育つ)。M1 達成。
だが「要件が満たされたか」は今 verify.run が人間判断 (needs_human) で返すだけで、機械判定できない。
完了3区別 (作業完了 / 要件完了 / 検証完了) のうち要件完了が機械判定へ移れば、旗「検証可能な正本」が現物で立つ。これで M2・コア完成。

参照する正本:

- `docs/requirements/20260611-core-要件.md` の A群と完了条件
- `docs/decisions/20260611-品質保証.md` (検証2系統・適応度関数・多視点レビュー)
- `docs/decisions/20260613-Phase4-tool記録層.md` (記録層・verify・封筒・状態保存ビューを持たない方針)
- `docs/roadmap/20260612-完成ロードマップ.md` の Phase 6

## 目的

要件が満たされたかを機械で判定できるようにする。

要件・受け入れ基準・検証を link でつなぎ、受け入れ基準の検証が全通過なら要件完了を機械判定する。
品質バー (依存方向・層境界・循環・複雑度予算) を宣言し機械検証する。
plan-alignment と多視点レビューの枠組みを owox 標準として持つ。

## 対象

Phase6 全体を4スライスへ分ける。各スライスで plan → design → impl → 実機検証を回す。

1. requirements 正本層 + tool: 要件・受け入れ基準・検証 link の正本と CRUD/list の tool
2. 要件↔テスト trace + 要件完了の機械判定: 受け入れ基準の検証 link の欠落を機械検出し、verify.run の要件完了を機械判定へ移す
3. quality.toml の適応度関数: 品質バー・依存方向・層境界・循環・複雑度予算を宣言し機械検証する
4. plan-alignment + 多視点レビューの枠組み: 観点と進め方を owox 標準として持つ (実行のオーケストレーションは Phase8)

## 対象外

- 多視点+敵対レビュー subagent の実行オーケストレーション (Phase8。本 Phase は観点と枠組みのみ)
- 役割別 subagent・モデルのティア振り分け (Phase8)
- 腐敗検知の適応度関数 (stale / 孤立 / 重複 / ゾンビ。Phase7)
- 要件仕様管理そのものの作り込み (将来 owlspec。owox は最小の requirements 正本層に留める)
- ミューテーションテスト・property test・決定論シミュレーションテスト (`docs/decisions/20260611-品質保証.md` の見送り)
- 全言語の構文解析による深い構造検証 (owox は言語非依存。深い解析はプロジェクトの検査コマンドへ委譲)

## 成功条件

- 要件・受け入れ基準・検証が link でつながる
- 受け入れ基準の検証 link の欠落を機械検出できる
- 受け入れ基準の検証が全通過なら要件完了を機械判定できる
- 検証 link が欠ける要件は従来どおり人間判断で返す (機械判定できないと明示する)
- 完了3区別が完成する (要件完了が機械判定へ移る)
- 品質バー違反 (層境界・複雑度予算違反) を機械で弾く
- requirements 正本も canon 直読み禁止に従い、読み書きはすべて tool 経由
- 再生成・並行ブランチで壊れない (記録層と同じ1件1ファイル方式)
- 将来 owlspec へ橋渡しできる素地を残す

## 仕様

### 要件の状態モデル (考察と提案)

要件には直交する2つの軸がある。

- ライフサイクル: この要件は合意され有効か。人間が宣言する
- 充足: 受け入れ基準が全通過し要件が満たされたか。機械が導出する

この2軸を1つの状態へ混ぜると2つの不都合が出る。1つ目は「確定したがまだ未実装」(accepted かつ未充足) を表せない。2つ目は充足をファイルへ保存すると編集後に古くなり、古い充足を信じて動く罠 (owox://state・owox://verification を却下したのと同じ罠。`docs/decisions/20260613-Phase4-tool記録層.md`) に陥る。

提案 (推奨):

- 要件ファイルはライフサイクル状態だけ持つ: draft (検討中) / accepted (確定・有効) / superseded (新要件へ置換)。人間が宣言する。来歴 (decisions) の状態 (open / adopted / superseded) と同じ思想で一貫させる
- 充足 (met / unmet) は保存しない。受け入れ基準の検証 link から verify.run が実行時に導出する (スライス2)。verify.run が真実、保存ビューは古くなる、という既存方針に乗せる

owlspec 接続の考慮 (この提案を選ぶ理由):

- owlspec は将来「要件仕様管理そのもの」を深く扱う (core 要件の対象外)。owox は最小の requirements 正本層を持ち、将来 owlspec へ橋渡しする
- 橋渡しに有利な形は、型付き・機械可読・並行ブランチでマージ可能 (1件1ファイル・ID=日付+slug・中央台帳なし)、充足を保存しない (owox が古い充足を抱え owlspec のビューと食い違わない)、ライフサイクル語彙を最小に留める (owlspec が深い状態機械を持つ時に上位互換にしやすい)
- ゆえに最小ライフサイクル状態 + 充足は導出、が owlspec 接続にも最も素直

### requirements 正本層 (スライス1)

`.owox/requirements/<id>.md` に1要件1ファイル。記録層 (decisions・tasks) と同形式。

- ID は日付+slug。中央台帳を持たず並行ブランチで衝突しない
- 形式は Markdown (prose 正本と一致)。既存の自作パーサを再利用し新文法を足さない
- canon 直読み禁止に従い、読み書きはすべて tool 経由 (skills のみ直読み例外)

要件1件の構造 (節は design で確定):

- title (1行目)
- 状態: draft / accepted / superseded
- 要件本文: 何を満たすべきか (散文)
- 受け入れ基準: 1基準ごとに given / when / then と検証 link を持つ。検証 link は検証 (テスト名・検査名) を指す。link は任意で後から張れる
- link: 関連来歴など
- supersedes: 置き換えた旧要件 ID

受け入れ基準は1基準ごとに検証 link を張る (要件全体に1本でなく)。どの基準が未検証かを基準単位で機械検出でき、スライス2の trace 欠落検出が精密になる。

tool (引数・data は design で確定):

- requirement.create: 要件を作る
- requirement.list: 要件一覧 (状態で絞れる)
- requirement.get: 要件1件を読む (canon 直読み禁止の読み口)
- requirement.update: 状態遷移・本文更新・受け入れ基準と検証 link の追加

要件本文の本質変更を来歴連動にするかは design で決める (task.update の title 変更が reason 必須・来歴連動という前例がある)。

### 要件↔テスト trace + 要件完了の機械判定 (スライス2)

- trace 欠落の機械検出: accepted な要件で、検証 link が欠ける受け入れ基準を機械検出する。適応度関数の trace 検証として verify に載せる
- 要件完了の機械判定: verify.run の要件完了を、今の一律 needs_human から次のハイブリッドへ移す
  - 要件に紐づく受け入れ基準すべてに検証 link があり、その検証が config.toml の検査で実行でき全通過する → 要件完了を met と機械判定する
  - 受け入れ基準または検証 link が欠ける → 従来どおり needs_human を返し、機械判定できない理由を明示する
- これで「trace のある要件は機械判定、無い要件は人間判断」のハイブリッドになる。完了3区別の意味を壊さず移行でき、第1段階の挙動 (`docs/requirements/20260611-core-要件.md` 完了条件) を温存する
- 完了3区別が完成する。旗「検証可能な正本」が現物で立つ

### quality.toml の適応度関数 (スライス3)

品質バーを quality.toml へ宣言し機械検証する。

- 置き場は別ファイル quality.toml。config.toml は生成・検証設定、quality.toml は品質制約、で役割を分ける (`docs/decisions/20260611-品質保証.md` で「rules = いつ止めるか、quality = 何を守るか」と正本化済み)
- owox は言語非依存。深い構文解析は持たず、言語非依存で機械検証できるものに絞る:
  - 層境界・依存方向: パス glob ベースの禁止依存 (例: あるパス配下は別パス配下を参照禁止)。owox が検出できる範囲に限る
  - 予算検証: ファイル行数・差分量・ファイル数の上限。owox が計測できる
  - 深い解析が要るもの (循環検出・言語別の import 解析・複雑度計測) はプロジェクトの検査コマンドへ委譲し、config.toml の検査として呼ぶ
- 検証は変更箇所・commit 前・CI で走らせ、state (phase) で厳格度を変える (コアゲートは常時維持)

quality.toml で owox が直接検証するものと、検査コマンドへ委譲するものの線引きは design で確定する。

### plan-alignment + 多視点レビューの枠組み (スライス4)

検証2系統のうち判断型レビュー (`docs/decisions/20260611-品質保証.md`) の枠組みを owox 標準として持つ。

- plan-alignment: 規約・既存資産に沿ったかを確かめる観点。決定論で検証できる部分 (quality.toml の規約宣言との整合) はスライス3の機械検証に乗せ、判断が要る部分はレビューの観点として持つ
- 多視点+敵対レビュー: 複数視点で指摘 → 事実性確認 → 別観点で再確認、の進め方を owox 標準の review 枠組みとして定義する (入口 skill の review は Phase5 で配置済み)
- 線引き: 本 Phase は観点・進め方の枠組みのみ。subagent による実行オーケストレーションは Phase8。「枠組みは owox 標準が持ち、実行は Phase8」とする (`docs/handoff/20260613-Phase6開始-検証を閉じる.md` の論点)

## 実現方針

### 段階分け

スライス1から順に縦貫する。各スライスで plan → design → impl → 実機検証。

1. requirements 正本層 + tool (記録層を手本に最短で立てる。価値が高く後続の前提)
2. trace 欠落検出 + 要件完了の機械判定 (完了3区別の完成。旗が立つ核)
3. quality.toml の適応度関数 (言語非依存の機械検証に絞る)
4. plan-alignment + 多視点レビューの枠組み (Phase8 と線引き)

スライス2末で M2 の核 (旗が立つ) が成立する。スライス3・4は品質の深さを足す。

### 変更対象

- crates/core: requirement の型・読込・tool ロジック (record.rs・task.rs を手本)、verify.rs の完了判定拡張、quality.toml の型と適応度関数
- mcp 側: requirement tool の引数型と封筒直列化、日付の供給
- 生成側 (targets/codex): requirement 系 tool の get_info 露出、quality 関連の生成物反映
- tests/fixtures と target サンドボックス
- docs: 本要件・design 判断・validation

### 危険

- 要件完了の機械判定への移行は完了3区別の意味を変える。trace の無い要件を needs_human のまま残すハイブリッドで慎重に移す
- quality.toml の構造検証を owox がどこまで担うか。言語非依存の範囲を超えると対象外 (汎用道具化) に近づく。owox 直接検証と検査コマンド委譲の線引きを誤らない
- 多視点レビューの実行を Phase6 へ引き込むと Phase8 と重複する。枠組みのみに留める
- 受け入れ基準の符号化 (1要件に複数基準・各基準に given/when/then と link) が既存パーサで素直に読めるか。新文法を増やさない範囲で design する

### 人間判断が必要な点

- 要件の状態モデル (上記提案でよいか)
- quality.toml で owox が直接検証する範囲と検査コマンドへ委譲する範囲
- 後回しの Stop ノイズを Phase6 で直すか別スライスか

## 検証

target サンドボックスで実機 Codex により確認する。

- 要件を書き、受け入れ基準に検証を link し、verify.run が要件完了を機械判定で返すか
- 検証 link が欠ける要件は needs_human で返り、理由が明示されるか
- 受け入れ基準の検証 link の欠落を機械検出するか
- quality 違反 (層境界・複雑度予算違反) を弾くか
- requirements の読み書きが tool 経由で回り、canon 直読み禁止に反しないか
- 再生成・並行作成で壊れないか (記録層と同じ確認)
- cargo test / clippy / fmt
