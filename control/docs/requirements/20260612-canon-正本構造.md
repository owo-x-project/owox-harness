# 正本構造 (第1段階)

## 背景

Phase 0・1 で配管と hook 発火を確証した。正本は現状 brand のみで、各ファイルの中身は最小の仮置き。Phase 2 で第1段階の正本構造を正式に決め、ルート指示ファイルを本実装する。

具体内容の正本: `docs/decisions/20260611-target-harness内容.md`
段階区切り: `docs/roadmap/20260612-完成ロードマップ.md` の Phase 2

## 目的

第1段階の正本 (brand 拡張 / rules / context / targets) の項目構造を型で確定し、人間が編集しやすく、生成物が最小で的確になる状態にする。

## 対象

- 正本ソース `.owox/` の型: brand 拡張 / rules / context / targets
- ルート指示ファイル (AGENTS.md) の本実装 (brand + rules + 文脈誘導)

## 対象外

- quality.toml (第1段階要件の対象外。Phase 6 で扱う)
- hook による決定的ブロックの実装 (Phase 3)。本 Phase は型確定と編集性確認まで
- state / agents / commands / permissions / release などの第2段階以降の正本
- MCP 操作・skills・requirements/ など動的部分

## 成功条件

- brand / rules / context / targets の型が確定し、TOML 必須項目・値の種類を型検証できる
- 人間が長文ルールを TOML で編集して苦痛でない
- 生成されたルート指示ファイルが短く的確で、必要な情報だけ出る
- 正本が部分的 (brand のみなど) でも壊れず生成できる (段階的に正本を増やせる)
- 再生成で生成物が壊れない

## 仕様

### 全体方針

- prose の正本 (brand/rules/context/glossary) は Markdown・英語固定、machine 設定 (config.toml。targets 等) は TOML (`docs/decisions/20260612-正本フォーマット.md`)。Markdown は見出し=項目、箇条書き=内容
- 各正本ファイルは任意 (brand 以外は無くても生成が通る)。段階的に増やせる
- 自由文の方針は箇条書きで持つ (人間が編集しやすい)
- 機械が後で強制する境界 (不可逆操作・人間ゲート) や用語集は `名前: 説明` の構造化箇条書きで持つ (区切りは最初のコロン。全キーボードで打てる)
- 型は読込後に Rust で検証する。未知見出し・必須欠落を弾く (誤記検出)
- 語彙変更: north_star → Vision (`docs/decisions/20260613-Vision語彙とセッション目標.md`)

### brand 拡張

既存 (vision / values / principles / non_goals / success_criteria) に追加:

- style: 表記・文体の方針 (文字列配列)

用語集は brand から外し、独立ファイル glossary.md にした (用語を直読みする時 brand の他項目を巻き込まない)。

### rules

- change_policy: 変更方針 (文字列配列)
- dependency_policy: 依存追加の条件 (文字列配列)
- deletion_policy: 削除基準 (文字列配列)
- safety: 安全性。秘密情報・外部送信・危険操作の扱い (文字列配列)
- irreversible: 不可逆操作。実行前に人間確認必須。各エントリは id / operation / reason
- human_gate: 人間ゲート。AI が必ず人間へ戻す判断点。各エントリは id / when / reason

検出方法は Phase 3 で確定 (`docs/decisions/20260612-Phase3-hook実装.md`)。irreversible は任意の `detect:` 行 (コマンド正規表現) を非破壊追加し、owox 同梱の既定検出器 ∪ target 記述で PreToolUse が機械検出する。human_gate はファジーで機械検出に向かないため検出フィールドを持たず、AGENTS.md と完了時の誘導で扱う。正規表現の妥当性は読込時に検証する。

### context

文脈地図。作業・場所ごとに必読と適用ルールを示す (`docs/decisions/20260612-段階的開示.md`)。

- 各 `## 見出し` が 1 エントリ。見出し = スコープ (作業の説明、またはパス glob)
- 配下は `key: value` の箇条書き: `kind` (task / path、既定 task) / `read` (必読パス、複数可) / `note` (適用ルール・スタイル、複数可)
- 未知キー・不正な kind はエラー
- 本 Phase は型確定のみ。スコープに応じて届ける実装は Phase 3 (編集前 hook) ・Phase 4 (owox://)。常時読む AGENTS.md には出さない

### glossary

用語集。独立ファイル glossary.md (Markdown)。

- `## 見出し` 配下の `- 用語: 説明` を全節から集める。見出しで用語を分類してよい
- 用語名は AGENTS.md に常時、定義はオンデマンド (hook の push)。`docs/decisions/20260612-段階的開示.md`

### config (machine 設定)

machine 設定を config.toml (TOML) に集約する。今は targets のみ。今後の設定的な項目もここへ足す。

- `[targets.<cli>]` が 1 CLI。`out` (出力先、既定 ".") / `[targets.<cli>.models]` の `<tier> = "<model>"` (ティア → 具体モデル)
- targets 以外のトップレベルキーは各設定の読み手が解釈する (未知キーで全体を壊さない)
- ティア → モデルは Phase 8 のオーケストレーションで使う。登録表 (target.rs) を将来この正本駆動へ寄せる素地

### ルート指示ファイルと段階的開示

旗「最小コンテキスト」。常時読むルートを小さくするのが狙い。

- 第1段階は AGENTS.md 1 枚に集約する (brand → rules の順)。正本が小さいうちは常時コスト差が僅少で、捨てる中間ファイルを作らない
- 真の段階的開示 (詳細を必要時のみ配信し常時トークンを削減) は Phase 4 の owox:// resource (MCP) で行う。AGENTS.md の節順を [Goal → 安全境界 → 詳細] に保ち、Phase 4 で詳細を owox:// へ持ち上げやすくする
- 編集禁止の注記は生成物に置かない (常時読むトークンを節約)。源が .owox/ で生成物は編集しない案内は setup/README で一度だけ行う (Phase 5)

### 失敗時の扱い

- 必須項目欠落・型違い・未知キー → 型検証エラーで生成を止め、原因パスを返す
- 任意ファイルが無い → その正本は空として扱い、対応する節を出さない

## 実現方針

縦スライスで正本を1つずつ貫通する (型確定 → 生成物反映 → 実機で編集性確認)。順序:

1. rules (制御の核。最も価値が高い。長文編集性を早期に確認)
2. brand 拡張 (glossary / notation)
3. context (文脈地図。session 開始 hook の「次に読むもの」を正本駆動へ)
4. targets (型確定のみ)

各スライスで: model.rs に型追加 → load.rs で任意読込 → codex.rs の生成へ反映 → tests/fixtures とサンドボックスで編集 → 再生成・検証。

変更対象: crates/core (model / load / targets/codex)、tests/fixtures、検証は target サンドボックス。

危険: 正本ファイルを必須にすると既存の最小 fixture が壊れる → 任意読込で回避。

人間判断が必要な点: 各正本の項目が過不足ないか、TOML で編集して苦痛でないか (実機編集で確認)。
